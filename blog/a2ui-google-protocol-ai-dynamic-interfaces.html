A2UI in Action: Beyond Text Responses

Imagine asking an AI assistant for restaurant recommendations and receiving not just a list of names, but a complete visual interface with photos, ratings, and maps - all generated dynamically based on your query. This is the power of Google's A2UI protocol in action.

In traditional AI systems, every visual element requires pre-built components and explicit coding. A2UI flips this model by enabling the AI to assemble appropriate interface elements on demand. The demo shows three radically different interfaces - a map, a clock, and a restaurant finder - all generated by the same agent in response to different queries.

The key breakthrough: No pre-coded React components were created for these interfaces. The AI dynamically selected and composed appropriate UI elements from a pre-approved catalog based solely on the context of each query.

How A2UI Works: The JSON Protocol

A2UI solves a fundamental limitation of current AI systems - their inability to generate rich interfaces while maintaining security. The protocol works by having the AI describe interfaces using structured JSON rather than generating executable code.

When a user makes a query, the AI agent processes it normally but instead of returning plain text, it outputs a JSON description of the appropriate interface components. These components must come from a pre-approved catalog - the AI can't invent new components or inject arbitrary code. The JSON is then rendered by a secure client-side renderer.

Analogy: Think of A2UI like HTML for AI interfaces. Just as HTML describes web pages but needs a browser to render them, A2UI JSON describes interfaces but needs a renderer to display them.

Security Advantages Over AI-Generated Code

The most common objection to AI-generated interfaces is security - how can you trust code created dynamically by an AI? A2UI solves this through its constrained JSON approach.

Unlike systems where AI generates executable code, A2UI only allows description of interfaces using pre-defined component types. This means:

  • No arbitrary code execution - components are rendered by trusted client code
  • No injection vulnerabilities - the AI can't introduce malicious scripts
  • Controlled data access - components only receive data explicitly provided

Google's reference implementation includes rigorous schema validation to ensure all A2UI JSON complies with the protocol before rendering.

The Renderer Ecosystem

A2UI's power comes from its open protocol approach. Google provides a Lit-based renderer for web apps, but the ecosystem already includes:

  • Copilot Kit: A React-based renderer for JavaScript applications
  • Flutter Gen UI: Native mobile renderer for iOS and Android
  • Angular Renderer: Official Google implementation for Angular apps

This means the same A2UI JSON can power native interfaces across web, mobile, and desktop platforms while maintaining each platform's look and feel. Developers can even create custom renderers for specialized use cases.

Implementation Steps

Getting started with A2UI involves three main components:

Step 1: Set Up the Agent

Configure your AI agent to output A2UI JSON instead of plain text. Google's Agent Development Kit (ADK) provides tooling to simplify this process.

Step 2: Choose a Renderer

Select an appropriate renderer for your platform - web, mobile, or desktop. The open-source implementations provide solid starting points.

Step 3: Define Your Component Catalog

Create the library of UI components your AI can compose. Start with basic elements like cards, lists, and maps, then expand based on your application needs.

Pro Tip: Google's GitHub repository includes sample implementations for common use cases like restaurant finders and contact lookup that can be adapted to jumpstart your project.
