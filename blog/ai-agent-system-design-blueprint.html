The 3-Layer Framework for AI Agent Design

Effective agent systems require thinking at three distinct levels of abstraction. Like architectural blueprints for a building, each layer serves a specific purpose in the design process:

1. Context Diagram (10,000 ft view)

Identifies all external systems and actors that interact with your agent. Answers: Who/what depends on this? What boundaries does it operate within?

2. Application Diagram (1,000 ft view)

Shows the internal deployable units and their interactions. Answers: What services/components make up the system? How do they communicate?

3. Component Diagram (Ground level)

Details the logic flow within each application. Answers: How does the agent make decisions? What's the sequence of operations?

Why this works: This layered approach prevents teams from getting bogged down in implementation details too early while ensuring all integration points are considered from the start.

Layer 2: Application Diagram (Internal Components)

Zooming into the help desk triage system reveals four core applications working together:

1. Chat UI (React frontend)

Handles customer interactions through a conversational interface. Authenticates users via Auth0 before passing requests to the triage agent API.

2. Triage Agent (.NET/TypeScript API)

Orchestrates the agentic loop - loads context, calls OpenAI, evaluates responses, and executes tools. The brains of the operation.

3. Injection Worker (Python service)

Processes Confluence webhooks to keep the vector database updated with the latest documentation through OpenAI embeddings.

4. Vector Store (Postgres with pg_vector)

Enables semantic search across historical tickets and knowledge base articles using vector similarity rather than exact keyword matching.

At 12:15, the video demonstrates how data flows between these components during a typical ticket resolution process.

Layer 3: Component Diagram (Agent Logic Flow)

The triage agent's internal architecture follows a proven pattern for building reliable AI agents:

Planner

Formats prompts with conversation history and system instructions before calling the LLM. Handles retries and fallback logic.

Assessor

Evaluates LLM responses to determine next steps: reply to user, execute tools, or continue reasoning. Implements guardrails to prevent infinite loops.

Tool Executor

Translates LLM tool requests into actual API calls to Salesforce, Confluence, etc. Validates inputs and handles errors gracefully.

System Prompt

Contains the agent's instructions, tone guidelines, and tool definitions. The most frequently updated component as the agent evolves.

Key insight: At 18:30, the video shows how the assessor prevents cost overruns by limiting tool executions per conversation. This is critical for production systems where unbounded LLM calls can quickly become expensive.

How the Agentic Loop Works in Practice

The magic happens in the continuous cycle of reasoning and action:

  1. Load Context - Retrieve conversation history and relevant records
  2. Reason - Have the LLM analyze the situation with all available data
  3. Assess - Determine if tools need execution or if response is ready
  4. Act - Execute required tools and incorporate their outputs
  5. Repeat - Continue until resolution or escalation criteria met

At 22:45, the video demonstrates a real ticket flowing through this loop three times before resolution - each iteration building on the last with additional context from tool executions.

7 Implementation Tips From Production Systems

1. Start small with one painful process
Don't boil the ocean. The help desk triage agent began by handling just password reset requests before expanding to other ticket types.

2. Instrument observability from day one
Track token usage, tool execution stats, and conversation loop counts. These metrics reveal optimization opportunities.

3. Implement circuit breakers
Hard limits on loops (max 5) and tool executions prevent runaway costs from edge cases.

4. Version your system prompts
Treat them like code - track changes and maintain the ability to roll back if performance degrades.

5. Cache embeddings aggressively
Regenerating embeddings for unchanged documentation wastes cycles and money.

6. Design for graceful degradation
When external systems fail, agents should explain delays rather than break completely.

7. Plan the human handoff
Clear escalation paths maintain customer trust when the agent reaches its limits.

An agentic loop is the continuous cycle where an AI agent: 1) Loads context and history, 2) Reasons about the task using an LLM, 3) Determines if tools need to be executed, 4) Gathers additional information through tools if needed, then repeats until it has a complete response.

This loop enables autonomous decision-making by allowing the agent to seek additional information when needed rather than requiring perfect initial inputs.

  • Typical loops complete in 3-5 iterations
  • Each iteration adds context from tool outputs
  • Circuit breakers prevent infinite loops

Vector databases enable semantic search across unstructured data like documentation or past tickets. Unlike traditional databases that require exact matches, vector stores find conceptually similar content using embeddings.

This allows AI agents to retrieve relevant information even when the user's query doesn't match the exact wording in your knowledge base - critical for handling natural language variations.

  • Convert text to vectors using embedding models
  • Search by conceptual similarity, not keywords
  • Postgres with pg_vector is a popular option

Authentication should be handled at the API gateway level before requests reach your agent logic. Common patterns include: 1) Validating JWT tokens from providers like Auth0, 2) Checking API keys for system-to-system communication, and 3) Implementing role-based access control for different agent capabilities.

Always authenticate before loading any conversation history to prevent data leaks between users. The agent itself should never contain auth logic - that belongs at the perimeter.

  • Use standard providers (Auth0, Okta, etc.)
  • Validate tokens before processing requests
  • Implement least-privilege access

Key metrics include: 1) LLM token usage and cost per operation, 2) Tool execution success rates, 3) End-to-end latency for agent responses, 4) Conversation loop iterations before resolution, and 5) Escalation rates to human operators.

These help optimize cost and performance while maintaining quality. For example, noticing that certain ticket types consistently require 4+ loops might indicate needed improvements to your system prompt or knowledge base.

  • Track costs at the conversation level
  • Monitor for tool execution failures
  • Alert on abnormal loop counts

Implement three safeguards: 1) Maximum iteration limits (e.g., 5 loops), 2) Timeout thresholds for total processing time, and 3) Circuit breakers that trigger when consecutive tool executions don't progress the conversation.

Always include a fallback path to human escalation when limits are reached. Well-designed agents recognize their own limitations and gracefully hand off rather than failing catastrophically.

  • Hard limits prevent runaway costs
  • Timeouts ensure responsiveness
  • Circuit breakers catch edge cases

The planner orchestrates the overall flow - it loads context, formats prompts, and calls the LLM. The assessor evaluates the LLM's response to determine next steps: whether to reply to the user, execute tools, or continue reasoning.

This separation of concerns makes the system more maintainable and debuggable. Changes to prompt engineering happen in the planner, while conversation flow logic stays in the assessor.

  • Planner = prompt construction
  • Assessor = response evaluation
  • Separate these responsibilities
