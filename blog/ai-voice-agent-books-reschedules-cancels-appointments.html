How the AI Receptionist System Works

The system architecture combines three core components:

  1. ElevenLabs Voice Agent - Handles call answering, speech recognition, and natural conversation flow
  2. Cal.com API - Manages calendar availability, booking creation, and meeting modifications
  3. Knowledge Base - Provides answers to common questions about services, pricing, and FAQs

When a call comes in, the agent follows this decision flow:

1. Answers call with branded greeting
2. Routes based on caller intent (booking, question, transfer request)
3. For bookings: Collects name → email (with verification) → preferred time
4. Checks calendar API for availability
5. Confirms booking and sends calendar invite

Configuring the ElevenLabs Voice Agent

The ElevenLabs agent requires careful prompt engineering to handle the complex scheduling logic. The system prompt contains six critical sections:

  • Role Definition - "You are Amy, the receptionist agent for [Business Name]"
  • Personality - Friendly yet professional tone with clear enunciation
  • Scheduling Logic - Step-by-step instructions for time zone conversion
  • Email Verification - Mandatory spelling confirmation for accuracy
  • Transfer Rules - When and how to escalate to human team members
  • Error Handling - Fallback procedures when API calls fail

The demo uses GPT-4 for inference with temperature set to 0.7 - balanced between consistency and natural variation in responses.

Calendar Integration with Cal.com

Cal.com serves as the scheduling bridge between the AI agent and your existing calendar. Key advantages over direct Google/Microsoft integration:

  • Unified API for 20+ calendar providers
  • Built-in availability rules and scheduling logic
  • Webhook support for real-time updates

The integration requires five custom tools in ElevenLabs:

1. Cal_availability - Checks open time slots
2. Cal_book - Creates new appointments
3. Cal_get - Retrieves existing bookings
4. Cal_reschedule - Modifies appointment times
5. Cal_cancel - Removes scheduled meetings

The Appointment Booking Workflow

The booking process demonstrates the system's sophistication:

Step 1: Caller provides name → AI confirms spelling
Step 2: Email collection with dual verification (spoken + spelled)
Step 3: Preferred time request → Time zone conversion to UTC
Step 4: API checks availability → Offers alternatives if needed
Step 5: Final confirmation with all details restated

The workflow includes smart error recovery - if the initial time isn't available, the system automatically checks +3 hours from the requested time while maintaining natural conversation flow.

Rescheduling & Cancellation Logic

Modifying existing appointments requires careful state management:

  1. Caller provides booking email (with spelling verification)
  2. System queries Cal.com API for matching appointments
  3. If multiple found, lists options for caller to choose
  4. For reschedules: Checks new time availability via API
  5. For cancellations: Confirms intent before processing

Critical detail: The system always converts displayed times back to the caller's local time zone, never showing raw UTC values that would confuse non-technical users.

Human Transfer Functionality

When callers request human assistance, the agent:

  1. Confirms the reason for transfer
  2. Verifies availability of the requested person/department
  3. Initiates transfer via SIP or conference bridge
  4. Provides fallback option if transfer fails

The demo configures transfers to:

  • Conference mode: For Twilio numbers (keeps both parties on line initially)
  • SIP transfer: For direct SIP trunks (immediate handoff)

Automatic Time Zone Conversion

The system handles time zones through a three-step process:

1. Input: Caller says "Tomorrow at 2pm" → Interpreted as local time (e.g. Australia/Sydney)
2. Processing: Convert to UTC for Cal.com API operations
3. Output: Convert API responses back to local time for display

This happens seamlessly through prompt engineering that includes:

  • Explicit time zone declaration in system prompt
  • Example conversions for the LLM to follow
  • Validation that displayed times are always localized
