Zoho API Setup

The first step is configuring Zoho's API console to generate the necessary authentication credentials. At 2:15 in the video, you'll see the exact navigation path to create a self-client in Zoho's developer console.

Three critical pieces you'll generate:

  1. Authorization Code: Valid for 10 minutes (set this as the maximum duration during testing)
  2. Client ID: Your permanent application identifier
  3. Client Secret: The private key for API authentication

Store these securely - you'll need all three for the next authentication step in Postman.

Postman Authentication

Postman serves as our intermediary to convert the temporary authorization code into a long-lived refresh token. At 4:30 in the tutorial, you'll see the exact POST request configuration needed.

The key elements of your Postman request:

  • Endpoint: https://accounts.zoho.com/oauth/v2/token
  • Method: POST
  • Body parameters: code, client_id, client_secret, grant_type=authorization_code

Critical note: The response will include a refresh_token - this is your golden ticket for ongoing API access. Store it securely as it allows your Make.com workflow to generate fresh access tokens indefinitely.

Make.com Workflow

The Make.com blueprint (available in the video description) contains two HTTP modules that handle the entire process:

Module 1: Token Generation

This module uses your refresh token to generate a time-limited access token. Configure it with:

  • Your Zoho client_id and client_secret
  • The refresh_token from Postman
  • grant_type=refresh_token

Module 2: Calendar Query

This module makes the actual free/busy request, requiring:

  • The access_token from Module 1
  • Your Zoho email address
  • Date range parameters (defaults to tomorrow + 3 days)

At 7:45 in the video, you'll see how to test each module individually before running the full workflow.

Timezone Conversion

Zoho returns all times in UTC by default, which can cause confusion when scheduling across timezones. The tutorial shows the raw UTC output at 9:20, where a 1:00 PM meeting appears as 7:30 AM in the API response.

For production use, we recommend adding a timezone conversion module that:

  1. Identifies the user's local timezone
  2. Converts UTC timestamps to local time
  3. Formats the output for clear readability

This extra step prevents costly scheduling errors when coordinating across regions.

Testing and Troubleshooting

Common issues and their solutions:

401 Unauthorized errors: Usually mean your refresh token expired or was revoked. Generate a new one following the initial Postman steps.

Other frequent challenges:

  • Empty responses: Verify your date range includes scheduled events
  • Timezone mismatches: Double-check UTC conversion logic
  • Rate limits: Zoho allows ~100 requests/hour - space out your checks
