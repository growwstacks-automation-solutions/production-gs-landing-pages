LangChain4j's Agentic Module Overview

Traditional AI implementations often rely on a single, large language model trying to handle everything - leading to expensive, slow, and sometimes confused responses. LangChain4j's new Agentic module solves this by letting you create specialized AI agents that each excel at specific tasks while collaborating through shared memory.

The demo shows how four agents worked together to plan a party for LangChain4j contributors: a recipe agent generated an apple pie recipe, a GitHub agent fetched contributor data, a shopping list agent assigned ingredients to contributors, and a gist agent published the results. This modular approach provides better results at lower cost than a single monolithic agent.

Key innovation: The Agentic module introduces shared memory where agents can store and retrieve information, allowing them to collaborate without direct coupling. This makes complex workflows possible while keeping each agent simple and focused.

Building the Recipe Agent

The simplest agent in our demo was the recipe generator - a single interface with one method that takes a recipe name and returns the ingredients. Despite its simplicity, it demonstrates core Agentic module concepts:

 @Agent("Provides recipes based on recipe names") public interface RecipeAgent {     @UserMessage("I'm going to a recipe called {recipeName}. Give me the list of ingredients.")     Recipe getRecipe(String recipeName); }

The @Agent annotation describes what the agent does, while @UserMessage provides the prompt template. LangChain4j automatically generates the implementation that connects to your configured LLM. At runtime, calling getRecipe("apple pie") returns a structured Recipe object that gets stored in shared memory for other agents to use.

Implementation insight: The actual agent implementation is generated automatically by LangChain4j - you just define the interface with annotations. This makes creating new agents remarkably simple while maintaining type safety.

Creating the GitHub Contributors Agent

While the recipe agent only used an LLM, the GitHub contributors agent demonstrates how agents can integrate external tools. This agent fetches the last 50 contributors to the LangChain4j repository using GitHub's API through an MCP server:

 @Agent("Reads the last GitHub contributors from the provided repository") public interface GitHubContributorsAgent {     @UserMessage("Give me the list of the last 50 contributors to {repository}")     Contributors getContributors(String repository); }

The key difference is the agent's configuration includes a ToolProvider that connects to the GitHub MCP server. When invoked, the agent first uses its LLM to understand the request, then delegates to the MCP server to actually fetch the data. The contributors list joins the apple pie recipe in shared memory.

Tool integration: Agents can use three types of tools - MCP servers (like GitHub API), Java functions, and Spring beans. This lets them go beyond simple LLM responses to interact with real systems and data sources.

Agent Orchestration Patterns

With both the recipe and GitHub agents working, we needed to coordinate them. LangChain4j provides two orchestration approaches shown in the demo:

1. Parallel Execution

The first orchestrator runs both agents simultaneously using Java virtual threads:

 @Agent("Orchestrates recipe and contributor agents") public interface DataWorkerAgent {     @Workflow     void orchestrate(String repository, String recipeName); }

The @Workflow annotation indicates this agent will explicitly define the execution pattern rather than letting an LLM decide. The implementation uses ParallelExecutionBuilder to run both agents concurrently, with their results automatically stored in shared memory.

2. Sequential Execution

The second orchestrator runs agents in sequence - first the shopping list agent processes the recipe and contributors, then the gist agent publishes the results:

 @Agent("Sequentially executes shopping list then gist agents") public interface PartyPlannerAgent {     @Workflow     String planParty(String repository, String recipeName); }

This demonstrates how different orchestrators can reuse the same underlying agents in different patterns depending on the workflow needs.

Developing the Shopping List Agent

The shopping list agent shows how agents can use Java functions as tools. It takes the recipe ingredients and contributor list, then assigns each ingredient to a contributor:

 @Agent("Creates shopping lists from recipes and contributors") public interface ShoppingListAgent {     @UserMessage("Create a shopping list with {contributors} buying items for {recipe}")     String createShoppingList(Contributors contributors, Recipe recipe); }

Unlike previous agents, this one uses a @Tool-annotated Java function to do the actual assignment logic rather than an LLM. The function takes the contributors and recipe from shared memory, implements the distribution logic in regular Java code, and returns a Markdown-formatted shopping list.

Hybrid approach: By combining LLMs for understanding requests with Java functions for precise logic, you get both natural language flexibility and deterministic execution where needed.

Implementing the Gist Service Agent

The final agent publishes the shopping list as a GitHub gist, demonstrating how agents can write data to external systems. It uses a Spring bean tool to call GitHub's API:

 @Agent("Creates GitHub gists from content") public interface GistServiceAgent {     @UserMessage("Create a gist with this content: {content}")     String createGist(String content); }

The configured Spring bean handles authentication and makes the actual HTTP request to GitHub's API. The agent itself focuses on coordinating with the LLM - understanding the request, formatting the content, and returning the gist URL. This separation keeps the agent code clean while allowing integration with complex external systems.

In the demo, running the full workflow produces a GitHub gist with the shopping list assigning apple pie ingredients to LangChain4j contributors - all coordinated through four specialized agents working together.

Benefits of Specialized Agents

While a single large agent could theoretically handle this entire workflow, the modular approach provides three key advantages:

1. Better Results

Each agent uses the most appropriate LLM and tools for its specific task. The recipe agent might use a culinary-focused model, while the GitHub agent uses one trained on developer documentation. This specialization leads to higher quality outputs.

2. Lower Costs

By dividing the work, each agent only processes the relevant context. The recipe agent doesn't need to know about GitHub API details, and the gist agent doesn't need culinary knowledge. This focused context reduces token usage and costs.

3. Easier Maintenance

With clear responsibilities, agents can be updated independently. Changing the recipe format doesn't affect the GitHub integration, and updating the gist API client doesn't risk breaking the shopping list logic.

Real-world impact: In business applications, this approach lets you build complex AI systems where marketing, sales, and operations each have specialized agents that collaborate through shared data - much like human departments do.

Key Takeaways

LangChain4j's Agentic module represents a significant evolution in how we build AI systems - moving from monolithic models to collaborative teams of specialized agents. By combining the strengths of LLMs with traditional programming constructs like interfaces and shared memory, it creates a powerful framework for building maintainable, cost-effective AI solutions.

In summary: 1) Create focused agents with @Agent annotations, 2) Connect them to tools with @Tool, 3) Orchestrate workflows with parallel and sequential execution, and 4) Let them collaborate through shared memory. This architecture scales better than single-model approaches for complex business workflows.
