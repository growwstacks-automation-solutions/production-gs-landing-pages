The 3 Essential Components of an AI Security Agent

Security teams often struggle with alert fatigue - drowning in Wazuh notifications while missing critical threats. Traditional SIEM tools require complex queries and manual analysis, leaving junior analysts overwhelmed. The breakthrough comes when we structure our AI agent with three core components that mirror human security operations.

First is the LLM (Large Language Model) - acting as the agent's analytical brain. While you could use GPT-4, we start with Gemini because it offers free API access without payment requirements. Next comes memory - simple RAM storage that maintains conversation context. Finally, tools - primarily HTTP requests that let the agent interact with Wazuh's API just like a human analyst would through Postman.

Key architecture insight: This agent structure processes alerts at scale while maintaining explainability - every analysis step is visible in n8n's execution logs, unlike black-box security AI solutions.

Why Gemini (Not GPT) for Your First Security Agent

Many security teams assume they need the most powerful LLM available, but this often leads to unnecessary costs during development. Gemini provides three critical advantages for prototype security agents:

  1. Zero-cost prototyping: No credit card required to generate API keys
  2. Sufficient capability: Handles basic alert summarization and analysis
  3. Easy migration path: Switch to GPT-4/5 later by simply changing the model credential

In our tests, Gemini successfully analyzed common Wazuh alerts including package installations, sudo activity, and SELinux events. For more complex threat detection scenarios, you might eventually upgrade - but Gemini proves you don't need premium models to start automating security analysis.

Step 1: Setting Up Your n8n Environment

Before connecting to Wazuh, we need to configure n8n's workflow structure. Unlike traditional scripting, n8n uses visual workflows that make security automation accessible to non-coders.

Start by creating a new workflow in your n8n instance (community edition works fine). We'll use three key nodes:

  1. Chat Trigger: Our conversation interface (upgrade to Teams/Slack later)
  2. AI Agent: The core processing unit with Gemini
  3. HTTP Request: For Wazuh API communication

Pro tip: Name your workflow clearly like "Wazuh Security Agent" - this becomes critical when managing multiple automation workflows.

Step 2: Configuring Gemini as Your AI Brain

With our workflow skeleton ready, we now bring our AI agent to life by connecting Gemini. The process involves:

  1. Generating a free API key from Google AI Studio
  2. Creating a new credential in n8n named "Wazuh Demo Key"
  3. Selecting Gemini 1.5 Flash as our model (balances speed and capability)

Critical configuration note: Gemini requires separate tools for data retrieval and analysis. We implement this by adding both an HTTP request tool (for Wazuh) and a Gemini tool (for summarization). This dual-tool approach ensures clean separation of data fetching and intelligence generation.

Step 3: Connecting to Wazuh's API

Here's where most security automation projects fail - improper API integration. Unlike Postman, n8n requires specific configuration to communicate with Wazuh's Dockerized components:

  1. Container naming: Use sudo docker ps to find your Wazuh indexer's exact container name
  2. Port specification: Must include port 9200 in requests
  3. Authentication: Basic Auth with admin/secret credentials (match your Postman env)

The HTTP request tool needs particular attention to:

  • Set as POST (not GET) for alert queries
  • Include Content-Type: application/json header
  • Pass the full query body from Postman (initially)

We simplify by letting the agent control just one parameter initially - number of alerts to retrieve - while keeping other query elements static. This balances flexibility with control during testing.

Crafting the Perfect System Prompt

The system prompt determines whether your agent provides vague generalizations or actionable security insights. After testing multiple approaches, we found this structure works best:

Prompt framework: Role (Security Analyst) + Task (Alert Analysis) + Output Format (Concise Bullets) + Alert Severity Definitions

Example effective prompt: "You are a senior security analyst reviewing Wazuh alerts. Summarize key patterns, highlight medium/high severity events, and suggest investigation steps. Format responses with: 1) Alert overview, 2) Severity assessment, 3) Recommended actions."

We generate this in ChatGPT first, then refine for conciseness. The prompt must balance specificity (to guide analysis) with flexibility (to handle diverse alerts).

Testing Your Agent's Threat Detection

With everything connected, we validate our agent with real Wazuh alerts. The test conversation flow:

  1. "Get me the last 20 alerts and summarize them"
  2. "What would you recommend I do to investigate these alerts?"

In our demo, the agent successfully:

  • Identified frequent sudo/PAM activity (from our Docker commands)
  • Flagged medium-severity package management events
  • Suggested verifying user roles for Docker commands

The n8n execution logs provide full visibility - you can see exactly when the API call succeeds and how the LLM processes the raw alerts into actionable insights.
