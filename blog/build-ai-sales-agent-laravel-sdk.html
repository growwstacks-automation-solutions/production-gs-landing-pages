Laravel AI SDK Overview

The newly released Laravel AI SDK provides everything needed to build conversational agents directly into your Laravel application. Unlike generic chatbot solutions, this SDK is specifically designed for developers who need to integrate AI capabilities with their existing database models and business logic.

The SDK handles three critical aspects of AI agent development: conversation management (storing message history), tool execution (connecting LLM capabilities to your database), and model switching (between providers like OpenAI, Anthropic, or Gemini). This means you can focus on defining what your agent can do rather than how it communicates with AI models.

Key architecture: The SDK automatically creates two database tables - agent_conversations and agent_conversation_messages - to store chat history. Each conversation is associated with a user, allowing for personalized interactions while maintaining data isolation.

Setting Up the Sales Agent

Implementing a basic sales agent requires just four steps after installing the Laravel AI SDK package. First, publish the configuration and migration files. Second, run the migrations to create the conversation tables. Third, configure your AI provider credentials (like OpenAI API key). Fourth, create your agent class.

The agent class defines three key components: the agent's instructions (its purpose and boundaries), available tools (functions it can call), and conversation handling. In our sales assistant example, we defined clear boundaries - the agent should only provide information about products, categories, and orders, nothing else.

 // Creating the sales assistant agent php artisan make:agent SalesAssistant // Defining agent instructions protected function instructions(): string {     return "You are a helpful sales assistant. Your job is to provide              product and order information to customers."; }

Creating Tools for Product & Order Queries

Tools are the bridge between natural language queries and your database. Each tool is a class that defines: (1) what the tool does, (2) what parameters it accepts, and (3) how to execute the query. Our sales agent uses three tools: product listing, category filtering, and order history.

The product tool demonstrates intelligent filtering - it accepts an optional category parameter. When provided, it filters products by category; otherwise, it returns all products. This flexibility allows the agent to handle both broad ("show me products") and specific ("show men's tops") requests naturally.

Tool security: Notice how the order tool automatically scopes queries to the authenticated user. This pattern prevents data leakage while maintaining natural language convenience. The AI only sees filtered results, never direct database access.

Implementing Conversation History

One of the SDK's most powerful features is automatic conversation history. By simply adding the RemembersConversation trait to your agent, all interactions are stored in the database with no additional code. Each conversation gets a unique ID, with messages stored as related records.

This history enables two key benefits: context awareness across multiple interactions ("What was that product you showed me earlier?") and analytics on common customer queries. The demo shows how passing a conversation ID retrieves previous messages, making the agent seem more intelligent and personalized.

 // Saving conversation history use RemembersConversation; $response = SalesAssistant::make()     ->forUser($user)     ->prompt($request->message);

Frontend Integration with React

The backend agent shines when paired with a responsive frontend. The tutorial demonstrates integration with React (via Inertia.js), but the approach works with any JavaScript framework. The key components are: a message input, send button, and message display area.

For richer output, we added markdown parsing to format product lists returned by the agent. The frontend makes POST requests to a dedicated agent endpoint, passing the user's message and receiving the agent's response. Loading states and error handling create a polished user experience.

Pro tip: Always sanitize markdown output before rendering to prevent XSS attacks. The demo uses dangerouslySetInnerHTML for simplicity, but production apps should use DOMPurify or similar libraries.

Production Considerations

While the demo shows core functionality, production implementations need additional safeguards. Rate limiting prevents abuse, input validation ensures clean queries, and output sanitization blocks malicious content. You'll also want proper error handling when the AI provider is unavailable.

For complex agents, consider: splitting tools into separate classes, adding usage analytics, and implementing a fallback mechanism when the AI can't understand a query. The Laravel AI SDK's model-switching capability also lets you fail over to a different provider if your primary one has issues.
