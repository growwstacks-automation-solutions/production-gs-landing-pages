What is an MCP Server?

An MCP (Model Context Protocol) server acts as a specialized toolbox that AI assistants can access whenever they need specific tools to respond to user queries. Unlike standalone AI models that have fixed capabilities, MCP servers extend functionality by providing on-demand access to custom tools and functions.

Imagine your AI assistant needs to perform a complex calculation, access a database, or execute specialized business logic. Instead of trying to handle everything internally, it can call upon your MCP servers to perform these tasks efficiently. This architecture keeps your AI lightweight while providing unlimited extensibility.

Key benefit: MCP servers let you add new capabilities to your AI without retraining the model. Just create a new tool on your server, and it becomes immediately available.

Setting Up Your Development Environment

Before building our MCP servers, we need to set up our Python environment with the right tools. The modern Python package manager UV (written in Rust) provides faster and more reliable dependency management than traditional PIP.

After installing UV, run uvinit to generate the project structure. This creates all the necessary files including main.py and sets up a clean workspace. Create a requirements.txt file listing your dependencies - for MCP servers, you'll need at minimum the mcp and fast-mcp packages.

Pro tip: FastMCP reduces hundreds of lines of boilerplate code to just a few lines, making server development dramatically faster and less error-prone.

Building an SDDO MCP Server

SDDO (Standard Input/Output) servers are ideal for local tool access where your AI client runs on the same machine as the server. They communicate directly through input/output streams without network overhead.

Start by importing FastMCP and creating your server instance. The name parameter identifies your server to clients, while instructions tell the AI how to use it. For our math server example, we'll add a simple addition tool:

 from mcp.server.fast_mcp import FastMCP mcp = FastMCP(     name="Demo Server",     instructions="This server provides mathematical calculation tools." ) @mcp.tool async def add(a: float, b: float) -> float:     """Add two numbers together and return the result."""     return a + b def main():     mcp.run(transport="sddo") if __name__ == "__main__":     main()

The @mcp.tool decorator is what transforms a regular Python function into an MCP-accessible tool. Notice we use async def - this allows the server to handle multiple requests concurrently.

Creating an HTTP MCP Server

HTTP MCP servers allow remote access to your tools via web requests. This is essential when your AI client runs on different machines or in the cloud. The setup is similar to SDDO but with additional HTTP-specific parameters.

 from mcp.server.fast_mcp import FastMCP mcp = FastMCP(     name="Demo HTTP Server",     port=8001,     stateless_http=False,     streamable_http_path="/my_mcp",     host="127.0.0.1" ) @mcp.tool async def add(a: float, b: float) -> float:     """Add two numbers together and return the result."""     return f"The sum of {a} + {b} = {a + b}" def main():     mcp.run(transport="streamable_http") if __name__ == "__main__":     main()

Key differences include specifying a port number, HTTP path, and host address. The stateless_http parameter controls whether the server maintains state between requests. When false (as in our example), it remembers information across calls.

Key Differences Between SDDO and HTTP

While both server types provide the same core functionality, their use cases differ significantly. SDDO servers offer lower latency for same-machine communication, while HTTP servers enable distributed access across networks.

SDDO is ideal for:

  • Local development and testing
  • High-performance tools where latency matters
  • Secure environments where network access is restricted

HTTP shines when you need:

  • Remote access from cloud-based AI systems
  • Scalability across multiple machines
  • Integration with existing web infrastructure

Performance note: In benchmarks, SDDO servers typically respond 3-5x faster than HTTP for local requests, making them preferable when low latency is critical.

Testing Your MCP Servers

Proper testing ensures your MCP servers work as expected before connecting them to production AI systems. For SDDO servers, you'll need to create a simple client script that starts the server process and sends test requests.

HTTP servers can be tested using tools like Postman or simple curl commands. Try calling your add tool with different number combinations to verify correct responses. At 4:32 in the video tutorial, you can see a live demonstration of testing both server types.

Key testing considerations:

  • Verify all tools respond with expected outputs
  • Test error handling with invalid inputs
  • Check performance under concurrent requests
  • Validate security measures for HTTP servers
