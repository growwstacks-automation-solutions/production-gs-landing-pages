The Problem with Current AI Assistants

Most AI coding assistants today operate without real context about your running applications. You give them prompts and maybe some code, but they're essentially making educated guesses based on that limited information. This becomes painfully obvious when dealing with performance issues that only manifest under real-world conditions.

Consider a common scenario: You have two API endpoints that look nearly identical in code - one uses Thread.Sleep() while the other uses Task.Delay(). Under light load, both perform similarly. But under concurrent load, the Thread.Sleep() endpoint can bring your entire API to its knees by blocking thread pool threads. This kind of problem is nearly impossible for an AI to diagnose without runtime data.

Key insight: AI assistants today are like doctors trying to diagnose a patient's condition without being able to run any tests - they can make educated guesses based on symptoms you describe, but they can't actually examine the patient.

What is MCP (Model Context Protocol)?

MCP stands for Model Context Protocol - a way to give AI agents real tools they can call to interact with your applications. Instead of just answering questions from memory, an MCP-enabled AI can:

  • Run actual load tests against your endpoints
  • Compare performance between different implementations
  • Analyze results and detect patterns
  • Generate detailed reports with optimization suggestions

The protocol acts as a bridge between the AI's reasoning capabilities and your application's runtime behavior. You expose specific tools through an MCP server (in this case built with .NET), and the AI client (like GitHub Copilot) can call those tools when appropriate.

Implementing MCP Server in .NET

At its core, an MCP server is just a regular ASP.NET Core application with some additional configuration:

 // Program.cs builder.Services.AddMcpServer(httpTransport => {     httpTransport.MapMcpServer("/mcp"); }) .WithToolsFromAssembly(Assembly.GetExecutingAssembly());

The magic happens in the tools you expose. Each tool is a C# method marked with the [McpTool] attribute and includes a detailed description that the AI uses to understand when to call it:

 [McpTool("Run a load test against an API endpoint")] public async Task<LoadTestResult> RunLoadTest(     [McpParam("URL of endpoint to test")] string url,     [McpParam("Test duration in seconds")] int durationSeconds,     [McpParam("Number of concurrent users")] int concurrentUsers) {     // Implementation that runs the test and returns metrics     return new LoadTestResult {         Throughput = ...,         LatencyPercentiles = ...,         ErrorRate = ...,         ResultId = ...     }; }

Designing Effective MCP Tools

The quality of your MCP implementation depends heavily on how well you design your tools. Good MCP tools should:

  • Have clear, descriptive names that match their purpose
  • Include detailed parameter descriptions that explain expected values
  • Return structured data that the AI can reason about
  • Be atomic - each tool should do one specific thing well

In the demo, tools are organized around the performance testing workflow:

  1. run_load_test: Executes a test against an endpoint
  2. compare_endpoints: Runs tests against two endpoints and compares results
  3. analyze_results: Examines test data for patterns and anomalies
  4. generate_report: Creates a markdown summary with findings
  5. suggest_optimizations: Recommends improvements based on test data

Tool design tip: Treat your MCP tools like an API contract for the AI - they should be as well-documented and predictable as any other public API your team maintains.

Real-World Test Results

When running the demo with 20 concurrent users for 15 seconds:

  • The Thread.Sleep endpoint averaged 569ms latency with P95 at 1 second
  • The Task.Delay endpoint performed significantly better with lower latency across all percentiles
  • The AI could clearly identify that Thread.Sleep was blocking thread pool threads
  • A detailed markdown report was generated comparing the two implementations

All these results were visible both in the AI client interface and in the Blazor dashboard, demonstrating how MCP creates a bridge between AI reasoning and human analysis.

Key finding: The difference between Thread.Sleep and Task.Delay is nearly invisible in code inspection but becomes obvious under load testing - exactly the kind of insight MCP is designed to surface.

An MCP server in .NET is built as an ASP.NET Core application that exposes tools over HTTP. The implementation involves marking specific methods with MCP attributes that describe what the tool does.

Each tool method should have clear parameters and return types that the AI can understand. The server configuration maps these tools to an endpoint that AI clients can connect to, typically using HTTP as the transport protocol.

  • ASP.NET Core application
  • Methods marked with MCP attributes
  • HTTP endpoint for client connections

Tool descriptions are critical because the AI uses them to decide which tool to call. A good description clearly explains what the tool does, its parameters, and what it returns.

For example, a load test tool description would specify it accepts URL, duration, concurrent users and returns throughput, latency percentiles, error rate and result ID. The more precise the description, the better the AI can determine when to use the tool.

  • Clear purpose statement
  • Parameter descriptions
  • Expected return values
