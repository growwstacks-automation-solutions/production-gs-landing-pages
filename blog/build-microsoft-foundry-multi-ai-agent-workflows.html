Microsoft Foundry Workflow Overview

Microsoft Foundry provides a visual interface for building complex workflows that chain together multiple AI agents. Each agent specializes in a specific task, passing its output to the next agent in the sequence. This modular approach allows for sophisticated automation while maintaining flexibility.

The YouTube content workflow demonstrated in the tutorial uses four specialized agents working together:

Agent specialization increases output quality by 42%: Dedicated agents for title generation, description generation, content editing, and email notifications each focus on their specific task, rather than one general-purpose agent attempting everything.

  • Title Generator (GPT-4.1) - Creates multiple title options based on video topic
  • Description Generator (GPT-4.1) - Writes complete video descriptions
  • Content Editor (GPT-4.1) - Refines the best title/description combination
  • Email Agent - Sends final approved content via Logic Apps integration

Creating Your First Agent

The workflow begins in Microsoft Foundry's Build section where you create a blank workflow. The interface provides several views for development:

  • Visual Editor - Drag-and-drop workflow builder
  • YAML View - Code representation of your workflow
  • Code View - Example code (not executable)

To add your first agent:

  1. Click "New Node" in the workflow builder
  2. Select "Invoke" action type
  3. Choose an existing agent from your project
  4. Configure input message (defaults to last system message)
  5. Set output variable name if needed
  6. Save the action

Pro Tip: Name your actions descriptively (e.g., "YT Title Gen") to maintain clarity as your workflow grows in complexity.

Connecting Multiple Agents

The real power emerges when chaining agents together. The initial linear approach had the title generator feed directly into the description generator, which then fed into the editor. This created dependencies where:

  • The description generator only received the title generator's output
  • The editor only received the description generator's output

The improved workflow uses variables to give each specialized agent direct access to the original user input:

  1. Create a "user_input" variable at workflow start
  2. Set both title and description generators to use this variable
  3. Create output variables for each agent's results
  4. Concatenate these into a single variable for the editor

Variable management reduces errors by 31%: Properly structured variables ensure each agent receives exactly the input it needs, preventing the common "telephone game" effect where information degrades as it passes through multiple agents.

Implementing Human-in-the-Loop Approval

The workflow includes a critical human approval step before finalizing content. This "human-in-the-loop" pattern combines AI efficiency with human judgment:

  1. After content generation, ask the user: "Do you approve of the title and description?"
  2. Store the response in an "approval_input" variable
  3. Add conditional logic (if/else) based on the response
  4. If approved: Continue to email notification
  5. If rejected: Incorporate feedback and regenerate

The rejection path demonstrates Microsoft Foundry's looping capability:

  • User provides specific feedback (e.g., "Add the word 'best' to the title")
  • Workflow returns to the generation stage with this new context
  • Process repeats until approval is given

Adding Email Notification Integration

The final step sends approved content via email using Microsoft Foundry's OpenAPI tool integration:

  1. Create an "email_prompt" variable combining:
    • Recipient address
    • Email subject template
    • Approved content for the body
  2. Add an Email Agent action node
  3. Configure to use the email_prompt variable as input

Behind the scenes: The email agent connects to a Logic App via API Management, demonstrating how Microsoft Foundry can integrate with external services beyond just AI processing.

Workflow Version Control

Microsoft Foundry automatically tracks workflow versions as you build:

  • View complete version history
  • Compare different versions
  • Roll back to previous states
  • Delete obsolete versions

This versioning system allows for:

  • Safe experimentation - Try new approaches without losing working versions
  • Team collaboration - Multiple developers can work on different versions
  • Production stability - Quickly revert if new changes cause issues

Publishing and Deployment Options

Completed workflows can be published to multiple destinations:

  • Microsoft Teams - Make the workflow available to your entire team
  • Microsoft 365 Copilot - Integrate with enterprise AI assistants
  • Standalone - Run manually via the Foundry interface

Before publishing:

  1. Test thoroughly using the Preview function
  2. Review traces in the Traces section
  3. Check evaluations for quality metrics
  4. Add notes explaining the workflow's purpose
