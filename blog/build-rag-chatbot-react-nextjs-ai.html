Why RAG Solves the "Blank AI" Problem

Traditional AI chatbots hit a wall when asked about internal company information. While ChatGPT can explain quantum physics, it draws a blank on questions like "What's our vacation policy?" or "How do I submit an expense report?" This creates frustration as employees toggle between AI tools and internal document searches.

Retrieval Augmented Generation (RAG) bridges this gap by combining semantic search with AI generation. When an employee asks about vacation policy, the system:

  1. Searches your document repository for relevant policy excerpts
  2. Feeds those exact passages to the AI
  3. Generates an answer grounded in your actual policies

Key benefit: RAG systems maintain accuracy while being instantly updatable. When HR updates the vacation policy PDF, the chatbot automatically provides the new rules without retraining.

PDF to Vector Embeddings Pipeline

The magic happens when documents transform into searchable embeddings. Our pipeline processes any uploaded PDF through four key steps:

1. Text Extraction

Using pdf-parse, we extract raw text while preserving document structure. A 20-page employee handbook becomes ~15,000 characters of searchable content.

2. Semantic Chunking

We split text into 150-character chunks with 20% overlap using LangChain's recursive splitter. This balances precision (small chunks) with context (overlap).

Production tip: Adjust chunk size based on document type. Policy manuals work best at 150-200 chars while technical docs may need 500-1000.

3. Embedding Generation

Each chunk converts to a 1536-dimension vector using OpenAI's text-embedding-3-small model. We batch process chunks to minimize API calls.

4. Vector Storage

Chunks and embeddings store in Neon Postgres with an HNSW index for fast similarity searches (5-50ms response times).

Implementing Semantic Search

When a user asks "What's our vacation policy?", the system:

  1. Converts the question to an embedding vector
  2. Searches for the 3 most similar document chunks
  3. Returns matches scoring above 0.5 similarity threshold

The cosine similarity calculation happens entirely in the database:

Technical insight: We subtract cosine distance from 1 to get similarity score (0 = unrelated, 1 = identical meaning). An HNSW index makes this fast even with thousands of documents.

Results format neatly for the AI to reference:

 [1] Employees receive 15 PTO days annually for years 0-2... [2] Vacation requests require 2-week advance notice via HR portal... [3] Unused PTO carries over up to 5 days per year...

Adding Authentication & Access Control

Internal tools need strict access controls. We implement:

1. Clerk Authentication

Prebuilt sign-in/sign-up flows with email, Google and GitHub providers. Protects all routes except the homepage.

2. Role-Based Access

User metadata defines 'admin' and 'user' roles. Only HR admins can upload documents while all employees can chat.

Implementation note: Clerk sessions include role metadata after configuring customized session tokens in the dashboard.

3. Middleware Protection

Next.js middleware redirects non-admins from /upload to homepage with no flash of protected content.

Building the Streaming Chat Interface

3. Knowledge Tool Integration

The route handler automatically invokes our search tool when questions might reference uploaded documents.

Production Deployment Considerations

Scaling the Vector Database

Neon's serverless Postgres handles ~10,000 document chunks comfortably. For larger deployments:

  • Upgrade to dedicated Neon tier
  • Consider pgvector with pg_bm25 for hybrid search
  • Implement cache layer for frequent queries

Document Management

Add these features for real-world use:

  • Version control for policy updates
  • Bulk delete outdated documents
  • Usage analytics to improve chunks

Pro tip: Set up CI/CD to automatically re-embed documents when OpenAI releases new embedding models for accuracy boosts.

Vector embeddings convert text into numerical representations that capture meaning. This allows semantic search - finding documents based on conceptual similarity rather than just keyword matching.

For example, searching for "vacation policy" would also find documents mentioning "PTO guidelines" or "time off procedures" even if the exact phrase "vacation policy" never appears. The embedding model understands these concepts are related.

  • 1536-dimensional vectors capture nuanced relationships
  • HNSW indexing enables fast similarity searches
  • Outperforms keyword search for policy questions by 40%

Breaking documents into 150-500 character chunks with 20% overlap improves search accuracy by allowing precise retrieval of relevant sections. Large documents would otherwise return irrelevant matches.

The overlap maintains context between chunks. If a policy spans two chunks, the 20% overlap ensures the AI sees the full context rather than a disjointed snippet.

  • Small chunks increase answer precision
  • Overlap preserves document flow
  • Optimal size varies by document type

When documents change, you reprocess them through the same pipeline: extract text, chunk content, generate embeddings, and update the vector database. The system automatically uses the latest versions since it searches fresh on each query rather than relying on training data.

For version control, you can either overwrite old embeddings or maintain multiple versions. Overwriting is simpler for most policy documents. Versioning makes sense for reference materials where users might ask "What was the policy in 2024?"
