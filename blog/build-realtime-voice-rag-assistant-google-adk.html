The ADK Framework + File Search Combo

Traditional RAG (Retrieval Augmented Generation) systems require stitching together multiple components: document processing pipelines, vector databases, embedding models, and custom frontends. Google's newly released ADK (Agent Development Kit) framework and file search tools eliminate this complexity by providing managed solutions for each layer.

The ADK web UI components give you a complete interface out of the box - including text chat, voice interaction, file uploads, and debugging tools. Meanwhile, the file search tool handles document processing automatically, charging only $0.15 per million tokens during initial indexing (with free storage and query-time embeddings).

Key advantage: This combination means you can build a production-ready voice RAG assistant focusing solely on the agent logic - no frontend development, audio processing, or vector database management required.

System Architecture Overview

The demo system shown in the video implements three specialized agents that work together to handle document uploads and voice queries:

  1. File Manager Agent: Handles document uploads with tools to list and index files in the Gemini file search store
  2. Search Assistant Agent: Performs semantic search using Gemini's native audio model for voice responses
  3. Root Agent: Orchestrates requests between other agents based on input type (file uploads vs queries)

When a user uploads a PDF through the ADK web UI (like the New York City Marathon guide shown in the demo), the system automatically routes it to the File Manager Agent for indexing. Voice queries then get processed by the Search Assistant Agent which retrieves relevant passages and generates spoken responses with citations.

Building the Orchestrator Agent

The orchestrator is implemented as a custom base agent that overrides the standard ADK agent behavior. Its key responsibility is routing:

  • Detects file uploads in message parts (coming from the ADK UI upload button)
  • Routes file operations to the File Manager Agent
  • Directs queries to the Search Assistant Agent

This routing logic happens in the run_async method for text interactions and run_live for voice mode. The orchestrator checks for keywords like "upload" or examines inline data attachments to determine where to route each request.

Implementation note: The orchestrator doesn't need any tools itself - it's purely a coordination layer that leverages the specialized capabilities of the other agents.

File Upload & Indexing Logic

The File Manager Agent implements two critical tools for document processing:

  1. List Files Tool: Checks session history for uploaded files (since ADK UI uploads arrive as inline data rather than artifacts)
  2. Index File Tool: Handles uploading to Gemini's file search store with duplicate checking

Duplicate prevention works by maintaining a JSON file that tracks all file operations. Before indexing, the system checks if the document already exists in the file search store. If found, it uses the existing index instead of re-processing - saving both time and indexing costs.

At 7:32 in the video, you can see this in action when uploading the marathon guide PDF - the agent first checks if the file needs indexing before proceeding with the search interaction.

Implementing Voice Interaction

The voice capability comes from overriding the run_live method in the orchestrator agent. When a voice query is detected:

  1. Checks for uploaded files in the session
  2. Routes to File Manager Agent if indexing is needed
  3. Passes the query to Search Assistant Agent which uses Gemini's audio model
  4. Generates spoken responses with document citations

The Search Assistant Agent is configured with specific instructions to maintain a natural, friendly speaking style. It strictly uses the search tool to answer document-related questions, ensuring responses are grounded in the uploaded content (as seen at 4:18 when asking about the marathon route).

All audio processing - encoding, decoding, streaming - is handled automatically by the ADK web UI components, eliminating what would normally require significant custom development.
