The Stale RAG Problem

Every AI team faces the same frustrating scenario: You upload updated policies to Google Drive, but your customer support bot keeps giving yesterday's answers. Traditional RAG implementations force you to choose between completely rebuilding your vector database (losing all context history) or living with information.

The breakthrough came when we realized we could use Supabase's active/inactive flag system create versioned embeddings. This allows AI agents to access current information while preserving historical context for audit purposes - all automated through n8n.

90% of AI answer inaccuracies stem from stale knowledge bases. Our clients who've implemented this pipeline see 73% reduction in incorrect responses while maintaining complete version history.

Pipeline Architecture Overview

This solution uses four interconnected pipelines that work together maintain your knowledge base automatically:

1. Create Pipeline

Watches your designated Google Drive folder for new documents. When detected, downloads the workflow:

  • Downloads the file
  • Generates embeddings using OpenAI
  • Inserts records into Supabase with is_active=true

2. Update Pipeline

Monitors for changes to existing documents. When triggered, it:

  • Marks previous versions of this document as is_active=false
  • Downloads the updated file
  • Creates new embeddings
  • Inserts them with is_active=true

3. Archive Pipeline

Watches a designated "trash" folder. When documents are moved here:

  • All related embeddings for that document are set to is_active=false

4. Health Check Pipeline

Runs daily to:

  • Scan Google Drive for all files
  • Compare against Supabase records
  • Upload any missing documents

Create Pipeline: New Documents

The create pipeline solves the most basic RAG challenge: getting new content into your vector database. Configured to watch your primary knowledge base folder, it springs into action whenever new files appear.

At 2:35 in the video tutorial, you can see the Google Drive trigger configuration. We recommend:

  • Setting the poll time to every 5 minutes (balance between responsiveness and API quota)
  • Configuring to watch only your designated knowledge base folder
  • Filtering for specific file types if needed (PDFs, DOCs, etc.)

Pro Tip: Add metadata like department tags or document categories during the initial upload. This information gets stored in Supabase's JSONB column and can be used for advanced filtering later.

Update Pipeline: Changed Documents

The update pipeline addresses the most common pain point - policy changes that aren't propagate to your AI agents. Unlike brute-force solutions that wipe and rebuild, our approach maintains version history.

Key components:

  1. Google Drive Trigger: Watches for "updated" events in your knowledge base folder
  2. Supabase Update: Finds all embeddings for this document and sets is_active=false
  3. Embedding Generation: Processes the updated file with text-embedding-3-small
  4. New Insert: Creates fresh embeddings with is_active=true

This approach gives you complete version control - you can see what information was active during any period, while ensuring current queries only use the latest information.

Archive Pipeline: Removed Documents

Sometimes documents need to be retired entirely. Rather than deleting them (which breaks historical queries), we move them to designated trash folder to trigger archival.

The archive pipeline:

  • Uses a separate Google Drive trigger watching your trash folder
  • Finds all embeddings matching the document name
  • Sets their is_active flag to false

Compliance Note: Archived documents remain searchable with explicit filters (is_active=false), making this system ideal for regulated industries needing retention policies.

Daily Health Check Reconciliation

The health check pipeline is your safety net - it ensures no document slips through the cracks. Running once per day, it:

  1. Scans your knowledge base folder for all files
  2. Compares against Supabase records
  3. Identifies any files missing active embeddings
  4. Processes and uploads them automatically

Supabase Database Configuration

Proper Supabase setup is crucial for this pipeline success. You'll need:

1. Documents Table

 CREATE TABLE documents (   id UUID PRIMARY KEY DEFAULT gen_random_uuid(),   content TEXT,   metadata JSONB,   embedding vector(1536),   is_active BOOLEAN DEFAULT true );

2. Match Documents Function

 CREATE OR REPLACE FUNCTION match_documents (   query_embedding vector(1536),   match_count INT DEFAULT 20,   filter JSONB DEFAULT NULL ) RETURNS TABLE (   id UUID,   content TEXT,   metadata JSONB,   similarity FLOAT ) LANGUAGE plpgsql AS $$ BEGIN   RETURN QUERY   SELECT     id,     content,     metadata,     1 - (embedding <=> query_embedding)::FLOAT AS similarity   FROM documents   WHERE is_active = true   ORDER BY embedding <=> query_embedding   LIMIT match_count; END; $$;

This configuration ensures your AI agents only search active embeddings while maintaining full version history.

n8n Workflow Configuration

The complete n8n workflow requires several key configurations:

Google Drive Nodes

  • Separate triggers for create/update/archive folders
  • Proper credential setup with appropriate scopes
  • File download nodes configured to handle your document types

Supabase Nodes

  • Correct URL and service role key
  • Proper table name (documents) and match function (match_documents)
  • Batch size of 200 chunks per operation

OpenAI Embedding

  • text-embedding-3-small model selected
  • Proper credential configuration
  • Error handling for rate limits
