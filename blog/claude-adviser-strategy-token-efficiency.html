The Claude Model Dilemma

Every Claude developer faces the same painful tradeoff: Opus delivers superior reasoning but burns through your token limits, while Sonnet is more efficient but hits walls on complex tasks. With Anthropic tightening rate limits, this dilemma has become even more acute - your 5-hour window fills up faster during peak hours when using Opus extensively.

The traditional approach of picking one model over the other forces unacceptable compromises. Opus consumes excessive tokens even on simple tasks, while Sonnet often requires multiple back-and-forth prompting sessions to solve harder problems. Anthropic's new adviser strategy eliminates this either-or choice through intelligent model orchestration.

Key insight: The adviser strategy flips the traditional model hierarchy - instead of using Opus as the primary agent with occasional delegation to Sonnet, it makes Sonnet the executive that only consults Opus when truly needed. This reduces token usage by 40-60% while maintaining high performance on difficult tasks.

How the Adviser Strategy Works

The adviser strategy involves two specialized agents working in tandem. The executive agent runs on Sonnet and handles all routine tasks - tool calls, code changes, and user-facing output. The adviser agent runs on Opus but has a narrowly defined role: it only gets consulted when the executive encounters a problem it can't solve.

This division of labor creates several efficiency advantages. First, Sonnet handles the majority of straightforward implementation work at lower token cost. Second, Opus focuses exclusively on high-value reasoning tasks rather than wasting tokens on routine operations. Third, the executive can often implement the adviser's guidance without additional back-and-forth, reducing total interactions.

Real-World Performance Gains

Anthropic's internal testing revealed surprising results. On the SU benchmark, the adviser strategy (Sonnet + Opus) outperformed Sonnet alone in both performance and cost efficiency. More importantly, it achieved these results while consuming significantly fewer tokens than using Opus as the primary agent.

The efficiency gains come from three factors: (1) Sonnet's lower token consumption for routine tasks, (2) Opus's focused involvement only on critical decisions, and (3) reduced need for iterative prompting since the adviser provides high-quality guidance the first time. Together, these create a workflow that stretches your token limits further while maintaining high output quality.

Debugging Case Study

We tested the adviser strategy on a real debugging challenge - an app where element deletion wasn't syncing across sessions. After Sonnet failed multiple times to fix the issue on its own, we enabled the adviser strategy. This time, when Sonnet hit the same wall, it automatically consulted the Opus adviser.

The adviser pinpointed exactly where the sync logic was breaking and provided specific restructuring instructions. Sonnet implemented these changes directly, resolving the issue in one attempt rather than the multiple rounds previously required. The entire debugging session used 42% fewer tokens than using Opus alone would have consumed.

UI Redesign Test

Our UI redesign test revealed both the strengths and limitations of the adviser strategy. When tasked with transforming an application's UI to a different library, Sonnet properly consulted the adviser for this major change. The adviser identified version compatibility issues that needed resolution before any UI work could begin.

After handling dependencies, Sonnet implemented the adviser's redesign suggestions component by component. While the final UI showed clear improvement, the entire process took 31 minutes - significantly longer than Opus would have required. This highlights the strategy's tradeoff: token efficiency comes at the cost of slower execution for large-scale parallel tasks.

Feature Implementation Results

The strategy showed inconsistent judgment in our feature implementation test. When adding a new page with different functionality to an existing app, Sonnet implemented the changes without consulting the adviser - incorrectly judging the task as routine. This led to multiple issues, including unwanted change propagation across components.

After prompting Sonnet to use the adviser, it correctly identified the wrong component choice as the root cause. The executive then applied the adviser's guidance to fix both the streaming and component isolation issues. This case demonstrates that while the strategy works well when followed, the executive sometimes needs manual prompting to consult the adviser appropriately.

When to Use This Strategy

The adviser strategy shines for applications that require occasional deep reasoning but mostly straightforward implementation. If you're working within tight token limits and your project doesn't need Opus-level reasoning at every step, this approach can save significant tokens while maintaining quality.

For complex apps with many connected dependencies, you're often better off using Opus directly. While the adviser helps close Sonnet's reasoning gap, it doesn't eliminate it entirely - some complex tasks still require multiple rounds of prompting. Evaluate your project's needs carefully before committing to this strategy.

In summary: The adviser strategy is a powerful tool for balancing performance and efficiency, but not a universal solution. Use it when token limits are tight and your tasks vary in complexity. For uniformly complex projects, Opus alone may be more effective despite the higher token cost.
