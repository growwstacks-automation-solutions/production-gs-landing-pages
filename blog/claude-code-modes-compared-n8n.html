Claude Code Modes Explained

Many n8n users don't realize Claude Code offers four distinct interaction modes, each with different levels of autonomy and oversight. This lack of awareness leads to suboptimal workflow creation - either too much micromanagement or too many assumptions by the AI.

We tested all four modes with the exact same prompt to eliminate variables. The modes are:

Key Insight: The modes exist on a spectrum from maximum autonomy (Bypass Permissions) to maximum oversight (Plan mode). Surprisingly, Bypass Permissions still asked key questions rather than making poor assumptions, while Plan mode delivered 3-4x more documentation than other modes.

  1. Bypass Permissions: Claude operates with maximum autonomy, making changes without asking. Surprisingly, it still asked key questions about file formats and delivery methods.
  2. Edit Automatically: Claude makes changes without asking but will request approval for bash commands and API connections.
  3. Ask Before Edits: Claude requests approval for each significant change to the workflow.
  4. Plan Mode: Claude creates a detailed plan including architecture diagrams before building anything. This mode asks strategic questions upfront.

Test Workflow Overview

To compare the modes fairly, we used the same prompt across all tests: an n8n workflow that would automatically screen job candidates against a job spec and shortlist the top 3-5 candidates.

The workflow needed to:

  • Trigger when files are added to a Google Drive folder
  • Handle both PDF and Word document resumes
  • Use OpenAI to analyze resumes against the job spec
  • Score candidates from 1-100
  • Send a Slack message with the top candidates and analysis

This real-world business automation scenario tests each mode's ability to handle complex requirements while minimizing errors.

Bypass Permissions Results

Bypass Permissions mode completed the workflow fastest - in about half the time of other modes. Contrary to expectations, it didn't just make wild assumptions but asked key questions:

  • Where will files be stored (Google Drive/OneDrive)?
  • What format will resumes be in (PDF/Word/mix)?
  • Which AI service to use (OpenAI/Anthropic/Gemini)?
  • How to deliver results (Slack/email)?

The resulting workflow was functional but had a critical architecture error - it placed the data processing node inside rather than after the file download loop. This would cause unnecessary processing on each iteration.

Surprise Finding: Even in maximum autonomy mode, Claude Code demonstrated restraint by asking key questions rather than making poor assumptions. This makes Bypass Permissions useful for rapid prototyping when you need a working version fast.

Edit Automatically Results

Edit Automatically mode asked for approval on bash commands and API connections but otherwise proceeded without confirmation. It completed the workflow slightly slower than Bypass Permissions but with similar quality.

Like Bypass Permissions, it made the same loop architecture error. The workflow was nearly identical except it used a basic LLM chain node instead of a direct OpenAI message node.

Key observations:

  • Asked 2-3 more questions than Bypass Permissions (mainly about API connections)
  • Produced similar documentation quality
  • Made the same loop architecture mistake
  • Took about 20% longer than Bypass Permissions

This mode offers little advantage over Bypass Permissions for initial workflow creation but may be safer for API-heavy workflows.

Ask Before Edits Results

Ask Before Edits mode theoretically offers more control, but in practice asked only slightly more questions than Edit Automatically mode. It did, however, produce a workflow with one key improvement:

It correctly structured the loop architecture, placing the data processing node after rather than inside the file download loop. This shows the mode's potential for catching errors when given the chance.

Best Use Case: Ask Before Edits shines when making iterative improvements to an existing workflow. Once the core architecture is validated (perhaps using Plan mode), switching to Ask Before Edits allows for safer incremental changes.

The mode still lacked the comprehensive documentation of Plan mode and took nearly as long to complete the workflow.

Plan Mode Results

Plan mode delivered by far the highest quality workflow. Before writing any code, it created:

  • A detailed workflow architecture diagram
  • Google Drive folder structure plan
  • Node-by-node configuration documentation
  • OpenAI prompt templates
  • Slack message formats

Critically, it was the only mode to get the loop architecture correct on the first attempt. It also properly handled both PDF and Word documents (other modes only implemented PDF extraction).

The comprehensive planning upfront resulted in:

  • 50% fewer architectural errors
  • 3-4x more documentation
  • Better organized node structure
  • Clearer implementation instructions

While it took the longest to produce the initial workflow, the time invested upfront likely saves debugging time later.

When to Use Each Mode

Based on our testing, here's when to use each Claude Code mode in n8n:

Pro Tip: Start with Plan mode for the initial workflow design, then switch to Ask Before Edits or Bypass Permissions for iterative improvements once the core architecture is validated.

  1. Plan Mode: Use for any production workflow or complex automation. The upfront planning prevents architectural errors and creates documentation others can follow.
  2. Ask Before Edits: Best for making incremental improvements to an existing workflow after the core architecture is validated.
  3. Bypass Permissions: Ideal for rapid prototyping when you need a working version fast and can refine it later.
  4. Edit Automatically: Least recommended - offers few advantages over other modes while still allowing significant errors.
