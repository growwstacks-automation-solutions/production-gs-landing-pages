Agent Teams vs Sub-Agents: The Key Differences

For months, Claude Code users relied on sub-agents - parallel workers that executed tasks independently but lacked coordination. The new agent team feature changes this fundamentally by enabling real-time communication between agents. Imagine a development team where programmers never talk to each other versus one where they constantly collaborate - that's the difference between sub-agents and agent teams.

In practical terms, this means when one agent encounters an unexpected problem (like a bug or missing dependency), it can immediately consult with other team members to adjust the workflow. The video demonstrates this beautifully when building a Next.js website - the layout designer communicates directly with the data layer builder to ensure compatibility.

Key insight: Agent teams completed a Next.js golf tournament directory build in just 4 minutes - a task that previously took 40 minutes with sub-agents. This 10x speed improvement comes from dynamic coordination rather than sequential execution.

The Setup Process for Agent Teams

Getting started with agent teams requires three key components: Claude Opus 4.6 (the latest model), tmux terminal multiplexer, and enabling experimental features in your Claude Code settings. The tmux requirement is particularly interesting - it provides the visual interface that lets you monitor multiple agents working simultaneously.

The setup process involves:

  1. Installing tmux (brew install tmux on Mac)
  2. Creating a new Claude Code session
  3. Adding "claude.experimental.teams: true" to your settings.json
  4. Starting a tmux session (just type 'tmux')

What's remarkable is how Claude Code itself can guide you through this setup if you simply ask it to "please set up agent swarm/teams." The system will locate your settings file, make the necessary modifications, and verify tmux is installed - showcasing its self-configuration capabilities.

Current Limitations and Considerations

While revolutionary, agent teams aren't without challenges. The current implementation works best with 3-5 agents - attempting larger teams can lead to coordination issues. There's also a learning curve to effectively monitoring and directing multiple agents through tmux.

Another consideration is cost - running multiple Opus 4.6 agents simultaneously consumes significant tokens. The golf tournament demo used approximately 30 billion tokens (as mentioned in the video), which could be cost-prohibitive for some applications.

Pro tip: Start with small agent teams (2-3 members) focused on discrete tasks before scaling up to more complex workflows. This allows you to identify coordination patterns that work for your specific use case.
