API Requirements and Setup

Before connecting ElevenLabs with Cal.com, you'll need three key pieces of information from your Cal.com account. Without these, the integration won't work properly.

First, generate an API key specifically for this integration. Navigate to Settings → API Keys in your Cal.com dashboard and create a new key. Second, identify your event type ID from the URL when viewing your event types. Third, note the required API version headers which differ between the availability check and booking endpoints.

Essential Components:

  • Cal.com API key (keep this secure)
  • Event Type ID (numeric value from your event URL)
  • Two different API version headers
  • Content-Type: application/json header for POST requests

Handling Time Zones Correctly

Time zone issues cause most meeting scheduling problems. The integration must properly convert between the caller's local time and UTC for the API.

Always collect the caller's time zone (like America/Sao_Paulo) and include it in the booking request. Cal.com will handle the conversion, but your Voice AI should confirm the time in the caller's local time zone.

Time Zone Best Practices:

  1. Explicitly ask for time zone if not detectable from phone number
  2. Use IANA time zone database format (Continent/City)
  3. Confirm the converted time back to the caller
  4. Include time zone in the calendar invite subject

Common Errors and Troubleshooting

Even with proper setup, you might encounter issues. These are the most common problems and how to fix them.

API version mismatches cause 40% of integration failures. Double-check you're using 2023-01-01 for availability checks and 2023-04-17 for bookings. Other frequent issues include:

  • 401 Unauthorized: Usually means your API key is invalid or missing
  • 404 Not Found: Typically indicates an incorrect eventTypeId
  • 400 Bad Request: Often caused by malformed time strings or missing fields
  • 409 Conflict: Means the time slot is already booked

The Cal.com API requires all time parameters in ISO 8601 format in UTC timezone. This standardized format ensures consistent time handling across different systems and locations.

Dates should be formatted as YYYY-MM-DD and times should include the timezone offset (Z for UTC). For example: 2024-03-15T15:00:00Z represents March 15, 2024 at 3:00 PM UTC. Your Voice AI agent will need to convert local time requests to this format before making API calls.

  • Always use UTC to avoid timezone confusion
  • Include seconds in the time string
  • The 'T' separates date from time
