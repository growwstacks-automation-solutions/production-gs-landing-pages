The Problem With Generic Chatbots

Most AI chatbots for e-commerce fall into one of two traps: either they're simple ChatGPT wrappers that hallucinate product details and prices, or they're rigid decision-tree systems that can't handle natural conversation flow. Both approaches fail when customers ask follow-up questions like "What about the second one?" or "How do I use this with my sensitive skin?"

The solution demonstrated in this tutorial combines the flexibility of large language models with the reliability of structured product data. By strictly separating data retrieval from response generation, we prevent AI hallucinations while maintaining natural conversation flow.

Key insight: Generic chatbots fail e-commerce use cases because they either invent product details or can't maintain conversation context. A production-ready solution needs both retrieval-augmented generation and explicit intent handling.

Solution Architecture

The chatbot system has three main components working together: a Telegram frontend connected via n8n, a FastAPI backend handling the AI logic, and a vector database storing product information. Each part plays a specific role in delivering accurate, context-aware responses.

The architecture deliberately separates concerns: n8n handles platform integration and message routing, FastAPI manages conversation state and intent detection, while the vector database ensures only verified product data is used in responses. This separation makes the system both more reliable and easier to debug.

Data Ingestion Pipeline

The system uses two complementary data sources: structured JSON files containing product metadata (names, prices, URLs) and markdown documents with richer descriptions and usage guidelines. During ingestion, we carefully chunk the data so all information about a single product stays together.

This chunking strategy prevents common retrieval problems where prices get separated from products or URLs go missing. Each product chunk includes both the structured metadata and relevant descriptive text, ensuring the chatbot can answer both specific ("What's the price?") and open-ended ("Is this good for sensitive skin?") questions accurately.

Conversation Memory System

The chatbot maintains conversation context through a combination of short-term memory (the current conversation) and long-term memory (product data). When a user refers to "the second one" or asks follow-up questions about pricing, the system tracks which products were previously discussed.

This memory system is implemented through FastAPI's session management, with n8n passing along conversation IDs with each message. The result is natural dialogue flow without requiring users to repeat themselves, while still keeping all responses grounded in verified product data.

Intent Detection Logic

The system uses explicit intent detection to handle common e-commerce scenarios like purchase follow-ups or price inquiries. Rather than relying solely on the language model to interpret these requests, we use deterministic logic to identify and handle specific intent patterns.

For example, when a user asks "I want to buy the second one," the system first confirms which product they're referring to using conversation memory, then retrieves the exact price and URL from the structured JSON data. This approach prevents the AI from guessing or inventing information.

n8n Integration Flow

n8n serves as the glue between Telegram and our FastAPI backend. Its workflow captures incoming messages, forwards them to the API, then delivers the structured responses back to users. The n8n execution logs provide complete visibility into every step of this process.

One major advantage of using n8n is its cloud execution - the chatbot continues working even when your local development machine is off. The workflow also handles error cases gracefully, with automatic retries and fallback responses if the backend is temporarily unavailable.

Production Deployment

The FastAPI backend runs on a Linux server using systemd for process management. This ensures automatic restarts if needed and continuous operation without hot-reloading that could disrupt conversation memory. The entire system remains available 24/7 with minimal maintenance.

For scaling, the architecture allows each component to be deployed independently. The vector database can be scaled separately from the API servers, and multiple n8n workers can handle increased message volume. This makes the solution suitable for both small shops and larger e-commerce operations.
