Step-by-Step: Formatting Timestamps Correctly

Let's walk through implementing this in a real workflow. We'll use the call center example from the video where call timestamps need to display in the local timezone.

Step 1: Identify Your Timestamp Source

In our example, we're using a Vapi webhook that provides the call start time in UTC. This is common with most APIs - they return timestamps in UTC by default.

Step 2: Use the Date Format Function

In your Make.com scenario, add a "Format date" operation. The function syntax is:

formatDate(timestamp; formatPattern; timezone)

Note the semicolons between parameters - this is crucial and different from typical programming languages.

Step 3: Specify Your Timezone

Use the IANA timezone format (like "America/New_York"). You can find these in Make.com's helper tooltip or in the documentation.

Step 4: Set Your Display Format

Common formats include:

  • "MM/dd/yyyy hh:mm a" → 12/01/2025 06:13 PM
  • "yyyy-MM-dd HH:mm" → 2025-12-01 18:13
  • "EEE, MMM d, h:mm a" → Sun, Dec 1, 6:13 PM

Common Timezone Mistakes to Avoid

Even with the new solution, there are pitfalls to watch for:

  1. Using commas instead of semicolons: The date function parameters must be separated by semicolons, not commas.
  2. Incorrect timezone names: "PST" won't work - you need "America/Los_Angeles".
  3. Hardcoding formats: Consider making the format pattern a variable if you need flexibility.
  4. Assuming UTC: Always verify your source data's timezone - not all APIs use UTC.

Common questions about this topic

Timezone issues occur because Make.com processes all timestamps in UTC (Coordinated Universal Time) by default. When this data gets displayed in applications like Google Sheets or CRM systems, it often shows the wrong local time unless properly converted.

The UTC timestamp might show as 1:13 AM when it should display as 6:13 PM in your local timezone. This discrepancy happens because the receiving application doesn't know how to interpret the UTC timestamp without explicit timezone information.

  • Make.com uses UTC internally for consistency
  • Most APIs return timestamps in UTC by default
  • Display applications need timezone context to show correct local times

The simplest solution is using Make.com's built-in date formatting function with the timezone parameter. Instead of making external API calls, you can now specify the timezone directly in the function.

For example, use 'America/Los_Angeles' for Pacific Time or 'Europe/London' for GMT. This eliminates the need for complex workarounds that were previously necessary. The function handles all the conversion logic internally.

  • Use the formatDate function with three parameters
  • Specify the output format pattern you want
  • Include the IANA timezone identifier

To format a timestamp with timezone in Make.com: 1) Use the date formatting function, 2) Include the timestamp value, 3) Add the desired format pattern (like 'MM/dd/yyyy hh:mm a'), 4) Specify the timezone parameter (like 'America/New_York').

The complete function call looks like: formatDate(webhook.timestamp; 'MM/dd/yyyy hh:mm a'; 'America/Los_Angeles'). Note that parameters are separated by semicolons, not commas, which is a common point of confusion.

  • Parameters must be separated by semicolons
  • The format pattern uses Java-style date format symbols
  • Timezone must be in IANA format (City/Region)

Common timezone format strings include: America/New_York (Eastern), America/Chicago (Central), America/Denver (Mountain), America/Los_Angeles (Pacific), Europe/London (GMT), and Asia/Tokyo (JST). These follow the IANA Time Zone Database format.

You can find the complete list in Make.com's documentation or by checking the helper tooltip when using the date function. The key is to use the region/city format rather than abbreviations like PST or EST which don't account for daylight saving time.

  • Use continent/city format (America/New_York)
  • Avoid three-letter abbreviations (PST, EST)
  • Most major cities are supported worldwide

UTC (Coordinated Universal Time) is the standard time reference used internally by Make.com, equivalent to GMT. Local time refers to the time in a specific geographic region, which may be offset from UTC by several hours and may observe daylight saving time.

The key difference is that UTC doesn't change with seasons or location, while local time does. Make.com's 2025 update makes converting between them much simpler with native timezone support in the date formatting function.

  • UTC is constant (no DST, same worldwide)
  • Local time varies by location and season
