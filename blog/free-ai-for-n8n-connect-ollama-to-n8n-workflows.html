Docker Networking Solution

Running AI models locally through Ollama solves the API cost problem, but creates a new challenge: getting n8n to communicate with the Ollama container. Most Docker installations keep containers isolated by default, preventing cross-container communication.

The breakthrough comes from creating a shared Docker network (called AppNet in this tutorial) that both Ollama and n8n join. This acts like a private phone system where containers can call each other using internal URLs instead of exposing ports to the public internet.

Key benefit: You maintain security by keeping ports blocked from external access while enabling free communication between your automation tools and AI models.

Verifying Container Connections

Before making any changes, it's crucial to verify your current Docker network setup. The tutorial demonstrates using docker network inspect appnet to check which containers are connected.

Initially, only the Ollama container appears in the network. After modifying the n8n Docker compose file and restarting the container, both services should show as connected. This verification step prevents hours of debugging later by confirming the fundamental networking works before building workflows.

Editing n8n Docker Config

The critical configuration change involves adding two network sections to n8n's Docker compose file:

  1. Networks declaration: Added under the n8n service definition to specify appnet as the network
  2. Networks configuration: Added at the file bottom to define appnet as an external network

Using the nano text editor in the Hostinger terminal makes these edits straightforward. The tutorial shows the exact placement of these additions in the YAML file to maintain proper syntax.

Pro tip: After saving changes, always verify them in Hostinger's YAML editor which provides syntax highlighting and error detection.

Testing the Connection

With networking configured, the tutorial builds a practical test workflow:

  1. Manual trigger node starts the workflow
  2. Edit fields node provides content to process (a 1600-word article)
  3. Ollama chat model node performs the summarization
  4. Basic LLM chain node formats the output

The key configuration moment comes when setting up the Ollama credential in n8n. Instead of using localhost (which would look inside the n8n container), you specify http://ollama:11434 - the internal URL for the Ollama container on your shared network.

Workflow Performance Benchmarks

Initial runs with the Llama 3.2 model took 2 minutes 40 seconds to process the test article - significantly slower than commercial APIs. However, subsequent runs completed in just 27 seconds as the model cached its initialization.

Switching to the smaller Quen 2.5 model showed better performance:

  • First run: 15 seconds
  • Subsequent runs: 12 seconds

While not as fast as paid services, this performance is more than adequate for most automation use cases while providing complete cost control.

Easy Model Management

One of Ollama's strongest features is how easily you can add and remove models:

  • Adding models: Single command like docker pull ollama/gemma:2b
  • Listing models: docker exec ollama ollama list shows installed models
  • Removing models: docker exec ollama ollama rm gemma:2b deletes unused models

New models appear instantly in n8n without container restarts. The tutorial demonstrates adding the Gemma 2B model which immediately became available for workflow testing.
