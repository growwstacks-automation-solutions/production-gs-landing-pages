Why Gmail OAuth Fails in Make.com

Every business using Make.com for email automation hits the same wall: Gmail's authentication errors that appear without clear explanation. At 2:15 in the video, you'll see the exact "Authorization Error" message that stops workflows dead.

Google intentionally makes personal accounts difficult to connect because their API terms prohibit using free Gmail for commercial automation. The system is designed to push users toward Google Workspace accounts or properly configured OAuth credentials.

Critical insight: The official Make.com documentation only covers about 40% of what you actually need. Their redirect URI examples are outdated and the API enablement sequence matters more than they indicate.

Google Cloud Console Setup

Begin at console.cloud.google.com - not the Gmail interface. Many users waste hours trying to find API settings within Gmail itself, when all configuration happens in the separate Cloud Console.

Create a new project with any name (organization doesn't matter for this purpose). The project acts as a container for your API credentials and usage tracking. At 4:30 in the video, you'll see how to navigate past Google's upsell screens to reach the actual configuration area.

OAuth Consent Screen Configuration

This is where most implementations fail. The consent screen requires specific details that Google uses to verify your app's legitimacy:

  1. Application name (use "Make.com Integration")
  2. Support email (must match your domain)
  3. Authorized domains (add make.com)
  4. Privacy policy URL (Make.com provides one)

At 7:45 in the video, notice how leaving any field blank triggers immediate authentication failures. Google's verification team will reject incomplete submissions, delaying your automation rollout by days.

Credentials Creation Process

Under "APIs & Services", select "Credentials" then "Create Credentials". Choose "OAuth client ID" and configure as:

  • Application type: Web application
  • Name: Make.com Gmail Integration
  • Restrictions: None (Make.com handles security)

Warning: The credentials creation section (9:20 in video) appears simple, but capitalization in the name field matters more than you'd expect. "Make.com" works while "make.com" fails authentication.

The Redirect URI Solution

This is the most critical and most commonly wrong step. The redirect URIs must exactly match:

  1. https://www.make.com/oauth/callback
  2. https://www.make.com/oauth/callback/

Yes, both versions - with and without the trailing slash. At 11:05 in the video, you'll see how missing just one variation causes the entire authentication to fail. These URIs tell Google where to send users after they approve access.

Required API Enablement

Two APIs must be enabled for full functionality:

  1. Gmail API (obviously)
  2. Google Drive API (less obvious but required)

Enable both through the API Library before testing your credentials. The video shows at 13:40 how skipping Drive API causes attachment-related automations to fail silently.

Testing vs Production Status

New credentials start in "Testing" status with limitations:

  • Expire after 7 days
  • Show unverified app warnings
  • Limited to 100 test users

Submit for production status immediately (takes 3-7 days). At 15:00 in the video, you'll see the verification form that requires business details. Production credentials don't expire and provide seamless authentication.

Key Takeaways

Persistent Gmail authentication errors in Make.com stem from incomplete OAuth configuration rather than platform bugs. Following Google's requirements precisely solves 95% of connection issues.
