What Is OpenClaw and How Does It Work?

OpenClaw is an open-source framework for creating AI agents that can perform tasks autonomously. Unlike standard chatbots that just respond to messages, these agents can actually take actions in the real world - browsing websites, making purchases, scheduling tasks, and more.

At its core, OpenClaw is simply a NodeJS process that combines large language models (like GPT-4) with various "tools" that give it capabilities beyond just text generation. These tools include:

Key Capabilities: Browser automation (via Playwright), file system access, cron jobs for scheduling, and native integrations with messaging platforms like WhatsApp, Slack, and email.

What makes OpenClaw different from tools like ChatGPT is its ability to maintain long-running processes and memory between interactions. While ChatGPT forgets everything when you start a new chat, OpenClaw agents can remember context through simple markdown files (user.md, memory.md, soul.md) that they can read and update.

How I Set Up My AI Assistant "Claudius"

Creating a functional AI assistant required careful consideration of security and practical implementation. Here's exactly how I set it up:

  1. Cloud Hosting: Used xie.dev to provision a virtual machine rather than running it locally
  2. WhatsApp Integration: Created a separate WhatsApp Business account with a new SIM card
  3. Financial Controls: Limited to a prepaid Revolut card with £20 balance
  4. Additional Accounts: Set up dedicated Gmail and Amazon accounts
  5. API Keys: Implemented strict billing limits on all service integrations

The WhatsApp integration was particularly important for making the assistant feel natural to use. By creating a separate business account, I could keep my personal communications separate while still accessing both from my phone.

Testing Real-World Tasks: Successes and Failures

To properly test Claudius, I gave it a series of real-world challenges - planning a trip to Scotland, making online purchases, and creating presentations. The results were mixed but revealing.

Success Story: The assistant successfully researched and purchased an outdoor swimming guide from Amazon, even canceling and reordering with faster delivery when requested.

However, other tasks revealed significant limitations. When asked to book train tickets or create Google Slides presentations, the agent would often claim success while having actually accomplished nothing. At one point it confidently stated it was working on a slideshow, only for me to discover it had merely opened Google Slides and done nothing else.

The Browser Automation Problem

The biggest technical challenge came from browser automation via Playwright. While OpenClaw can interact with websites, complex JavaScript interfaces often caused problems:

  • Date pickers and rich text editors frequently failed
  • Bot detection systems on travel sites blocked actions
  • Dynamic page elements would change before interaction completed
  • The agent would sometimes get stuck on cookie banners

Ironically, when I switched to API-based approaches (like using HackMD's REST API for presentations), the agent performed much better. This suggests that while AI agents struggle with visual interfaces, they excel at structured programmatic interactions.

Security and Privacy Concerns

During development, several security issues became apparent that any business considering AI agents should address:

Critical Risk: The permission system is entirely based on markdown files with no technical enforcement. My assistant once nearly ordered hemorrhoid cream for a friend because it couldn't distinguish appropriate requests.

Other risks included potential unauthorized access to files, unintended sharing of personal information, and the possibility of the agent scheduling malicious cron jobs. While my sandboxed setup mitigated these, they represent serious considerations for business implementations.

How Memory Works in AI Agents

OpenClaw's memory system uses simple markdown files that the agent can read and update. This provides persistent storage between sessions but requires active management:

  • user.md: Contains information about the human user
  • memory.md: Stores facts learned during interactions
  • soul.md: Defines the agent's personality and purpose

Without prompting, the agent would often forget important context. I found myself regularly instructing it to "write this in your memory file" to ensure critical information persisted. This manual memory management highlights a key limitation in current AI agent architectures.

Key Takeaways From My Experiment

After weeks of testing, several important lessons emerged about the current state of AI agents:

In summary: AI agents excel at structured tasks with clear APIs but struggle with visual interfaces. They require careful sandboxing and monitoring due to security risks, and their memory systems need active management to be effective.

Looking ahead, I believe we'll see increased focus on "bot accessibility" in web design - creating interfaces that work well for both humans and AI agents. As these tools become more common, businesses that understand their capabilities and limitations will gain a significant advantage.
