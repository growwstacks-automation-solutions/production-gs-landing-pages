Custom Project Roles Explained

n8n's role system works like building blocks. You define exactly what each team member can do with every resource type:

Step 1: Create the Role

Navigate to Settings > Roles. Click "Add Role" and name it (e.g., "Workflow Builder").

Step 2: Set Resource Permissions

For each resource (workflows, credentials, etc.), toggle these permissions:

  • View: See but not interact
  • Create: Make new items
  • Edit: Modify existing items
  • Delete: Remove items
  • Transfer: Move between projects
  • Publish: Push to production

Step 3: Assign to Users

Select a project, choose the role, and assign team members. Changes apply immediately.

Pro tip: Create role templates (like "Analyst" or "Approver") to maintain consistency across projects. At 2:15 in the video, we show how to clone roles between environments.

SSO User Provisioning

Manual user management becomes unsustainable at scale. n8n's SSO integration automatically:

  • Creates accounts on first login via Okta, Azure AD, etc.
  • Assigns roles based on group membership
  • Deactivates users when removed from your IDP

Configuration takes three steps:

  1. Enable SAML or OIDC in n8n Settings
  2. Map IDP groups to n8n roles
  3. Test and activate the connection

Enterprise benefit: SSO reduces permission errors by 92% compared to manual assignment, while cutting onboarding time from days to minutes.

6 Granular Permission Types

n8n's permission matrix covers every interaction:

View

Read-only access to workflows and resources

Create

Make new workflows or credentials

Edit

Modify existing items

Delete

Remove items (often restricted)

Transfer

Move between projects

Publish

Push changes to production

These can be mixed and matched per role. For example, you might allow:

  • Analysts: View + Create (but not Publish)
  • Approvers: View + Publish (but not Edit)
  • Admins: Full permissions

Separation of Duties Enforcement

The most powerful security feature isn't technical - it's organizational. n8n now enforces separation of duties by:

  • Requiring different users for development vs approval
  • Blocking self-review of changes
  • Maintaining clear audit trails

For example, to delete a role:

  1. First unassign all users from the role
  2. Then confirm deletion

This two-step process prevents accidental or malicious removal of critical permissions.
