Camunda AI Agent Basics

Traditional workflow automation hits a wall when processes require human-like decision making or natural language interactions. Camunda's AI agent capability bridges this gap by integrating large language models directly into BPMN workflows. This creates intelligent processes that can analyze data, make context-aware decisions, and interact with users conversationally.

The tutorial demonstrates building a "Tech Helper" agent that assists users in choosing technology devices. What makes this powerful is the agent's ability to not just fetch data (like a list of devices), but also analyze and compare them based on user follow-up questions - all within the same automated workflow.

Key advantage: Camunda AI agents combine the structure of BPMN workflows with the flexibility of LLMs, allowing you to create processes that adapt to user needs while maintaining auditability and control.

Setting Up the BPMN Diagram

Every Camunda AI agent workflow begins with a BPMN diagram that defines the process scope and structure. The tutorial starts by creating start and end events labeled "Query Asked" and "Query Answered" to establish the boundaries of the interaction.

The core of the AI agent is implemented as a subprocess configured with the "AI Agent" element type. This special subprocess serves as the container for all the agent's tools and decision logic. Importantly, the AI agent functionality requires Camunda 8.8 or higher, so version compatibility must be verified before beginning.

At 1:20 in the video, the instructor demonstrates dragging the subprocess onto the diagram and configuring it as an AI agent. This visual approach makes it easy to see how the agent fits into the overall workflow while maintaining Camunda's standard modeling paradigm.

Configuring the LLM Connection

The AI agent's intelligence comes from its connected large language model. The tutorial shows how to configure this connection in the agent's properties panel, using Amazon Bedrock as the LLM provider (though other providers are supported).

Security best practices are highlighted as the instructor demonstrates using Camunda's secrets vault to store API credentials rather than hardcoding them. The model selection (Claude 4 in this case) and system prompt configuration are also covered, with the default prompt being sufficient for this basic implementation.

Critical detail: The agent requires both a user prompt (the initial request) and a response variable to store its output. These are configured in the properties panel along with options to include assistant messages and agent context for richer interactions.

Building Agent Tools

An AI agent without tools is like a brain without hands - it can think but not act. In Camunda, tools are implemented as tasks within the AI agent subprocess. The tutorial creates a REST connector task to fetch tech device data from an external API.

Each tool requires clear documentation explaining its purpose to the agent. The video (at 4:15) shows how to write natural language descriptions that help the agent understand when and how to use each capability. The REST tool is documented as "used to get the list of all available tech stuff."

Perhaps most importantly, every tool must output its results to a variable named "tool call result" - this standardization allows the agent to properly process and utilize each tool's output in its decision-making process.

Creating Interactive User Tasks

For the agent to be useful, it needs to interact with users. The tutorial demonstrates adding a user task that displays the agent's findings and collects follow-up questions. This is where the workflow transitions from backend automation to user-facing functionality.

The user task includes a form builder with two key components: a text box to display the agent's markdown-formatted response (generated from the raw API data), and an input field for user follow-up questions. At 7:30, the video shows how these map to process variables that flow back into the agent's decision cycle.

Powerful feature: The agent can generate variables on-the-fly using the "from AI" function, allowing it to transform raw data (like JSON API responses) into user-friendly formats (like markdown tables) without custom code.

Deploying and Testing the Workflow

With the workflow designed, the tutorial moves to deployment and testing. The deployment process includes providing a sample user request ("can you show me the list of all tech stuff?") that kicks off the first test run.

The video (at 10:45) shows the workflow in action - the agent receives the request, decides to call the REST API tool, processes the results, and presents them to the user. The real test comes with the follow-up question ("can you compare the phones in more detail?"), demonstrating how the agent can handle multi-turn conversations within the same workflow instance.

This section highlights Camunda's operational visibility, showing how you can monitor the agent's decision path and tool usage through the Camunda interface.

Advanced Agent Capabilities

The tutorial concludes by showcasing the agent's ability to go beyond simple tool usage. When asked to compare devices, the agent leverages the LLM's built-in reasoning capabilities rather than making additional API calls - demonstrating how AI agents can combine pre-fetched data with generative analysis.

This hybrid approach - using tools when needed but falling back to the LLM's native capabilities - creates more efficient and natural interactions. The final output (visible at 12:30) shows a detailed comparison of smartphones that wasn't explicitly provided by the API, proving the agent's value as more than just a workflow automation tool.

Implementation tip: Well-designed AI agents in Camunda should balance tool usage with LLM capabilities, using external data when necessary but leveraging the model's reasoning power for analysis and synthesis.

Common questions about this topic

Camunda's AI agent capability allows you to integrate large language models (LLMs) directly into your BPMN workflows. This enables the creation of intelligent processes that can make decisions, analyze data, and interact with users using natural language.

The AI agent functionality was introduced in Camunda 8.8 and requires specific configuration to connect to LLM providers like Amazon Bedrock. Agents operate within defined subprocesses and can utilize various tools while maintaining context throughout workflow execution.

  • Combines BPMN structure with LLM flexibility
  • Requires Camunda 8.8 or higher
  • Supports multiple LLM providers

An AI agent in Camunda can perform several types of tasks including making REST API calls, processing and formatting data, generating natural language responses, and making decisions based on complex criteria.

The agent can also handle follow-up questions from users by maintaining context throughout the workflow execution. This makes it ideal for recommendation systems, customer support flows, and data analysis processes.

  • Data retrieval via API calls
  • Natural language generation
  • Context-aware decision making

To connect an LLM to Camunda, you need to configure the AI agent subprocess with your LLM provider details. This includes specifying the model (like Claude 4), providing API credentials through Camunda's secrets vault, and setting up the system prompt.

The tutorial demonstrates using Amazon Bedrock, but Camunda supports other providers as well through similar configuration steps. The connection is made in the properties panel of the AI agent subprocess element.

  • Configure in agent subprocess properties
  • Use secrets vault for credentials
  • Set system and user prompts

The tool call result variable is a special variable that Camunda's AI agent expects as output from every tool it uses. This standardized variable name allows the agent to properly process and assess the results of each operation.

All tools in your AI agent workflow must output their results to this variable for proper integration with the agent's decision-making process. The tutorial shows how to configure this when building both REST connectors and user tasks.

  • Standardized output variable
  • Required for all agent tools
  • Enables proper result processing

User tasks in Camunda's AI agent workflows allow the process to interact with human users. You can create forms that display information generated by the AI (like formatted markdown responses) and collect user input (like follow-up questions).

The agent processes these interactions and determines the next steps in the workflow, creating a conversational interface within your business process. The tutorial demonstrates building a form with both display elements and input fields.

  • Display AI-generated content
  • Collect user input
  • Enable conversational workflows
