Why AI Agents Fail Differently Than Traditional Software

Traditional software fails in predictable ways - a null reference, a timeout, a validation error. You can step through the code with a debugger to pinpoint exactly where things went wrong. AI agents powered by large language models fail in fundamentally different ways that break conventional debugging approaches.

As explained in the video by Arise AI's CPO, "LLM-powered agents are non-deterministic with complex emergent behaviors. You're no longer debugging code - you're debugging prompts, reasoning paths, tool integrations, and more." At 2:15 in the demo, we see a perfect example - the financial advisor agent returns "I cannot fulfill this request" without any error or exception.

60% of production AI agent failures stem from prompt and reasoning issues rather than code bugs. Without proper instrumentation, you're left guessing why simple queries fail while complex ones succeed.

The Observability Solution for AI Agents

Observability provides the instrumentation needed to understand agent behavior in production. Instead of hoping your agent works, you can know it works - and quickly diagnose it when it doesn't. The solution combines three key components:

  1. Execution tracing that records every LLM call, tool invocation, and reasoning step
  2. Evaluation frameworks to score agent outputs against business criteria
  3. Visualization tools that highlight failure patterns across many interactions

At 4:30 in the video, the Phoenix tool demonstrates this perfectly - showing side-by-side traces of successful and failed agent executions, with clear visual indicators of where the paths diverged.

Google's Agent Development Kit (ADK)

Google's open-source Agent Development Kit provides the foundation for observable agents. As Ivan explains at 3:10, "ADK was designed from the ground up with callbacks and plugins that let developers inspect all actions and tool calls." Key features include:

  • Built-in after-call handlers for every LLM invocation
  • Standardized tracing using OpenTelemetry conventions
  • Pluggable architecture for custom observability integrations

The financial advisor demo shows how ADK makes it simple to instrument agents without boilerplate code. By adopting these standards early, you ensure your agents can work with any observability backend.

Phoenix Observability Tool in Action

Phoenix provides the visualization layer that makes ADK traces actionable. As shown at 6:45 in the demo, Phoenix runs locally or in the cloud, providing:

  • Trace comparison - View successful and failed executions side-by-side
  • Evaluation dashboards - Score agent outputs across many queries
  • Error clustering - Group similar failures to identify patterns

The video's "bright red flag" visualization at 7:20 demonstrates how Phoenix surfaces issues that would take hours to discover manually. By highlighting missing tool calls and incorrect reasoning paths, it transforms debugging from guesswork to systematic improvement.

Step-by-Step Implementation

Step 1: Instrument Your Agent

Add the Phoenix tracer to your ADK-based agent. The video shows this requires just 3 lines of configuration code (at 5:15):

 from phoenix.trace import Tracer tracer = Tracer() agent = FinancialAdvisorAgent(tracer=tracer)

Step 2: Run Evaluations

Define evaluation criteria that matter for your business. The demo uses correctness checks for financial advice, but you might evaluate tone, compliance, or completeness.

Step 3: Analyze Traces

Use Phoenix's UI to identify failure patterns. At 8:10, we see how incorrect answers cluster around missing data - suggesting needed improvements to error handling.

Implementation summary: 1) Instrument with ADK and Phoenix, 2) Define business-specific evaluations, 3) Analyze traces to find and fix failure patterns. The complete code is available in Google's GitHub repository.

Online vs Offline Evaluations

At 9:30, the video explains the critical distinction between two evaluation approaches:

Online Evaluations Offline Evaluations
Monitor live production traffic Run against test datasets
Detect real-time issues Verify before deployment
Similar to APM monitoring Similar to unit testing

For reliable agents, you need both. Online evaluations catch prompt drift and model degradation, while offline evaluations ensure new versions meet quality standards before release.

Production Best Practices

Based on the video's insights and industry experience, follow these practices for production-ready agents:

  1. Sample aggressively - Evaluate at least 10% of production traffic
  2. Monitor key metrics - Success rate, latency, and cost per invocation
  3. Version everything - Prompts, models, and tools should all be versioned
  4. Automate rollbacks - When evaluations detect degradation, revert automatically

The financial advisor demo at 10:45 shows how Phoenix helps implement these practices by providing the data needed for automated quality gates.

Online evaluations monitor live production traffic to identify issues in real-time, similar to application performance monitoring. They help detect prompt drift and model degradation as they happen.

Offline evaluations are used during development like unit tests - you run test cases against known inputs to verify expected outputs before deployment. Both are essential for maintaining agent reliability over time.

  • Online: Production monitoring with real user queries
  • Offline: Pre-deployment testing with known test cases
  • Combine both for comprehensive coverage
