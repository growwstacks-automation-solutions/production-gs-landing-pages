Why Apex Over Flow for AI Agents?

Salesforce Flows provide an excellent low-code solution for many automation scenarios, but they fall short when you need granular control over AI Agent interactions. The critical limitation? Flow creates a new session with each agent call, forcing users to repeat information and breaking the natural flow of conversation.

Apex solves this by giving you complete control over session management. You can maintain the same session ID across multiple calls, preserving context and creating true conversational experiences. This is particularly valuable for complex processes like healthcare appointment scheduling (shown at 2:15 in the video) where maintaining context across interactions is essential.

Key benefit: Apex calls maintain 100% session continuity compared to Flow's stateless interactions, enabling true multi-turn conversations with your AI Agents.

Understanding the Apex Code

Salesforce provides a ready-made code snippet for calling agents from Apex (visible at 0:45 in the video). The core method is InvocationForce.callAgent(), which takes three parameters: the agent API name, your input message, and optional parameters.

The code structure follows this pattern:

  1. Create an Action.InvocableParameters object
  2. Set your input message using actionSetInvocableParameters
  3. Call the agent with InvocationForce.callAgent()
  4. Handle the response which contains both the agent's reply and session information

Pro tip: Always wrap your agent calls in try-catch blocks. AI interactions can fail due to network issues, governor limits, or invalid inputs.

Selecting Your AI Agent

Before calling an agent from Apex, you need to specify which agent to invoke (shown at 1:30 in the video). This is done using the agent's API name, which you can find in the Salesforce Agentforce interface:

  1. Navigate to Agentforce in your Salesforce setup
  2. Open your desired agent (e.g., a healthcare appointment scheduler)
  3. Copy the API name from the agent details
  4. Paste it into your Apex code where you set the agent to invoke

This API name is case-sensitive, so ensure you copy it exactly as shown. The rest of the code can remain unchanged - Salesforce handles the actual agent invocation behind the scenes.

Passing Parameters to Your Agent

The real power of Apex integration comes from dynamically passing parameters to your AI Agent (demonstrated at 1:45 in the video). Unlike Flow's static inputs, Apex lets you generate parameters programmatically based on business logic.

To pass parameters:

  1. Create an Action.InvocableParameters object
  2. Use actionSetInvocableParameters to add your parameters
  3. Structure your input as a string that your agent understands

For example, in a healthcare scenario, you might pass patient details like: "Patient: John Doe, Symptoms: Fever and cough, Preferred Date: 2026-02-15". The agent parses this input and responds appropriately.

The Session Management Challenge

The biggest hurdle when calling agents from Apex is session management (highlighted at 2:30 in the video). By default, each callAgent() invocation creates a new session, causing the agent to "forget" previous interactions.

This becomes apparent when:

  • The agent asks for information you already provided
  • Conversational context is lost between calls
  • Users have to repeat themselves at each step

In our healthcare example (3:00 in the video), the agent initially responds to symptoms but then fails to connect that to subsequent appointment details because the session changed. This creates a frustrating user experience that Apex can solve.

Maintaining Conversation Context

To maintain session continuity (shown at 3:45 in the video), you need to:

  1. Capture the session ID from the first agent response
  2. Store it in a variable or database field
  3. Pass it back with subsequent calls

There are two approaches to implementing this:

  1. Hardcoded session ID: Useful for testing (as shown at 4:00)
  2. Dynamic session management: Store and retrieve the ID from a database or cache

The healthcare example at 4:30 demonstrates successful session continuity - the agent remembers the patient's symptoms when scheduling the appointment because we maintained the same session ID across calls.
