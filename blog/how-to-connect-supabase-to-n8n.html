Why You Need Both HTTP and Postgres Nodes

Many developers assume connecting Supabase to n8n is as simple as adding a database node, only to discover their workflows fail with authentication errors. The reality is Supabase requires two distinct connection methods for complete functionality.

The HTTP node handles API authentication through Supabase's REST interface, while the Postgres node establishes direct database access for querying and modifying data. Together they provide the full range of operations needed for robust automation.

Key insight: Supabase intentionally separates API access (HTTP) from direct database operations (Postgres) for security and scalability. Your n8n workflows need both to function properly.

Step 1: HTTP Node Setup with Service Role Key

The HTTP node authenticates with Supabase using a service role key rather than the more limited anon key. This grants your workflows the necessary permissions to bypass Row Level Security when needed.

To set it up:

Step 1.1: Retrieve Your Service Role Key

Log into your Supabase dashboard and navigate to Project Settings > API. Locate the "service_role" secret key (not the public anon key) and copy it to your clipboard.

Step 1.2: Create n8n Credentials

In your n8n workspace, create a new credential of type HTTP Request. Paste your service role key into the appropriate authentication field and save.

Security note: Never expose your service role key in client-side code or public repositories. This key provides full database access and should only be used in secure server environments like n8n.

Step 2: Postgres Node Configuration

While the HTTP node handles authentication, the Postgres node manages direct database operations. This requires different connection parameters from your Supabase dashboard.

Step 2.1: Gather Connection Details

In Supabase, navigate to Project Settings > Database. You'll need:

  • Host (starts with db.)
  • Port (usually 5432)
  • Database name
  • User (typically 'postgres')

Step 2.2: Configure Postgres Node

Add a Postgres node to your n8n workflow and create new credentials. Enter all connection details exactly as provided, paying special attention to the host string format.

Step 3: SSL Security Setup

SSL encryption is non-negotiable for database connections. Supabase requires it for all hosted databases, and n8n's Postgres node must be configured accordingly.

Step 3.1: Enable SSL in Postgres Node

When setting up your Postgres credentials in n8n, ensure the SSL option is enabled. This encrypts all data transmitted between n8n and your Supabase database.

Step 3.2: Verify Connection

Test your connection with a simple SELECT query. If successful, you'll see data returned from your Supabase tables. If not, double-check all credentials and SSL settings.

Performance tip: For high-volume workflows, consider using Supabase's connection pooler settings (found in Database > Connection Pooling) to optimize performance.

Common Connection Mistakes to Avoid

Even with proper setup, several pitfalls can derail your Supabase-n8n integration. Here are the most frequent issues we see:

Using the Wrong Host String

The HTTP node uses your project URL (https://your-project.supabase.co) while the Postgres node requires the db. prefix version. Mixing these up is the most common connection failure.

SSL Configuration Errors

Forgetting to enable SSL in the Postgres node credentials will cause immediate connection failures. Always verify this setting is active.

Insufficient Permissions

Using the anon key instead of the service role key limits what your workflows can accomplish. Ensure you're using the correct authentication method for your use case.
