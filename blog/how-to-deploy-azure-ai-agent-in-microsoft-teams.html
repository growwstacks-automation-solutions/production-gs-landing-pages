Why Teams Is the Best Place for AI Agents

Most AI initiatives fail not because of technical limitations, but because employees simply don't adopt the new tools. When you require team members to log into yet another portal or learn a new interface, you're fighting against ingrained work habits. Microsoft Teams, used by over 300 million people daily, has become the digital headquarters for modern organizations.

By deploying AI agents directly in Teams, you eliminate the adoption barrier. Employees can access AI capabilities without context switching - they can ask policy questions during meetings, get IT help while collaborating on documents, or query knowledge bases without leaving their workflow. This seamless integration is what transforms AI from a novelty to a daily productivity tool.

Adoption rates increase by 3-5x when AI capabilities are embedded in existing workflows rather than offered as standalone tools. Teams integration ensures your AI investment actually gets used.

Azure AI Foundry vs. Copilot: Choosing the Right Approach

Microsoft offers two primary ways to bring AI into Teams: Copilot and custom Azure AI agents. While Copilot provides broad AI capabilities across Microsoft 365 apps, it has limitations for specialized organizational needs. At 4:20 in the video, we demonstrate how Copilot can't access custom knowledge bases or connect to internal data sources.

Azure AI Foundry gives you complete control to build agents that understand your specific business context. You can connect to internal databases, train on proprietary documents, and create workflows tailored to your operations. The trade-off is requiring more technical setup, which this guide walks you through step-by-step.

Setting Up Your Azure AI Agent

The foundation of your Teams integration is an Azure AI Foundry agent. As shown at 7:15 in the video, you'll first create a new Foundry hub and project in the Azure portal. The key components you'll configure:

Step 1: Deploy Your Foundation Model

Azure AI Foundry supports multiple LLMs including GPT models. Choose one that balances cost and performance for your use case. For our HR policy assistant, we used GPT-4 for its strong comprehension of policy documents.

Step 2: Create Your Agent

Name your agent and configure its capabilities. For document-based assistants, enable the knowledge tool and upload your files (like HR policies). The agent will use RAG (Retrieval Augmented Generation) to answer questions from these documents.

Step 3: Test in Playground

Before connecting to Teams, verify your agent works in the Azure AI Foundry playground. Ask sample questions to ensure it retrieves accurate answers from your documents. At 9:30 in the video, we demonstrate testing dress code policy queries.

Pro Tip: Start with a small set of high-quality documents. Well-structured source material dramatically improves answer accuracy.

Creating the Python Bridge Application

To connect your Azure AI agent to Teams, you need a middleware application. We built ours in Python using Flask, but you could use Node.js or other frameworks. The key requirements:

Step 1: Set Up Authentication

Modern Azure AI services use managed identity rather than API keys. At 12:45 in the video, we show how to enable system-assigned managed identity on your App Service and grant it Azure AI User permissions.

Step 2: Create the Chat Interface

Your Python app needs endpoints to: create chat threads, send messages, and receive responses from your Azure AI agent. We implemented these as simple API routes that Teams will call.

Step 3: Add Teams-Specific Endpoints

Teams requires certain endpoints like /privacy and /terms for app validation. These don't need complex functionality - just return simple pages confirming your app's compliance.

At 14:20 in the video, we walk through the complete app.py code, highlighting the key sections you'll need to adapt for your own agent.

Deploying to Azure App Service

With your Python application working locally, the next step is deployment to Azure App Service:

Step 1: Create App Service

Set up a new Linux App Service with Python 3.9+ runtime. The Basic SKU (B1) is sufficient for most conversational agents.

Step 2: Configure Environment Variables

Add your Azure AI agent ID and endpoint as application settings. This keeps sensitive configuration out of your code.

Step 3: Deploy Your Code

Use the VS Code Azure App Service extension (shown at 16:30 in the video) to deploy directly from your development environment. Verify your app works by browsing to its URL.

Critical Step: Don't forget to enable managed identity and assign permissions as shown earlier. This is the most common deployment issue.

Packaging with M365 Agent Toolkit

The Microsoft 365 Agent Toolkit CLI transforms your web application into a Teams package. At 18:50 in the video, we demonstrate the complete packaging process:

Step 1: Install the Toolkit CLI

Run npm install -g @microsoft/365-agent-toolkit to install the CLI globally. This gives you the m365-agent command.

Step 2: Prepare Manifest Files

Create manifest.json and environment.ml files defining your app's metadata, icons, and configuration. The video shows the exact structure these files require.

Step 3: Generate the Package

Run m365-agent package to create your .zip file. The CLI validates your configuration and bundles all required assets.

This package contains everything Teams needs to install your application, including the connection to your Azure-hosted Python bridge.

Deploying to Teams

The final step is submitting your package to Teams:

Step 1: Upload to Dev Portal

Go to dev.teams.microsoft.com and create a new application. Upload your .zip package and fill in any additional metadata.

Step 2: Submit for Approval

After validation, submit your app for organizational approval. As shown at 22:10 in the video, this process can take several hours.

Step 3: Admin Approval

Your Teams admin must approve the app in the Teams admin portal before it becomes available to users. We demonstrate this approval interface at 23:40.

Note: The first message to your agent may be slow as it establishes a new chat thread. Subsequent responses will be faster.

You need three main technical components to deploy an AI agent in Microsoft Teams. First is an Azure AI Foundry agent with deployed foundation models that power your AI capabilities. This handles the actual AI processing and knowledge retrieval.

Second is a middleware application (typically Python or Node.js) that bridges between Teams and your AI agent. This application handles authentication, message routing, and response formatting. Third is the Microsoft 365 Agent Toolkit CLI that packages your solution for Teams deployment.

  • Azure AI Foundry agent with deployed models
  • Middleware application (Python/Node.js)
  • Microsoft 365 Agent Toolkit CLI

The Python application connects to Azure AI Foundry using Azure's managed identity authentication system. This is more secure than traditional API keys and follows Microsoft's current security recommendations.

You need to enable system-assigned managed identity on your Azure App Service instance. Then, in the Azure portal, assign this identity the "Azure AI User" role on the resource group containing your AI Foundry resources. This grants your application the necessary permissions without exposing credentials in your code.

  • Enable system-assigned managed identity on App Service
  • Assign "Azure AI User" role to the identity
  • No API keys needed in your application code

To package a Teams application using the Microsoft 365 Agent Toolkit, you need three key files. First is a color PNG icon that will represent your application in Teams. This should be at least 192x192 pixels with a transparent background.

Second is a manifest.json file containing your application's metadata - name, description, version, and configuration details. Third is an environment.ml file that defines your deployment configuration for the Agent Toolkit CLI. These files are packaged into a zip file for submission to Teams.

  • Color icon PNG (192x192 minimum)
  • manifest.json with app metadata
  • environment.ml for toolkit configuration
