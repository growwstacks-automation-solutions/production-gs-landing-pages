Understanding HTTP 400 Errors

HTTP 400 Bad Request errors indicate that the server couldn't understand your request due to client-side issues. In Make.com workflows, these often appear when your API module configuration contains errors that prevent successful communication with the endpoint.

Unlike server errors (500-series), 400 errors mean the problem lies in how your request is structured. The error might appear immediately when testing your scenario or after several successful runs if authentication tokens expire or API requirements change.

Key insight: HTTP 400 errors are your first clue that something needs fixing in your module setup. They're actually helpful signals that prevent wasted API calls with invalid requests.

URL Troubleshooting

The first place to check when encountering HTTP 400 errors is your endpoint URL. Even small formatting issues can trigger these errors. At the 1:15 mark in our video tutorial, we demonstrate how to spot common URL problems.

Follow this checklist for URL troubleshooting:

Step 1: Verify the base URL

Check for typos in the domain name and ensure you're using HTTPS if required. Some APIs will reject HTTP connections entirely.

Step 2: Examine the path

Look for extra or missing slashes between path segments. The API documentation will show the exact expected format.

Step 3: Validate query parameters

Ensure all required parameters are included with correct names and values. URL-encode any special characters.

Quick test: Try pasting your complete URL into a browser address bar (for GET requests) or API testing tool. If it fails there, the problem is definitely in your URL structure.

Header Checks

Incorrect headers account for about 30% of HTTP 400 errors in Make.com workflows. The headers tab in your HTTP module needs careful attention to detail.

Common header-related issues include:

  • Content-Type mismatch (sending JSON as text/plain)
  • Missing required headers specified in API documentation
  • Malformed Authorization headers
  • Case sensitivity in header names

Always double-check the API documentation for exact header requirements. Some APIs are strict about header ordering or specific formatting of values like dates or tokens.

Payload Formatting

The body section of your HTTP module is where payload formatting issues often hide. These problems can be subtle but will consistently trigger 400 errors.

For JSON payloads:

  • Validate your JSON structure using a tool like JSONLint
  • Ensure all required fields are included
  • Check that data types match what the API expects
  • Remove any trailing commas

For form-data payloads:

  • Verify field names exactly match API requirements
  • Check file upload encoding if including attachments
  • Confirm Content-Type header matches payload format

Authentication Issues

Authentication problems frequently manifest as HTTP 400 errors when the authentication format is incorrect, rather than the credentials being invalid (which would typically give a 401).

Key authentication checks:

Token placement

Some APIs require tokens in headers (Authorization: Bearer), others in the body. Using the wrong location triggers 400 errors.

Token formatting

Check for required prefixes like "Bearer" or "Token". Verify no extra spaces or characters are included.

Expiration

While expired tokens usually give 401 errors, some APIs return 400 if the token format becomes invalid after expiration.

Pro tip: When debugging authentication, first test with a fresh token to eliminate expiration as a variable.

Testing Strategy

Effective troubleshooting requires a methodical approach to isolate the cause of HTTP 400 errors. Follow this testing sequence:

Step 1: Minimal viable request

Strip down your request to only essential elements - basic URL, required headers, and mandatory fields in the body.

Step 2: Verify each component

Test the URL separately, then headers, then body content. This helps identify which component contains the error.

Step 3: Gradual expansion

Once the minimal request works, gradually add back optional elements until the error reappears.

Step 4: External validation

Use tools like Postman or curl to test the request outside Make.com, eliminating platform-specific variables.

Remember: The Make.com error message often provides clues about which part of your request needs attention. Look for specific mentions of headers, body, or URL issues.

Common questions about HTTP 400 errors in Make.com

  • URL formatting issues (extra/missing slashes, wrong protocol)
  • Header problems (missing required headers, incorrect Content-Type)
  • Payload formatting errors (invalid JSON, missing required fields)

To verify your endpoint URL in Make.com, follow this checklist:

First, check for extra or missing slashes in the path. Then ensure all required query parameters are included with correct values. Confirm you're using HTTPS if required by the API, and validate the base domain is spelled correctly.

  • Test the URL directly in a browser (for GET requests) or API tool
  • Compare against the API documentation's example URLs
  • Check for URL encoding issues with special characters

HTTP 400 indicates a bad request where the server couldn't understand the request due to client-side errors. HTTP 401 means unauthorized access - the request lacks valid authentication credentials.

While both may relate to authentication, 400 suggests the authentication format is incorrect (like a malformed token), while 401 means credentials are missing or invalid. 400 errors can occur even without any authentication requirements.

  • 400: Syntax/formatting problem in the request
  • 401: Authentication failure (missing/invalid credentials)
  • 403: Forbidden (credentials valid but insufficient permissions)

To troubleshoot JSON payload issues, start by validating your JSON structure using a tool like JSONLint. Remove optional fields to isolate the problem, then ensure all required fields are included with correct data types.

Build your payload incrementally - begin with minimal required fields and gradually add complexity. Verify nested objects are properly structured and watch for trailing commas that some APIs reject.

  • Use JSON validators to check formatting
  • Compare against API documentation examples
  • Test with hardcoded values before adding dynamic data

Common header-related issues that trigger 400 errors include Content-Type mismatch (like sending JSON with text/plain header), missing required headers specified in API documentation, malformed Authorization headers, and case sensitivity problems.

Some APIs are particular about header ordering or specific formatting of values like dates or tokens. Always verify the exact header requirements in the API documentation, as small deviations can cause failures.

  • Content-Type must match payload format exactly

To test authentication tokens, first verify the token is properly placed in headers or body as required by the API. Check for expiration and refresh if needed. Test the token directly with the API using a tool like Postman to confirm it works outside Make.com.

Ensure the token has required permissions and contains no extra characters or spaces. Some APIs require tokens to be base64 encoded or prefixed with specific strings like "Bearer".

  • Test tokens in isolation before adding to workflows
  • Verify token placement (headers vs body)
  • Check for required prefixes or encoding

Using the wrong HTTP method is a frequent cause of 400 errors. Common mistakes include using GET when POST is required, sending PUT instead of PATCH, or using DELETE on endpoints that don't support it.

Some APIs strictly enforce method requirements and will reject requests that use incorrect verbs, even if the URL and payload are correct. Always check the API documentation for the exact required method for each endpoint.

  • GET for retrieving data
  • POST for creating resources
  • PUT/PATCH for updates (check which the API requires)
