Understanding the 401 Unauthorized Error

When Make.com throws an HTTP 401 error, it's telling you that the server recognized your request but refuses to authorize it. This typically happens when your authentication credentials (API keys, OAuth tokens, or basic auth) are either incorrect, expired, or insufficient for the requested operation. Unlike a 404 "Not Found" error, a 401 specifically indicates an authentication failure.

Modern APIs rotate keys regularly for security, and services frequently update their authentication requirements. What worked yesterday might stop working today if the connected service has implemented new security policies or rotated their keys. The key (pun intended) is to systematically check each component of your authentication setup.

Pro Tip: 401 errors often strike after password changes, service updates, or when API keys expire. They're the most common - and most fixable - automation interruption.

Step 1: Locate the Failing Module

The first troubleshooting step is identifying exactly which module in your Make.com scenario is failing. Log into your Make.com account and navigate to the scenario giving you trouble. Look for any modules marked with a red error indicator - these are your likely culprits.

As shown at 0:45 in the video tutorial, HTTP modules and webhooks are particularly prone to 401 errors since they rely on external authentication. Click into each failing module to view the detailed error message. Make.com will usually tell you exactly which authentication method failed, giving you a starting point for your investigation.

Step 2: Verify Your API Credentials

With the failing module identified, your next move is to verify the authentication credentials. If you're using an API key (common with services like Airtable, Notion, or Google APIs), log into the respective service and generate a fresh key. Copy it exactly - including any special characters - and paste it into Make.com.

For OAuth connections, you may need to re-authenticate. Click the "Refresh" or "Reconnect" button in Make.com to initiate a new authentication flow. Some services like Shopify require you to manually approve the connection from their admin panel after initial setup.

Remember: API keys are case-sensitive and must be copied exactly. Even an extra space character can cause a 401 error.

Step 3: Check Headers and Endpoints

Authentication isn't just about keys - it's also about how you present them. In HTTP modules, verify that your headers are correctly formatted. Common headers like "Authorization" must be spelled exactly as the API expects, with the proper prefix (usually "Bearer " for OAuth tokens).

Also confirm your API endpoint URL is correct. Some services version their APIs (like "/v2/" vs "/v3/"), and using an outdated version can trigger 401 errors. Check the API documentation for the current recommended endpoint structure.

Step 4: Test for IP Restrictions

Some enterprise APIs restrict access to specific IP addresses for security. If your credentials are correct but you're still getting 401 errors, check whether the service has IP whitelisting enabled. Make.com runs from their cloud servers, so you may need to whitelist their IP ranges.

You can find Make.com's current IP ranges in their documentation. Alternatively, consider using a service like ngrok to tunnel requests from your local IP if the API only accepts connections from known addresses.

Step 5: Validate Account Permissions

Finally, confirm that the account you're authenticating with has sufficient permissions. Many APIs implement scopes - specific permission sets that must be explicitly granted. Your Make.com connection might have been working with basic read access, but fail with a 401 when trying to perform write operations.

Log into the service directly and check your account's API permissions. Some services like GitHub and Salesforce allow you to see exactly which permissions each connected application has been granted. If necessary, reconnect with expanded permissions.

Success Metric: After updating credentials and permissions, your scenario should run without errors. If it does, congratulations! If not, there may be deeper API changes requiring investigation.

Key Takeaways

HTTP 401 errors in Make.com are frustrating but fixable interruptions to your automation workflows. By methodically checking each component of your authentication setup - from API keys to permissions to IP restrictions - you can usually restore access within minutes.

In summary: Always start by locating the failing module, then verify credentials, check headers and endpoints, test for IP restrictions, and validate account permissions. Document your API keys' expiration dates to prevent future disruptions.

A 401 error means authentication failed (invalid credentials), while 403 means the credentials are valid but lack permission to access the resource. If you get 403 after fixing 401, check your account permissions with the API provider.

401 is an authentication failure - the server doesn't know who you are. 403 is an authorization failure - the server knows who you are but won't let you perform the requested action. Both interrupt workflows but require different solutions.

  • 401: Wrong username/password, expired token, invalid API key
  • 403: Missing user role, insufficient scopes, IP restriction
  • 403 often requires admin intervention on the API provider side
