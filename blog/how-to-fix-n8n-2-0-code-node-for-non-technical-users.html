What Broke in n8n 2.0?

If you've recently upgraded to n8n 2.0 and suddenly found your code nodes failing, you've encountered one of the platform's most significant architectural changes. The update moved task runners to external mode, requiring separate Docker containers - a change that wasn't clearly communicated to non-technical users.

The documentation refers to "task brokers" and "external runners" using terminology that assumes deep technical knowledge. For business owners and operations managers who just want their automations working, this creates unnecessary frustration and downtime.

Key change: n8n 2.0 requires task runners to operate in separate Docker containers rather than running within the main n8n instance. This improves performance but breaks existing code node implementations.

The Simple Solution

After wading through GitHub issues and 20-minute YouTube tutorials that took forever to get to the point, we've distilled the fix down to three simple file updates you can copy and paste. No need to understand Docker internals or task broker architecture.

The solution involves updating your docker-compose.yml, Dockerfile.runners, and cnn-task-runners.json files with specific configurations that enable external task running. Each change has been tested to work with DigitalOcean, Linode, and other VPS providers.

Before you begin: Always take a snapshot of your instance before making configuration changes. This gives you an easy rollback point if anything goes wrong during the update process.

Step 1: Access Your Docker Instance

Rather than struggling with terminal commands, we recommend using VS Code with the Remote SSH extension for a more user-friendly editing experience. This approach works whether you're on Mac, Windows, or Linux.

How to connect:

  1. Install the Remote SSH extension in VS Code
  2. Use Command+Shift+P (Mac) or Ctrl+Shift+P (Windows) to open the command palette
  3. Search for "Connect to Host" and add your new SSH connection
  4. Enter your instance IP (found in your VPS dashboard)
  5. Authenticate with your password or SSH key

Once connected, navigate to your n8n installation directory (typically /opt/n8n/docker/) where you'll find the files needing updates.

Step 2: Update Configuration Files

Three files require specific updates to work with n8n 2.0's external task runner requirements. Each serves a distinct purpose in the new architecture:

1. docker-compose.yml

This file defines your Docker services and their relationships. The critical additions include the runner token configuration and explicit runner service definitions that enable external task processing.

2. Dockerfile.runners

Specifies the Python packages and libraries available to your code nodes. The default configuration includes numpy, pandas, and pillow for common automation tasks.

3. cnn-task-runners.json

Controls which libraries are permitted in your sandbox environment. The asterisk wildcards provide convenience but should be replaced with explicit library lists for production environments.

Pro tip: After updating these files, run docker-compose up --build to apply your changes. The process may take several minutes depending on your system resources.

Step 3: Test Your Changes

After applying the configuration updates, verify everything works using simple test nodes that check Python package availability. The video tutorial includes ready-to-use test nodes for:

  • NumPy functionality
  • pandas operations
  • pillow image processing
  • BeautifulSoup (as a negative control)

These tests confirm whether your external task runner is properly configured and which libraries are available in your sandbox environment. The BeautifulSoup test specifically verifies that unauthorized libraries remain blocked.

Security Considerations

While the default configuration gets your code nodes working quickly, it's important to understand the security implications:

The asterisk wildcards in cnn-task-runners.json allow any library to be installed and run. For business-critical automations, you should:

  • Replace wildcards with explicit library lists
  • Regularly audit installed packages
  • Consider container scanning tools for production environments

Remember: These configurations assume you're running a private n8n instance. If multiple users access your installation, implement stricter security controls.
