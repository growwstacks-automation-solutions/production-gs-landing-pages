The Two Types of Guardrail Nodes

n8n provides two distinct safety mechanisms for AI workflows, each serving different protection needs. The Check Text for Violations node acts as a content moderator, while the Sanitize Text node functions as a data scrubber.

At the 0:45 mark in the video, you can see how both nodes share the same interface but offer completely different protection layers. The violations node requires connecting an LLM to analyze message intent and context, while the sanitize node uses pattern matching to remove sensitive data without AI processing.

Key difference: Violations checks understand meaning (needs LLM), while sanitize removes patterns (no LLM needed). Use both for complete protection.

How the Violations Node Works

The Check Text for Violations node provides three critical safety checks in one package. First, it screens for NSFW content that could damage your brand. Second, it detects prompt injection attempts where users try to hijack your AI's instructions.

Most importantly, it verifies topical alignment - ensuring your agent stays on-subject. As shown at 1:20 in the tutorial, this prevents embarrassing situations where customer service bots start discussing politics or generating creative fiction instead of solving support tickets.

What the Sanitize Node Removes

The Sanitize Text node acts like a digital bouncer, removing sensitive information before it enters your workflow. At 1:50 in the video, you'll see it targeting three categories: personal data (IDs, addresses), financial information (card numbers), and technical secrets (API keys).

Unlike the violations node, sanitize uses regular expressions to find and redact patterns without understanding context. This makes it faster and more reliable for data cleaning tasks, though less capable of detecting conceptual violations like harmful intent.

Where to Place Guardrails in Your Workflow

Strategic placement turns guardrails from simple filters into a comprehensive security system. The tutorial demonstrates three critical positions at 2:30:

  1. Workflow start: Sanitize node removes sensitive data before processing
  2. Pre-agent check: Violations node blocks malicious prompts
  3. Post-agent validation: Final check ensures responses stay on-topic

This layered approach creates multiple security checkpoints without significantly impacting workflow speed. Each guardrail serves a distinct purpose in the message lifecycle.

Protecting Against Malicious Inputs

Prompt injection attacks increased 300% in , making pre-agent guardrails essential. The violations node at the workflow's entry point analyzes incoming messages for hidden commands or social engineering attempts.

As shown at 3:10, properly configured guardrails can detect and block attempts to make your agent reveal system prompts, access restricted data, or perform unauthorized actions. This protection layer is particularly crucial for customer-facing AI applications.

Validating AI Outputs

The final guardrail position validates agent responses before delivery. This catches both accidental hallucinations and deliberate circumvention attempts where users trick the AI into generating harmful content.

At 3:45, the video demonstrates how the topical alignment check maintains conversation quality. If the agent starts drifting off-subject (a common issue with complex queries), the guardrail can either correct the response or trigger a graceful failure message.

Performance Considerations

While guardrails add security, they also introduce processing overhead. The violations node typically adds 300-500ms per check due to LLM analysis, while sanitize operations complete in under 50ms.

For high-volume workflows, consider running sanitize nodes on all messages but reserving violation checks for suspicious inputs flagged by initial filters. This balanced approach maintains security without unacceptable latency.
