The Problem With Hardcoded Credentials

Every developer knows the frustration of API keys scattered across multiple workflows. You configure an HTTP request node perfectly - URL, headers, authentication - only to realize weeks later you need to make the same API call in another workflow. Without saved credentials, you're forced to:

  • Hunt through old workflows to find where you used that API
  • Copy-paste the authentication configuration manually
  • Risk security breaches from API keys stored in plain text

85% of automation builders admit to hardcoding API keys at least once, creating maintenance nightmares when credentials need to be rotated.

How Native Integrations Handle Authentication

n8n's native nodes (like Airtable or Shopify) demonstrate the ideal credential management pattern. When you create credentials for these services:

  1. They're stored securely in n8n's credential vault
  2. They become available to all nodes for that service
  3. They automatically include the correct authentication type

The HTTP Request node can work exactly the same way - you just need to configure it properly. At 1:22 in the video, you'll see how native nodes maintain credential consistency across workflows.

Setting Up Generic Credentials

The secret lies in n8n's "Generic Credential Type" option. Instead of hardcoding your API key in the Headers section:

  1. Disable the default Headers toggle
  2. Enable Authentication
  3. Select "Generic Credential Type"
  4. Choose the authentication method your API requires

Pro Tip: Always check the API documentation to confirm whether it uses Header Auth, OAuth2, Basic Auth, or another method.

Step-by-Step Credential Creation

Step 1: Access Credential Management

Click the credentials button in the HTTP Request node (or go to the main Credentials page). Select "Create New Credential" and choose "Generic Credential Type".

Step 2: Configure Authentication

For most APIs using bearer tokens:

  1. Name your credential (e.g., "Krea.ai API")
  2. Select "Header Auth" as the type
  3. Set Name to "Authorization" (capital A)
  4. Paste your API key in the Value field

Step 3: Save and Apply

After saving, your credential will appear in the dropdown for all HTTP Request nodes. At 2:15 in the video, you can see how this creates consistency across workflows.

Using Saved Credentials Across Workflows

Once saved, credentials work exactly like native integrations:

  • Available in all HTTP Request nodes
  • Automatically apply the correct authentication
  • Can be updated centrally if the API key changes

This means you only need to configure:

  1. The endpoint URL
  2. Request method (GET/POST/etc.)
  3. Body parameters (if needed)

Time Saver: This method cuts credential configuration time by 90% for subsequent API calls to the same service.

Common Authentication Types Explained

Different APIs require different authentication methods:

Header Auth

Most common for modern APIs. Places credentials in request headers (like "Authorization: Bearer API_KEY").

OAuth2

Used by services like Google and Facebook. Requires token refresh flows.

Basic Auth

Encodes username:password in base64. Common for legacy systems.

Query Auth

Appends credentials as URL parameters (?api_key=XYZ). Less secure but simple.

Security Best Practices

Proper credential storage is just the first step. For maximum security:

  1. Rotate API keys quarterly
  2. Use IP whitelisting when available
  3. Limit API key permissions to only what's needed
  4. Never share workflows with credentials included

Critical: n8n automatically excludes credentials when exporting workflows, but always verify before sharing.
