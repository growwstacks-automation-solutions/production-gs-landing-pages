Why Proper Google Credentials Matter

That frustrating "access denied" error when connecting Make.com to Google Drive isn't just bad luck - it's Google's security system working as intended. Unlike simple username/password logins, professional automation requires API credentials that specifically define what data can be accessed and how.

Google's OAuth system protects your business data by requiring explicit permissions for each service (Drive, Sheets, etc.). The credential setup process creates a secure bridge between Make.com and Google's servers, with you in complete control of what access is granted.

Good news: While the process seems technical, our 7-step method makes it simple enough for any business owner to complete in under 15 minutes. Better yet, you only need to do this once to unlock permanent automation between Make.com and all Google services.

Step 1: Create a Google Cloud Project

Every API connection needs a home - in Google's world, this is called a "project." Think of it as an organizational container for all your credentials and settings. Here's exactly how to set it up:

  1. Go to console.cloud.google.com and sign in with your business Google account
  2. If prompted, add a payment method (required but won't incur charges for basic API use)
  3. Click "Create Project" at the top of the dashboard
  4. Name it something recognizable like "Make.com Automation"
  5. Leave "No organization" selected unless you're part of a Google Workspace
  6. Click "Create" - this takes about 30 seconds in the background

Pro tip: Google gives new users $300 in free credits (valid for 3 months) when adding a payment method. These can be used for other Google Cloud services beyond just API access.

Step 2: Enable Required Google APIs

With your project created, it's time to activate the specific Google services you want Make.com to access. While you might only need Drive today, enabling related APIs now saves future headaches:

  1. In the left sidebar, click "APIs & Services" > "Library"
  2. Search for "Google Drive API" and click "Enable"
  3. Repeat for any other services you might use:
    • Google Sheets API
    • Google Docs API
    • Google Slides API
    • Gmail API
    • Gemini API (for AI features)

Enabling these APIs doesn't grant access yet - it simply tells Google which services your credentials will be allowed to connect to. The actual permissions are controlled in later steps.

Step 3: Configure OAuth Consent Screen

This step defines what users will see when authorizing your Make.com connection. Even though it's just for your own use, Google requires this configuration:

  1. Go to "APIs & Services" > "OAuth consent screen"
  2. Select "External" user type (unless you're in a Workspace)
  3. Enter an app name like "My Make.com Automation"
  4. Add your business email as the developer contact
  5. Scroll to "App domain" and add:
    • make.com
    • yourbusinessdomain.com (if you have one)
  6. Under "Developer contact information," add your email again
  7. Click "Save and Continue" through the remaining screens

Don't worry about filling every field - the minimum required information is sufficient for personal automation use.

Step 4: Create OAuth Client Credentials

Now we create the actual keys that Make.com will use to authenticate with Google. These credentials are like a digital handshake between the two services:

  1. Go to "APIs & Services" > "Credentials"
  2. Click "Create Credentials" > "OAuth client ID"
  3. Select "Web application" as the application type
  4. Name it "Make.com Connection"
  5. Under "Authorized redirect URIs," add:
    • https://www.make.com/oauth/restricted
    • https://integramat.com/ocb (if using legacy Integramat)
  6. Click "Create"

Critical: Copy your Client ID and Client Secret immediately - the secret is only shown once! Store them securely. If lost, you'll need to generate new credentials and update all existing connections.

Step 5: Set Proper API Scopes

Scopes define exactly what actions your automation can perform. Setting these correctly prevents "permission denied" errors later:

  1. In your OAuth client settings, find the "Scopes" section
  2. For Drive access, include:
    • https://www.googleapis.com/auth/drive.readonly
    • https://www.googleapis.com/auth/drive.file
  3. For full Sheets access, add:
    • https://www.googleapis.com/auth/spreadsheets
  4. For Gmail, include:
    • https://www.googleapis.com/auth/gmail.send
    • https://www.googleapis.com/auth/gmail.modify

These scopes ensure your Make.com workflows can read, create, and update files as needed without requesting unnecessary permissions.

Step 6: Add Test Users

Since your app is in testing mode (which is fine for personal automation), you must explicitly authorize which Google accounts can use the connection:

  1. Go back to "OAuth consent screen"
  2. Scroll to "Test users"
  3. Click "Add users"
  4. Enter every Google account email that will access this Make.com connection
  5. Click "Save"

Without this step, you'll get "access denied" errors even with correct credentials. Each test user will need to approve the connection the first time they use it.

Step 7: Connect to Make.com

With everything configured on Google's side, it's time to connect Make.com:

  1. In Make.com, add a Google Drive (or other service) module to your scenario
  2. Click "Add" next to the connection dropdown
  3. Select "OAuth2" as the connection type
  4. Enter your Client ID and Client Secret from Step 4
  5. Name the connection something recognizable like "My Google Drive"
  6. Click "Save" then "Continue"
  7. Authorize the connection when prompted in the Google popup

Success: Your Google services are now properly connected to Make.com! This single setup works for all Google modules - you won't need to repeat it for Sheets, Docs, or other services you enabled.

Key Takeaways

Setting up proper Google credentials unlocks the full potential of Make.com automation while keeping your business data secure. Remember these core principles:

In summary: 1) Create one Google Cloud project to contain all credentials, 2) Enable only the APIs you need, 3) Configure OAuth consent for your specific use case, 4) Generate and securely store client credentials, 5) Set precise scopes to limit access, 6) Authorize test users, and 7) Connect once to automate forever.

Frequently Asked Questions

Common questions about this topic

Google requires API credentials to securely connect third-party apps like Make.com to their services. Without proper credentials, you'll encounter authentication errors when trying to use Google Drive, Sheets, or other Google services in your Make.com workflows.

Setting up credentials ensures secure access and prevents connection issues while giving you control over exactly what data and actions are permitted. It's part of Google's security model for enterprise applications.

  • Prevents "access denied" errors in your automations
  • Follows Google's security best practices
  • Only needs to be done once for permanent access

Setting up Google API credentials is free, though Google requires a payment method on file to activate the service. You'll receive $300 in free credits valid for 3 months, which can be used for other Google Cloud services.

The credential setup itself won't incur charges unless you exceed very high usage limits (thousands of API calls per day). For typical business automation, you'll likely never reach these thresholds.

  • No cost for basic API credential setup
  • $300 free credits included for other services
  • Payment method required but not charged

You can connect multiple Google services including Drive, Sheets, Docs, Slides, Gmail, and the Gemini API. Each service requires enabling its specific API in the Google Cloud Console.

The setup process is similar for all services - you just need to enable the relevant APIs and configure the proper scopes for each one. Some services like Gmail may require additional verification if you're sending emails.

  • Drive: For file storage and management
  • Sheets/Docs/Slides: For document automation
  • Gmail: For email sending and management
  • Gemini: For AI-powered features

After creating credentials in the Google Cloud Console, you'll see your client ID and secret on the credentials page. Important: The secret is only shown once during initial setup.

Be sure to copy and store it securely immediately. If lost, you'll need to generate a new secret and update all connections using the old one. The client ID remains visible, but the secret must be regenerated if misplaced.

  • Found in Google Cloud Console > Credentials
  • Secret is displayed only once - copy immediately
  • Store securely like a password

For Make.com connections, you need to add https://www.make.com/oauth/restricted as an authorized redirect URL. If you're using other automation platforms that connect to Google services, you may need to add their specific OAuth callback URLs as well.

Each platform will provide their required redirect URL in their documentation. Never use a generic or guessed URL - incorrect redirect URIs will prevent successful authentication.

  • Make.com: https://www.make.com/oauth/restricted
  • Legacy Integramat: https://integramat.com/ocb
  • Other platforms check their documentation

Access denied errors typically occur when: 1) You haven't added your email as a test user in the OAuth consent screen configuration, 2) The required scopes aren't properly configured, or 3) The app is still in testing mode.

Verify all three areas in your Google Cloud Console settings to resolve these issues. Also ensure you're using the correct Google account when authorizing the connection in Make.com.

  • Check test users in OAuth consent screen
  • Verify all required scopes are enabled
  • Ensure you're signing in with an authorized account

Google API credentials don't expire automatically, but you may need to renew them if: 1) You change your Google account password, 2) The credentials are compromised, or 3) Google updates their security policies.

It's good practice to review your credentials annually and regenerate secrets if you suspect any security issues. The client ID typically remains the same, while secrets may need occasional rotation.

  • Credentials don't auto-expire
  • Renew if security concerns arise
  • Annual review recommended
