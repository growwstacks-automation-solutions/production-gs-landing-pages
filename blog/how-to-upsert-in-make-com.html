The Duplicate Data Problem

This fragmentation causes real business problems: marketing emails sent multiple times to the same person, sales teams working from incomplete customer profiles, and analytics that don't reflect true customer behavior. Research shows that nearly 30% of CRM data becomes inaccurate within one year due to these synchronization issues.

The cost of bad data: Companies lose an average of $15 million annually due to poor data quality according to Gartner. Upsert operations provide a systematic solution to this expensive problem.

How Upsert Works

An upsert (update + insert) operation combines two database actions into one intelligent workflow. It first searches for an existing record matching your criteria (typically email or customer ID). Based on whether a match is found, it either updates that record or creates a new one.

In Make.com, this is implemented using three key modules working together:

  1. Search Records: Looks for existing entries matching your unique identifier
  2. Router: Splits the workflow into two paths based on search results
  3. Create/Update: Either creates a new record or updates the existing one

This elegant solution ensures your database maintains one complete record per customer, regardless of how many times or through which channels their information arrives.

Step-by-Step Implementation

Let's walk through building an upsert operation in Make.com, using the example from our video tutorial where we maintain clean customer records in Airtable.

Step 1: Set Up the Search Module

Begin with an "Airtable - Search Records" module configured to search your customer table. Set the filter to match on your unique identifier field (email in our example). Limit results to 1 since we only need to know if a matching record exists.

Step 2: Add the Router

Connect a router module to handle the two possible outcomes. The first route should have a filter checking if "Total bundles equals 0" (no match found). The fallback route handles cases where a record was found.

Step 3: Create the Insert Path

On the zero-results path, add an "Airtable - Create a Record" module. Map all available fields from your incoming data to the new record.

Step 4: Create the Update Path

On the fallback path, add an "Airtable - Update a Record" module. Use the Record ID from the search results, then map updated field values from your new data.

Implementation summary: 1) Search for existing record, 2) Route based on results, 3) Create new record if none exists, 4) Update existing record if found. This four-step pattern works across most database integrations in Make.com.

Common Pitfalls

While powerful, upsert operations require careful implementation to avoid these common mistakes:

  • Non-unique search criteria: Using fields like name that aren't truly unique can cause incorrect matches
  • Missing field mappings: Forgetting to include all relevant fields in both create and update paths
  • Performance issues: Not limiting search results can slow down large databases
  • Incomplete testing: Failing to validate all possible scenarios (new, existing, partial matches)

Always test your upsert workflow with various data scenarios before deploying to production. Pay special attention to edge cases where records might partially match your search criteria.

Advanced Variations

For businesses with complex data needs, these advanced patterns can transform your upsert workflows from simple deduplication tools into sophisticated data unification systems.

The router module creates decision paths based on search results. In an upsert workflow, it checks if the search returned any records. If zero records exist (filter path), it triggers record creation. If records exist (fallback path), it triggers an update.

This logical branching is what makes upsert operations possible. The router evaluates the search results bundle count, with the filter typically set to "Total bundles equals 0" to identify new records that need to be created rather than updated.

  • Evaluates search results to choose path
  • Uses filter conditions to route scenarios
  • Fallback path handles all non-zero result cases
