What Is an HTTP Request?

Manual data transfers between business systems consume countless hours. Employees copy-paste information between apps, risking errors and wasting time that could be spent on strategic work. The HTTP request node in n8n eliminates this inefficiency by automating communication between any web services.

HTTP requests are the foundation of web communication, allowing systems to exchange data seamlessly. Every time you load a webpage, check an API, or submit a form, HTTP requests work behind the scenes. With n8n's visual interface, you can harness this power without writing code.

90% of web services expose an HTTP API, making this single n8n node capable of connecting nearly any business tool. Whether you need to sync customer data, trigger actions, or aggregate reports, HTTP requests provide the bridge between systems.

The 4 Essential HTTP Request Types

Understanding the four fundamental HTTP methods unlocks the full potential of API automation. Each serves a distinct purpose in data operations:

1. GET: Retrieve Information

The most common request type, GET fetches data from an API without modifying anything. Use it to:

  • Pull customer records from your CRM
  • Check inventory levels from an eCommerce platform
  • Retrieve weather data or exchange rates

2. POST: Create New Records

POST sends data to create new resources. Typical applications include:

  • Adding new leads to your marketing automation system
  • Creating support tickets from form submissions
  • Logging transactions in accounting software

3. PUT: Update Existing Data

PUT modifies existing records by sending complete updated information. Examples:

  • Updating customer contact details
  • Modifying product listings
  • Changing project statuses

4. DELETE: Remove Resources

DELETE removes specified data, useful for:

  • Cleaning up test records
  • Removing outdated inventory
  • Deleting duplicate contacts

Pro Tip: Most APIs follow REST conventions where GET = read, POST = create, PUT = update, and DELETE = remove. This consistency makes learning one API easier when moving to another.

3 Authentication Methods Explained

APIs protect sensitive data by requiring authentication. n8n's HTTP request node offers three approaches to meet any API's security requirements:

1. No Authentication

Some public APIs require no credentials, like weather services or public datasets. While convenient, these typically offer limited access or rate-limited data.

2. Predefined Credentials

n8n includes built-in authentication for popular services like:

  • Google Workspace (Sheets, Drive, Gmail)
  • Slack and Microsoft Teams
  • Twitter and Facebook
  • Shopify and WooCommerce

These handle OAuth flows automatically - just select the service and authorize access.

3. Generic Credentials

For custom APIs, you can manually configure:

  • API keys in headers or query parameters
  • Bearer tokens for JWT authentication
  • Basic auth with username/password
  • Digest and OAuth 1.0 authentication

Security Note: Always store sensitive credentials in n8n's credential vault rather than hardcoding them in workflows. This encrypts them and allows rotation without breaking existing automations.

When to Use Query Parameters

Query parameters customize API responses by adding filters, pagination controls, and formatting instructions directly in the URL. They appear after a question mark (?), with multiple parameters separated by ampersands (&).

Common query parameter uses:

  • Filtering: ?status=active returns only active records
  • Search: ?q=keyword finds matching content
  • Pagination: ?page=2&limit=50 gets second page of 50 items
  • Sorting: ?sort=-date orders by newest first
  • Field selection: ?fields=name,email returns only specified columns

In n8n, you can add query parameters through:

  1. The dedicated query parameters section in the HTTP request node
  2. Manually appending them to the URL
  3. Using an expression to dynamically construct the parameter string

API Design Insight: Well-designed APIs use query parameters for optional filters while keeping essential identifiers in the main URL path. This makes URLs more readable and predictable.

Headers vs Body: What Goes Where

Understanding the distinction between headers and body prevents common API integration mistakes. Think of headers as the shipping label and body as the package contents.

Headers Contain Metadata

Headers tell the API how to process the request. Common headers include:

  • Authorization: API keys or tokens
  • Content-Type: Format of body data (JSON, XML, etc.)
  • Accept: Response format preference
  • User-Agent: Identifying your application
  • Cache-Control: How to handle caching

Body Carries the Payload

The body contains the actual data being sent, typically in:

  • JSON: Most modern APIs use this lightweight format
  • XML: Common in older enterprise systems
  • Form Data: For traditional web form submissions
  • Binary: When uploading files or images

In n8n, you'll enable the "Send Headers" and "Send Body" toggles when needed. The node provides intuitive editors for both sections with autocomplete for common header fields.

Performance Tip: Minimize header size - some APIs and proxies impose limits. Only include necessary headers to avoid hitting these restrictions.

Essential Error Handling Techniques

Robust workflows anticipate and manage API errors gracefully. Implement these strategies to make your HTTP requests more reliable:

1. Status Code Monitoring

HTTP responses include numeric status codes:

  • 2xx: Success (200 OK, 201 Created)
  • 3xx: Redirection
  • 4xx: Client errors (400 Bad Request, 404 Not Found)
  • 5xx: Server errors (500 Internal Server Error)

Use n8n's IF node to check status codes and route failures appropriately.

2. Retry Mechanisms

Transient failures often succeed on retry. Configure:

  • Exponential backoff between attempts
  • Maximum retry count (typically 3-5)
  • Conditional retries only on certain errors

3. Alerting

Notify teams when critical failures occur:

  • Slack messages for API outages
  • Email alerts for authentication failures
  • Ticket creation for persistent issues

4. Data Validation

Verify API responses before processing:

  • Check for expected fields
  • Validate data formats
  • Confirm non-empty responses

Monitoring Tip: Track API response times and failure rates over time. Gradual performance degradation may indicate an approaching rate limit or system issue before complete failure occurs.

Frequently Asked Questions

Common questions about HTTP requests in n8n

The four main HTTP request types are GET (retrieve data), POST (send data), PUT (update existing data), and DELETE (remove data). GET is the most common for reading information from APIs, while POST is frequently used for submitting forms or creating records.

PUT updates existing resources, and DELETE removes them. These methods form the foundation of REST API communication. Most APIs follow this convention, making skills transferable between different services.

  • GET requests should never modify server data
  • POST often returns the created resource
  • PUT requires sending the complete updated resource

Headers are like the label on a package - they contain metadata about the request such as authentication tokens, content type, and caching instructions. The body contains the actual data being sent, similar to the contents inside a shipped package.

Headers help route and process the request, while the body carries the payload. Common headers include Authorization for API keys and Content-Type to specify JSON or XML formats. The body typically contains the business data you're sending or receiving from the API.

  • Headers are always text key-value pairs
  • Body can be text (JSON/XML) or binary data
  • Some APIs require specific headers to function properly

n8n offers three authentication options: 1) None for open APIs requiring no credentials, 2) Predefined credentials for services like Google or Slack with built-in authentication, and 3) Generic credentials for manually entering API keys, tokens, or basic auth details.

The generic option provides the most flexibility for custom APIs while predefined credentials simplify connecting to popular services. For maximum security, always store credentials in n8n's encrypted vault rather than hardcoding them in workflows.

  • Predefined credentials handle OAuth flows automatically
  • Generic supports API keys, OAuth2, and other standards
  • No auth is suitable for public data APIs

Query parameters customize API responses by adding filters, pagination controls, and formatting instructions directly in the URL. They appear after a question mark (?), with multiple parameters separated by ampersands (&).

They're ideal for search filters, pagination (?page=2), or specifying response formats. Query parameters are most commonly used with GET requests to filter or sort the returned data without changing the underlying resource.

  • Keep URLs under 2048 characters total
  • URL-encode special characters in parameters
  • Some APIs limit the number of query parameters

HTTP requests can connect to any service with a web API, which covers most modern applications. However, some services may require special protocols like WebSockets for real-time data or OAuth for authentication.

The HTTP request node in n8n handles about 90% of common API integrations. For specialized cases, n8n offers dedicated nodes for services like Slack, Google Sheets, or Shopify that handle protocol specifics automatically.

  • Check API documentation for supported protocols
  • Some legacy systems use SOAP instead of REST
  • Real-time systems may require WebSockets

POST creates new resources while PUT updates existing ones. When you POST to /users, you're adding a new user. PUTting to /users/123 updates user 123's data. POST is non-idempotent (repeating may create duplicates) while PUT is idempotent (repeating has the same effect).

Use POST when you don't know the resource ID, PUT when you're modifying a specific known resource. Some APIs use PATCH for partial updates instead of PUT's complete resource replacement.

  • POST typically returns 201 Created
  • PUT usually returns 200 OK or 204 No Content
  • Some APIs implement PUT to create if nonexistent

n8n provides several ways to manage rate limits: 1) Use the Wait node between requests to add delays, 2) Implement error handling to retry after rate limit errors, 3) Schedule workflows during off-peak hours, and 4) Use API keys with higher limits when available.

For critical workflows, consider implementing a queue system or checking the API's rate limit headers to dynamically adjust request timing. Many APIs include X-RateLimit headers showing your current usage and remaining quota.

  • 429 status code indicates rate limiting
  • Respect Retry-After header if provided
  • Consider API usage tiers for high-volume needs
