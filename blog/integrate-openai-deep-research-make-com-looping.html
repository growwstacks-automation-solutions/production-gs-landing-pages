The 300-Second Timeout Problem

Make.com's 300-second execution limit per module creates a major roadblock when working with OpenAI's Deep Research API. Unlike standard ChatGPT responses that return quickly, deep research queries often require 5-10 minutes to compile comprehensive results from web sources, documents, or vector databases.

The moment your workflow hits that 5-minute mark, Make.com terminates the execution - leaving you with partial results or complete failures. This happens because Make.com assumes any module taking longer than 300 seconds has hung or failed, regardless of whether the external API is still processing.

Key insight: OpenAI Deep Research continues processing in the background even after Make.com's timeout. The challenge isn't the API itself - it's retrieving the completed results after the initial request.

Understanding Your OpenAI Deep Research API Options

When working with OpenAI's research capabilities, you'll encounter two primary model options - each with different performance and cost characteristics. The off deep research model provides the most comprehensive results but at higher cost, while of mini offers a more budget-friendly alternative for simpler queries.

All models support three core research methods: web search (scraping and analyzing online content), remote file search (processing documents at specified URLs), and vector stores (querying pre-indexed knowledge bases). The web search method demonstrated in this workflow is particularly valuable for marketing research, competitive analysis, and content ideation.

The Polling Solution Architecture

The breakthrough comes from recognizing that OpenAI provides both an initiation endpoint and a status checking mechanism for deep research queries. By separating the request into two phases - initiation and result retrieval - we can work around Make.com's limitations.

Here's the core architecture: First, we make an API call with the background=true parameter, which tells OpenAI to begin processing but not wait for completion. This returns immediately with a request ID. Then, we implement a polling loop that checks the status every 20 seconds until the research is complete.

Why 20-second intervals? This balances quick response times with not overwhelming the API. The repeater runs up to 20 times (about 6.5 minutes total), which covers most research queries while staying under Make.com's 30-minute scenario limit.

Step 1: The Initial API Request

Begin by configuring an HTTP request module in Make.com with these critical settings. The URL should point to OpenAI's API endpoint (shown in the video at 2:15), using POST method. You'll need two headers:

  1. Authorization: Bearer YOUR_API_KEY (get this from your OpenAI account)
  2. Content-Type: application/json

For the body, use this JSON structure (replace placeholders with your actual content):

 {   "model": "off-deep-research",   "role": "user",   "content": "Your research query here",   "background": true }

The background:true parameter is crucial - it's what allows the initial request to return quickly while OpenAI processes your query in the background.

Step 2: Building the Polling Loop

After the initial request returns a request ID, we set up the polling mechanism using Make.com's repeater module. Configure it to run 20 times with a 20-second sleep between iterations - this gives us up to 6.5 minutes of polling time.

Each iteration performs an HTTP GET to OpenAI's status endpoint (different from the initial POST URL), passing the request ID. The response will include a status field that changes from "in_progress" to "completed" when ready.

Use a router after the status check to evaluate the response. If status isn't "completed", the workflow routes back through the sleep module for another iteration. This creates our effective while loop despite Make.com not having native loop constructs.

Step 3: Implementing the Break Mechanism

When the status finally shows "completed", we need to exit the loop immediately rather than continuing unnecessary iterations. This is where our creative solution comes in - we use an HTTP request to a non-existent endpoint that intentionally fails.

Configure an error handler on this request that simply continues the scenario. Because Make.com treats module failures as terminal events unless handled, this gives us a controlled way to break out of the repeater's cycle while still processing the successful results.

Pro tip: Add text parsing after retrieving completed results to extract the most valuable information. The raw API response often contains metadata you may want to filter out before using the research in your business processes.

Critical Error Handling Considerations

When working with long-running processes and custom loops, robust error handling becomes essential. The video shows three key error handling points you should implement:

  1. Initial request timeout: Set a 60-second timeout on the first HTTP request to catch connectivity issues early
  2. Status check failures: Add error handling around each status check HTTP request to catch API issues
  3. Maximum iterations reached: Include notification logic if all 20 iterations complete without getting results

These safeguards prevent silent failures and ensure you're notified if something goes wrong during the extended processing period.
