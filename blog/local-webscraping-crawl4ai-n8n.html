Why Local Webscraping Matters

Traditional web scraping solutions force you to choose between expensive cloud APIs with usage limits or complex self-hosted setups requiring constant maintenance. Both approaches expose your data to third parties and create vendor lock-in.

This local solution using Docker containers solves these problems by keeping everything on your machine. No data leaves your network, there are no API rate limits, and the entire stack can be deployed with just a few commands.

Key benefit: Process sensitive data without compliance concerns since everything stays local. The system handles 100+ page scrapes on a laptop without needing cloud credits.

Docker Setup Overview

The foundation is a docker-compose.yml file that orchestrates four key services: n8n (v2.0) for workflow automation, Postgres for data storage, Crawl4AI for scraping, and Ollama for local AI processing.

After installing Docker Desktop, you simply run docker compose up -d --build to launch all containers. The environment variables in the compose file handle all configuration, including Postgres credentials and network connections between services.

  • n8n container: Latest version with all nodes pre-installed
  • Postgres container: Pre-configured with database, user and password
  • Crawl4AI container: Local instance of the 57K-star GitHub project
  • Ollama container: Ready to download and run local AI models

Crawl4AI - The Local Scraping Powerhouse

With over 57,000 GitHub stars, Crawl4AI has become one of the most capable open-source web scrapers. Unlike simple HTTP request tools, it handles JavaScript-rendered content, follows links, and extracts structured data - all without requiring browser automation.

In our workflow, Crawl4AI processes a list of URLs (like Y Combinator and its own documentation site), returning clean markdown content ready for AI analysis. The n8n integration happens through a simple HTTP request to the local Crawl4AI container.

Pro tip: Crawl4AI can extract specific page sections (like comments or products) using CSS selectors, making it far more precise than generic scrapers.

Adding Local AI with Ollama

Ollama provides a simple way to run open-source AI models locally. After pulling a model like the 2B parameter Quen-3 via docker exec, n8n can send scraped content to it for analysis through a REST API call to the local Ollama container.

The workflow demonstrates having the AI generate a brief summary with seven key points for each scraped page. Since everything runs locally, you avoid OpenAI API costs and maintain full data privacy.

  • Model selection: Choose based on your hardware - from 2B to 70B parameter models
  • Custom prompts: Easily modify the analysis instructions in the n8n workflow
  • No internet required: Once downloaded, models work completely offline

Structured Data Storage in Postgres

The workflow automatically creates a crawl_summary table in Postgres if it doesn't exist, with columns for:

  • Auto-incrementing ID
  • Brief summary text
  • Key points (stored as JSON)
  • Source URL
  • Creation and update timestamps

Connecting n8n to Postgres requires just the hostname (postgres), port (5432), and credentials from the docker-compose environment variables. The workflow shows how to both insert new records and query existing ones.

Building the n8n Automation Workflow

The complete workflow has six key steps that execute sequentially:

Step 1: Table Creation

Ensures the Postgres table exists with proper schema before inserting data.

Step 2: URL List Input

Accepts an array of URLs to scrape (like Y Combinator and Crawl4AI docs).

Step 3: Crawl4AI Processing

Sends each URL to the local Crawl4AI container and receives cleaned markdown.

Step 4: AI Analysis

Sends markdown to Ollama to generate summaries and key points.

Step 5: Postgres Insert

Stores all results in the database with source tracking.

Step 6: Verification Query

Optional step to confirm data was stored correctly.

In summary: The workflow automatically scrapes sites, analyzes content with AI, and stores structured results - all through local containers communicating with each other.
