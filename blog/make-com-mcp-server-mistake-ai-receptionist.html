The Problem: My AI Receptionist Stopped Working

Everything was running smoothly until my AI receptionist suddenly couldn't schedule appointments anymore. The 11Labs-powered assistant could still answer calls and chat with customers, but when it tried to check calendar availability or book appointments through my Make.com automations - complete silence.

At first, I assumed it was an API outage or authentication issue. But after checking logs and retesting connections, I discovered the problem was much simpler - and entirely my fault. The MCP server URL in my 11Labs configuration was formatted incorrectly, breaking the bridge between my AI agent and Make.com automations.

The symptom: My AI receptionist acted like it couldn't "see" any of my Make.com automations, even though they were properly configured and worked when triggered manually.

MCP Server Basics for Make.com

Make.com's MCP (Make Control Protocol) server acts as a gateway that allows external applications to discover and execute your automations. When configured properly, it turns your Make.com scenarios into API endpoints that tools like 11Labs can call.

The MCP server requires three key pieces of information: your Make.com zone (the regional server where your organization lives), your organization ID, and an authentication token. These combine to form a URL that external tools use to connect to your automations.

Key components: Zone URL (e.g., us1.make.com) + Organization ID (numeric) + MCP Token (authentication key) = Functional MCP server endpoint.

The Critical URL Formatting Mistake

Here's where I went wrong: When copying my Make.com zone for the MCP server URL, I only included "US1" - the zone prefix - rather than the full "us1.make.com" domain. This small oversight completely broke the connection.

The correct URL format should be: https://[zone].make.com/mcp/v2/[organization-id]. My mistake was using just the zone prefix where [zone] should be, rather than the complete zone URL. While the difference seems minor, it's enough to prevent external tools from properly connecting to your Make.com automations.

Visual example: Wrong: https://US1/mcp/v2/12345 vs. Right: https://us1.make.com/mcp/v2/12345

Correct MCP Server Configuration

After identifying the URL formatting issue, fixing it took less than a minute. Here's the step-by-step process to properly configure your MCP server connection:

Step 1: Identify Your Make.com Zone

Look at your Make.com dashboard URL after logging in. The zone is the subdomain before .make.com (e.g., us1.make.com). This is often visible in your browser's address bar.

Step 2: Find Your Organization ID

In the same dashboard URL, locate the numeric organization ID that appears after /organization/ in the path. For example: .../organization/12345/dashboard.

Step 3: Generate an MCP Token

In Make.com, navigate to Settings → API Tokens and generate a new MCP token. Copy this securely as it won't be shown again.

Step 4: Build the Complete URL

Combine these elements into the format: https://[zone].make.com/mcp/v2/[organization-id]. For example: https://us1.make.com/mcp/v2/12345.

In summary: Use full zone URL (us1.make.com) + Organization ID + MCP Token = Working MCP server connection that lets external tools like 11Labs interact with your Make.com automations.

Security Considerations

While MCP server connections are powerful, they should be configured carefully. The MCP token functions like an API key - anyone with it could potentially trigger your automations. Here are key security practices:

  • Store MCP tokens securely and rotate them periodically
  • Only expose necessary automations to external tools
  • Implement error handling in your scenarios
  • Monitor execution logs for unusual activity

In 11Labs and similar tools, you can granularly control which automations are accessible. I keep sensitive automations disabled for external access while enabling only the specific functions my AI receptionist needs.
