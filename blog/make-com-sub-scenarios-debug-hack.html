The Sub-Scenario Debugging Problem

If you've worked with Make.com sub-scenarios (and you should be - they're incredibly powerful for organizing complex workflows), you've likely hit this frustrating wall: when debugging, you can't see what input parameters were actually passed to the sub-scenario. The execution history only shows the results of operations, leaving you blind to the critical data that started the process.

This becomes particularly painful when troubleshooting complex workflows where data might be getting transformed incorrectly between steps. Without seeing the inputs, you're left guessing whether the problem lies in the parent scenario's data preparation or the sub-scenario's processing logic.

80% of sub-scenario debugging time is wasted trying to determine whether input data arrived correctly before you can even begin analyzing the actual workflow logic.

The Zero-Cost Solution

The solution is surprisingly simple yet often overlooked: add a variable module at the very start of your sub-scenario to capture and display all input parameters. This module doesn't process or transform the data - it simply makes the inputs visible in the execution history.

What makes this approach particularly valuable is that it consumes zero Make.com credits. The variable module operates without any computational cost, making it free to implement across all your workflows. At 0:45 in the video tutorial, you can see exactly how this module is configured to capture the input parameters without affecting the scenario's operation.

How to Implement the Debug Module

Implementing this debugging technique takes just three simple steps that any Make.com user can follow:

Step 1: Add a Variable Module

Place a variable module as the very first operation in your sub-scenario, before any other processing occurs. This ensures you capture the inputs exactly as they were received.

Step 2: Configure the Variable

Name the variable something clear like "Input Parameters Debug" and map all the input parameters you want to track. You can capture individual fields or entire data structures.

Step 3: Run and Review

Execute your scenario normally. Now when you check the execution history, you'll see the input parameters displayed clearly in the variable module's output.

In summary: Add variable module → Configure to capture inputs → View complete parameter history. This simple addition transforms your debugging capability.

Advanced Troubleshooting Tips

Once you've implemented the basic debug module, consider these pro tips to enhance your troubleshooting capabilities:

  • Add timestamps - Include a timestamp variable to track exactly when inputs were received
  • Capture metadata - Record which parent scenario triggered the execution
  • Use in testing - Create test scenarios that verify input structures before live deployment
  • Document expected formats - Add notes in the variable description about required input formats

These small additions can help you catch issues before they cause workflow failures, saving hours of debugging time down the line.
