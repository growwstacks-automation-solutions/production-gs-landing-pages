Step 1: Create Your API User

Your Autotask API user is the foundation of the integration. In Autotask, navigate to Admin > API Users and create a new credential set. Three critical pieces of information you'll need:

  1. Username (must be exact)
  2. Password (copy precisely)
  3. Tracking identifier (critical for authentication)

Store these credentials securely - we recommend a password manager rather than OneNote. The tracking identifier is especially important as it links your n8n instance to your specific Autotask database.

Step 2: Host Your n8n Instance

Unlike many integrations, n8n requires self-hosting to connect with Autotask. The cloud version won't work due to Autotask's network requirements. We recommend Hostinger for MSPs - it's affordable ($4.99/month) and includes one-click n8n installation.

When setting up your Hostinger account, decline all upsells and focus on the basic hosting plan with n8n. The platform will guide you through n8n installation - just follow the prompts until you reach your n8n dashboard.

Pro tip: Choose a server location closest to your Autotask database zone (WW2 for pre-release, etc.) to minimize latency.

Step 3: Install the Autotask Node

In your n8n dashboard, navigate to Settings > Community Nodes. Search for "n8n-nodes-autotask" and install it. You'll need to check the "I understand the risks" box - while the risks are minimal, this is required for community node installation.

Once installed, you'll have access to all 710 Autotask API endpoints through n8n's visual interface. The node handles authentication behind the scenes, so you can focus on building workflows rather than API calls.

Step 4: Configure the Connection

Create a new workflow in n8n and add an Autotask node. In the credentials section:

  1. Paste your tracking identifier (from Step 1)
  2. Enter your API username
  3. Input your password (n8n will encrypt it)
  4. Select your Autotask zone (critical - WW2 for pre-release)

Test the connection immediately. If it fails, double-check your zone selection - this is the most common point of failure. The connection test should return "successful" before proceeding.

Step 5: Test API Calls

Start with a simple API call like "Get API Usage Threshold" to verify everything works. This safe endpoint shows your current API usage without modifying any Autotask data.

Successful execution will return your usage stats - typically showing you're using less than 1% of your 10,000/hour limit. This confirms your n8n instance can both read from and write to Autotask as needed.

Note: The first execution creates a manual trigger node in your workflow canvas. This is normal - you'll replace it with automated triggers as you build out your workflows.

Common Pitfalls to Avoid

After implementing this integration with multiple MSPs, we've identified three frequent mistakes:

  1. Wrong time zone: Using your local zone instead of Autotask's zone identifier (like WW2)
  2. Typos in tracking ID: The 32-character identifier must be copied exactly
  3. Cloud hosting: Attempting to use n8n.cloud instead of self-hosting

All these issues manifest as authentication failures. If your connection test fails, systematically check each element before contacting support.
