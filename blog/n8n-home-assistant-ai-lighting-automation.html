Why Automate Lighting with AI?

Manual light adjustment is one of those small daily frustrations that add up over time. You walk into a room at night and the lights blast at full brightness, or try to read in the afternoon with insufficient lighting. Traditional automations based on simple time schedules can't account for seasonal changes or personal preferences.

By combining n8n's workflow automation with Home Assistant's device control and AI decision-making, we create a system that understands context. The AI considers not just the time of day, but can eventually incorporate factors like weather conditions, your calendar schedule, and even personal preferences learned over time.

Studies show that properly tuned lighting can improve productivity by up to 20% during daytime hours while promoting better sleep when dimmed appropriately at night. This automation brings those benefits automatically.

Setting Up n8n in Home Assistant

While Home Assistant excels at device control, n8n brings powerful workflow automation capabilities. The first step is installing n8n as a Home Assistant add-on, which requires adding the n8n repository since it's not included in the default store.

After adding the repository URL (available in the video description), you can install n8n like any other add-on. Before starting it, be sure to set your timezone in the configuration tab - this ensures time-based automations work correctly. The installation pulls down a Docker image and sets up all required components automatically.

Pro Tip: Allocate sufficient memory to your Home Assistant instance before installing n8n. The combined system works best with at least 4GB RAM available.

Configuring the Webhook Trigger

The most challenging part of this integration is establishing communication between Home Assistant and n8n. Since n8n can't directly subscribe to Home Assistant sensor changes, we use webhooks as the bridge between the systems.

In n8n, create a new webhook node and configure it to listen for POST requests. You'll need a long-lived access token from Home Assistant (created in your user profile's security section) to authenticate the connection. The webhook URL generated by n8n will be used in the Home Assistant automation later.

At 3:45 in the video, you'll see the critical troubleshooting step - replacing the default hostname with your Home Assistant's actual IP address to ensure reliable connectivity between the systems on your local network.

Adding AI Decision Making

The magic happens in the AI agent node where we define rules for lighting based on time of day. The prompt instructs the AI to act as a lighting designer, analyzing the current time and determining appropriate brightness levels.

For this implementation, we're using Deepseek's API (which recently reduced prices by 50%), but you could substitute OpenAI, Anthropic, or other providers. The AI node outputs a brightness percentage that flows into the next step of the workflow.

Example AI Prompt: "Act as a professional lighting designer. Given the current time is {time}, recommend an appropriate brightness percentage (0-100) for ambient room lighting. Consider human circadian rhythms and typical daily activities at this hour."

Controlling the Lights

The final node in the n8n workflow uses Home Assistant's service call capability to actually adjust the lights. You'll need your light's entity ID from Home Assistant and will configure the service to "turn_on" with the brightness attribute set to the value determined by the AI.

This separation of decision-making (in n8n) from device control (in Home Assistant) is powerful. It means you can change the AI logic or add new decision factors without modifying how the lights are controlled. The systems work together while maintaining their separate strengths.

Creating the Home Assistant Automation

With the n8n workflow ready, we need to create the Home Assistant automation that triggers it. This involves two parts: a REST command that knows how to call our n8n webhook, and an automation that fires when the presence sensor detects someone.

The automation watches for your presence sensor to change from "Clear" to "Detected" and then executes the REST command. This separation means you can easily add additional triggers later (like motion sensors in other rooms) that all use the same smart lighting logic in n8n.

Troubleshooting Tips

If your automation isn't working, check these common issues: First, verify the webhook URL in Home Assistant matches exactly what n8n shows (including using the IP address rather than hostname). Second, ensure your long-lived access token hasn't expired or been revoked.

Third, check that the presence sensor is properly reporting state changes in Home Assistant's developer tools. Finally, examine n8n's execution history to see if the workflow is being triggered and where any failures might occur in the process.

Going Beyond Basic Brightness

This implementation focuses on brightness control, but the same pattern can extend to other lighting attributes. You could have the AI recommend color temperature adjustments throughout the day (cooler in morning, warmer at night) or even dynamic color schemes based on weather or your activities.

The workflow could also incorporate multiple lights in a room, exterior lighting conditions, or even inputs from other smart devices. The n8n and Home Assistant combination provides a flexible foundation you can build upon as your smart home evolves.

Advanced Idea: Add a "preferred lighting profile" selection in Home Assistant that gets passed to n8n, allowing different household members to have personalized lighting preferences automatically applied.
