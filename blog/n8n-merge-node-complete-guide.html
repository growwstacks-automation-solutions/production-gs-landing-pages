Append vs Combine: When to Use Each

Data integration challenges often stem from not understanding the fundamental difference between appending and combining data. Append operations stack datasets vertically (like adding rows to a spreadsheet), while combine operations join data horizontally (like SQL joins).

The choice depends on your data structure needs: append when you want to aggregate similar records from multiple sources, combine when you need to enrich records with related data from another source.

Key insight: Append creates longer datasets, combine creates wider datasets. Use append for union operations, combine for join operations.

Append Examples: Stacking Data Vertically

Appending is the simplest merge operation - it takes multiple inputs and stacks them sequentially. At 2:15 in the video, we see a basic example combining two music datasets with identical structures.

However, append operations can handle mismatched structures too. When combining a customer list with an orders table (different columns), undefined values appear for non-matching fields. This flexibility comes with responsibility - you'll need subsequent nodes to handle these undefined values.

Watch for: Duplicate records are common with append operations. Use a Remove Duplicates node or custom JavaScript function afterward to clean your data.

Combine Joins: SQL-Style Data Merging

Combine operations bring SQL-like joining power to visual workflows. The inner join (keep matches) at 8:30 demonstrates joining language data where only matching records appear in the output.

When field names differ between sources (like "language" vs "language_spoken"), n8n lets you map equivalent fields. This mirrors SQL's ON clause while maintaining the visual workflow advantage. The output merges fields from both sources into unified records.

Left vs Right Joins: Priority Matters

Left joins (enrich input 1) preserve all records from your primary dataset while adding matching data from secondary sources. At 12:45, we see customer records remain complete even when order data is missing.

Right joins (enrich input 2) are less common but useful when your secondary data takes priority. The video shows how switching join direction changes which records survive when matches don't exist. In practice, 80% of workflows use left joins for their predictable behavior.

Cross Joins: Creating All Possible Combinations

Cross joins (combine all possible outcomes) generate every record combination between sources - powerful but dangerous. The 18:20 demo shows how 3 names × 2 languages creates 6 output records.

While useful for generating test cases or product variants, cross joins can explode your data volume. A 100-record × 100-record join creates 10,000 outputs! Always add limits or filters when using this operation.

Branch Selection: Conditional Data Routing

Sometimes you don't want to merge data - you want to choose between sources. The choose branch option at 22:10 lets you select one input stream while ignoring others.

This works like a switch statement for your workflow branches. Common uses include A/B testing different data sources, implementing failover logic, or toggling between live and test data without modifying your entire workflow.

Position Merging: Index-Based Matching

Combine by position matches records based on their order rather than field values. At 25:40, we see names paired with cities based solely on their position in each dataset.

This approach works well for pre-sorted data but fails silently when list lengths differ - the output uses only the shortest list's length. Always validate your data ordering and length before using position-based merging.

Clash Handling: Resolving Field Conflicts

When identically named fields collide during merging, n8n offers several resolution strategies. The 28:15 demo shows options from appending input numbers to field names to deep/shallow merging nested objects.

For clean results, consider renaming fields before merging or using an Edit Fields node afterward. The best approach depends on whether you need to preserve all data or prioritize one source's version.

SQL Queries: Advanced Merging Logic

For complex joins beyond the visual interface, n8n's merge node supports direct SQL queries. At 30:20, we see an INNER JOIN implemented with SQL syntax between labeled inputs.

While powerful, SQL in n8n works best for developers with existing SQL knowledge. For most workflows, the visual join options provide sufficient power without the maintenance overhead of raw SQL queries.

Frequently Asked Questions

Common questions about n8n merge operations

Append stacks data vertically (like adding rows to a spreadsheet) while combine joins data horizontally (like SQL joins). Append is best for combining similar datasets, while combine is used when you need to match records based on common fields or positions.

Append creates longer datasets by adding records end-to-end. Combine creates wider datasets by merging fields from related records. Choose append for simple aggregation, combine for relational data integration.

  • Append: Input 1 + Input 2 = More rows
  • Combine: Input 1 + Input 2 = More columns
  • Use append for data union operations

After using append, add a Function node with JavaScript to filter duplicates or use the Remove Duplicates node. The merge node itself doesn't have built-in deduplication, so you'll need to handle this in a subsequent step of your workflow.

For robust deduplication, create a unique key from multiple fields (like email + timestamp) rather than relying on single fields. This prevents false matches while catching true duplicates.

  • Use JavaScript: const unique = [...new Map(items.map(item => [item.id, item])).values()]
  • Remove Duplicates node works for simple cases
  • Consider fuzzy matching for near-duplicates

Use an inner join (keep matches) when you only want records that exist in both datasets. Use a left join (enrich input 1) when you want all records from your primary dataset (input 1) plus any matching records from the secondary dataset (input 2). Left joins are more common in business workflows.

Left joins preserve your "base" data while optionally adding related information. Inner joins act as filters, removing any records without matches in both sources. Choose based on whether missing matches should discard records (inner) or leave them incomplete (left).

  • Left join: Keep all customers + their orders if exist
  • Inner join: Only customers with orders
  • Right joins are rare - redesign as left joins when possible

Cross joins (combine all possible outcomes) create every possible combination between two datasets. Practical uses include generating all possible product variants, creating schedule combinations for team members, or building test case matrices for QA workflows.

Because cross joins multiply record counts (10 × 10 = 100), always add limits or filters. They're powerful for combinatorial scenarios but dangerous with large datasets. Use case: "Show me every possible pairing of products and target demographics."

  • Product variants (colors × sizes)
  • Team scheduling (people × time slots)
  • Testing matrices (inputs × environments)

Use the clash handling options in the merge node. You can append input numbers to field names, prefer one input's version, or perform deep/shallow merges. For clean results, consider renaming fields before merging or using an Edit Fields node afterward.

The best approach depends on your data structure needs. Appending input numbers (field_1, field_2) preserves all data. Preferring one input simplifies output but loses alternative values. Deep merging combines nested objects intelligently.

  • Add input suffixes: price_1, price_2
  • Prefer input 1: Keep only the first version
  • Deep merge: Combine nested object structures

Choose branch is useful when you need to conditionally select one input stream while ignoring others. Common scenarios include testing workflows (switching between live and test data), handling fallback data sources, or implementing simple routing logic in your automation.

Think of choose branch as a data switch statement. It's simpler than building entire conditional workflow branches when you only need to toggle between data sources. Example: "Use mock API responses during development, real data in production."

  • A/B test data sources
  • Failover to backup data
  • Toggle between dev/prod configurations

For most joins and basic operations, use the built-in merge operations - they're more visual and easier to maintain. Reserve SQL queries for complex joins, aggregations, or when you need to leverage existing SQL knowledge. The merge node's visual interface handles 90% of common data combination needs.

SQL shines when you need window functions, complex subqueries, or advanced filtering during the join. But for standard left/inner joins, the visual interface provides better visibility into your data transformation logic without SQL's learning curve.

  • Visual joins: Better for most workflows
  • SQL: Needed for advanced analytics
  • Hybrid approach: Visual joins + SQL functions
