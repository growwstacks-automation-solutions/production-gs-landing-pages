The Repetitive Prompt Problem

Every AI developer knows the frustration: you've perfected a complex workflow - maybe API documentation generation or database migration - only to find yourself rewriting nearly identical instructions for the next project. This redundancy wastes time and introduces inconsistencies across your implementations.

OpenCode's Skills feature addresses this exact pain point by letting you create modular, reusable components that your AI agents can discover and use on demand. Instead of bloating your initial prompt with every possible instruction, Skills sit ready to be pulled in when relevant - like a well-organized toolbox your agent knows how to use.

67% of AI developers report spending significant time rewriting similar prompts across projects, according to internal OpenCode surveys. Skills can eliminate this redundancy while ensuring consistent implementation of best practices.

What Are OpenCode Skills?

Think of Skills as reusable instruction sets that encapsulate specific capabilities or knowledge domains. Each Skill is a self-contained package containing:

  • A descriptive name and purpose
  • The core instructions or workflow
  • Optional supporting files (scripts, templates, etc.)
  • Permission settings controlling access

Unlike traditional monolithic prompts, Skills are loaded dynamically when needed. This keeps your agent's initial context window clean while providing access to specialized knowledge on demand. The latest OpenCode version supports Skills natively - no plugins or third-party tools required.

Skill Storage Locations

OpenCode provides three logical locations for storing Skills, each serving different use cases:

1. Project-specific Skills

Located in a .opencode/skills folder within your project directory, these Skills are tied to specific workflows. Perfect for React component patterns or project-specific documentation standards.

2. Global Skills

Stored in a central location (typically ~/.opencode/skills), these Skills are available across all projects. Ideal for company-wide standards like code review checklists or branding guidelines.

3. Cloud Skills

Hosted in compatible cloud storage, these Skills can be shared between OpenCode and other tools, making them perfect for cross-platform workflows or team collaboration.

Creating Your First Skill

Let's walk through creating a practical Skill for API endpoint documentation. At 3:22 in the video, we demonstrate this exact process:

Step 1: Set Up the Skill Folder

Create a folder named api-docs (following the lowercase alphanumeric-hyphen naming convention) in your chosen Skills location.

Step 2: Add the skill.md File

Create skill.md (must be lowercase) with required YAML front matter:

 --- name: api-docs description: Generates comprehensive API endpoint documentation ---

Step 3: Add Your Instructions

After the front matter, include the actual workflow instructions telling the agent how to generate documentation including paths, HTTP methods, request/response schemas, and authentication requirements.

Pro Tip: Keep Skills focused on single responsibilities. Instead of one giant "document everything" Skill, create smaller Skills for different documentation aspects (endpoints, errors, examples) that can be combined as needed.

Skill Discovery Mechanism

OpenCode agents automatically discover available Skills through an intelligent directory walk:

  1. When an agent starts, it begins searching from its current directory
  2. It walks upward through the directory structure until reaching a Git root
  3. Along the way, it collects all valid Skills found in .opencode/skills folders
  4. Discovered Skills become available to the agent by default

This discovery process means you don't need to manually configure each agent with every possible Skill. The agent naturally finds relevant Skills based on its working context while respecting permission controls.

Referencing External Files

Skills can reference additional files within their folder, enabling complex workflows without cluttering the main instruction set. For example:

  • A database migration Skill might reference a bash script for validation
  • An API testing Skill could use a Python script to generate test cases
  • A documentation Skill might pull from template files for consistent formatting

This modular approach keeps your Skills focused while allowing sophisticated functionality. The agent loads these supporting files only when the Skill is invoked, maintaining efficient memory usage.

Permission Management

OpenCode provides granular control over Skill access through a flexible permission system:

Global Permissions

Configured in code.json, these set default access levels:

 {   "permissions": {     "internal-*": "deny",     "experimental-*": "ask",     "docs-*": "allow"   } }

Agent-specific Overrides

Individual agents can override global settings. For example, you might allow experimental Skills for your testing agent while restricting them in production.

Complete Disabling

Skills can be completely hidden from specific agents by setting disabled: true in their configuration - they won't even appear as options.

Common Pitfalls to Avoid

When creating Skills, watch for these frequent issues:

1. Naming Problems

Skill folders must match the YAML name field exactly (case-sensitive). Names must follow the lowercase alphanumeric-hyphen pattern (e.g., api-docs not API_Docs).

2. File Location Errors

The main instruction file must be named skill.md (all lowercase) and placed directly in the Skill folder - subfolders won't work.

3. YAML Formatting Issues

Improper indentation or invalid YAML will cause the Skill to be silently ignored. Always validate your front matter with a YAML linter.

4. Overly Complex Skills

Skills should be focused and modular. If a Skill grows beyond 1,024 characters in its description, consider splitting it into smaller components.

Team Collaboration Tip: Maintain a shared Skill repository on GitHub where team members can contribute approved Skills. This ensures consistency while allowing controlled evolution of your AI workflows.
