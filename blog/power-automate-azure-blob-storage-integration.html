The Dataverse Storage Cost Problem

Many Power Platform developers default to storing files in Dataverse because it's readily available, but this convenience comes at a steep price. Dataverse database storage costs approximately $40 per GB per month, while Azure Blob Storage starts at just $0.018 per GB for hot storage - a 99.5% cost difference.

The financial impact becomes dramatic when storing hundreds of files. A typical business storing 50GB of documents would pay $2,000/month in Dataverse versus just $0.90 in Azure Blob Storage. This tutorial solves that cost problem while maintaining seamless integration with Power Apps.

Cost comparison: Storing 1TB of files would cost $40,000/month in Dataverse versus $18/month in Azure Blob Storage hot tier - enabling the same functionality at 0.045% of the price.

Why Azure Blob Storage Solves This

Azure Blob Storage provides object storage specifically designed for unstructured data like documents, images, and media files. Unlike Dataverse which stores files in a relational database, Blob Storage uses a flat namespace optimized for file storage at massive scale.

The integration works because Power Automate's Azure Blob Storage connector provides direct access to blob operations. You get all the benefits of cheap, scalable storage while maintaining the ability to trigger flows from Power Apps when users upload files. The files remain accessible through direct URLs or via Power Automate flows that retrieve them.

Prerequisites for the Integration

Before creating the flow, you'll need three key components configured in your Azure environment:

  1. Azure Storage Account: Create this in the Azure portal with performance set to "Standard" and account kind as "StorageV2"
  2. Blob Container: Created within your storage account with the appropriate access level (public or private)
  3. Power Automate Premium License: Required to use the Azure Blob Storage connector

The tutorial demonstrates using anonymous access for simplicity, but production environments should use Microsoft Entra ID authentication. At 4:32 in the video, you can see the exact storage account configuration used in the demo.

Configuring Your Blob Container

Container configuration determines who can access your files. In the Azure portal:

  1. Navigate to your storage account > Blob service > Containers
  2. Create a new container or select an existing one
  3. Set the access level to:
    • Private: Requires authentication for all access
    • Blob: Allows anonymous read access to blobs only
    • Container: Allows anonymous listing and read access

Best practice: Use private containers with Entra ID authentication for sensitive data, and blob-level public access only for intentionally public files like marketing materials.

Creating the Power Automate Flow

Follow these steps to build the file upload flow:

  1. In your Power Platform solution, create a new cloud flow with a manual trigger
  2. Add a "File" input parameter to accept the uploaded file
  3. Add the "Azure Blob Storage" connector and select "Create blob V2" action
  4. Configure the action with:
    • Storage account name
    • Container name
    • Blob name (use the dynamic content from the file input)
    • Blob content (use the file content bytes from the input)

At 7:15 in the video, you can see the exact flow configuration with the dynamic content selections that make this work automatically for any uploaded file.

Setting Up Authentication

The tutorial shows anonymous access for simplicity, but production environments should use one of these secure authentication methods:

  1. Microsoft Entra ID: Register an app in Entra ID and assign the Storage Blob Data Contributor role
  2. Shared Key: Use the storage account access keys (rotate regularly)
  3. SAS Tokens: Generate time-limited tokens with specific permissions

For Entra ID authentication (recommended), you'll need to:

  1. Register an application in Entra ID
  2. Grant API permissions for Azure Storage
  3. Assign RBAC roles in the storage account
  4. Configure the Power Automate connector to use Entra ID

Testing and Verifying the Flow

After creating your flow:

  1. Click "Test" and select the manual trigger option
  2. Upload a test file through the flow interface
  3. Check the run history to verify success
  4. Navigate to your Azure Storage container to confirm the file appears

At 11:40 in the video, you can see the complete testing process including how to verify the file appears in Azure Storage Explorer with the correct metadata and properties.

Troubleshooting tip: If files don't appear, check the container access level and ensure your authentication method has proper write permissions. Storage Explorer provides detailed error messages that help diagnose issues.

Production Environment Considerations

When moving this solution to production:

  1. Replace the manual trigger with a Power Apps trigger for seamless user uploads
  2. Implement error handling with conditional steps to retry failed uploads
  3. Add logging to track successful/failed upload attempts
  4. Set lifecycle policies in Azure to automatically archive or delete old files
  5. Consider Azure CDN if files need global distribution with low latency

The same flow pattern can be extended to handle file updates, deletions, and metadata management by adding additional Azure Blob Storage actions to your Power Automate solution.
