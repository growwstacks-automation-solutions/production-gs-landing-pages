The Problem With Static Avatars

Traditional AI avatars suffer from what we call the "buffering face" problem. When the agent stops speaking, the avatar freezes with a glassy stare, making conversations feel unnatural and one-sided. At 1:15 in the video, you can see a side-by-side comparison showing how most avatars become statue-like during pauses.

This creates a poor user experience because human communication relies heavily on visual feedback during listening. Nods, eye movements, and subtle expressions signal engagement and understanding. Without these natural reactions, users feel like they're talking to a recording rather than a responsive agent.

Key insight: An avatar that only moves when speaking is like a video call where the other participant turns off their camera between sentences. The conversation feels broken because 70% of human communication happens through non-verbal cues.

How Runway Characters Works

Runway Characters uses GWM Avatars, a specialized version of Runway's general world model optimized for real-time avatar interactions. The system generates natural movements directly from the audio waveform, creating lip sync, eye movements, and head turns that match the speech rhythm and content.

What sets it apart is the continuous animation stream. Even during silent moments, the avatar maintains subtle idle movements and reactive expressions based on the conversation flow. This is powered by WebRTC streaming that keeps the avatar session alive throughout the interaction.

LiveKit Integration Architecture

The integration follows LiveKit's avatar worker pattern. When you add the Runway plugin, it creates a secondary participant that handles all avatar processing. Here's how the audio flow changes:

  1. The agent generates TTS audio as usual
  2. Instead of sending directly to the room, audio routes to the avatar worker
  3. The worker sends audio to Runway for lip sync processing
  4. Synchronized audio and video streams publish back to the room

From the frontend perspective, nothing changes. The agent appears as a single participant with both audio and video tracks, just like a human video call participant would.

The 3 Lines That Change Everything

The actual code change is remarkably simple. Here's all you need to add to an existing LiveKit agent:

 // 1. Add the Runway extra to your agent setup import { runwayCharacters } from '@livekit/agent-extras-runway'; // 2. Create an avatar session (preset or custom) const avatar = await runwayCharacters.createAvatar({   preset: 'professional' // or custom: { imageUrl: '...' } }); // 3. Start avatar before your main session await avatar.start(); await agent.start();

The order matters because the avatar.start() call rewires the agent's audio output pipeline. This minimal change means you can add realistic avatars to existing agents without refactoring your entire codebase.

Frontend Considerations

One of the biggest advantages of this integration is that it requires zero frontend changes. The LiveKit React components automatically handle the avatar's tracks because:

  • The worker publishes with kind: 'agent'
  • The lk-publish-on-behalf attribute points to your agent
  • Standard track handling renders both audio and video

This means your existing video call UI will work perfectly with avatars. No special layout logic or custom components needed - the system leverages LiveKit's built-in participant management.
