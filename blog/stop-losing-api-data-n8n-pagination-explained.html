The API Pagination Problem Most Businesses Face

If you've ever connected n8n to an API and wondered why you're only getting partial data, you've encountered pagination. This common frustration causes businesses to miss critical records, make decisions on incomplete data, and waste hours troubleshooting what appears to be broken API connections.

The truth is simple: APIs intentionally split large datasets into manageable chunks. Without proper pagination handling, you might only see the first 20 records while believing you've accessed the full dataset. This leads to inaccurate reporting, incomplete customer lists, and missed opportunities.

85% of API data loss in automation workflows stems from unhandled pagination. The solution isn't complex code - it's understanding how APIs structure their responses and configuring n8n accordingly.

Pokemon API Example: From 20 to 1350 Records

The Pokemon API perfectly illustrates the pagination challenge. A simple GET request to pokeapi.co/api/v2/pokemon returns just 20 Pokemon by default - despite the API containing 1350 characters. This limitation exists to prevent server overload and improve response times.

By examining the API response, we see critical pagination metadata:

  • count: 1350 (total available records)
  • next: URL for the subsequent page
  • previous: URL for the prior page (null on first page)

Without pagination handling, your automation would only process Bulbasaur through Raticate - missing 1330 other Pokemon. The solution lies in n8n's HTTP Request pagination settings.

Limit & Offset Technique Explained

Many APIs support two fundamental pagination parameters: limit and offset. The Pokemon API documentation explicitly describes these:

  • limit=60 - Returns 60 items per request (instead of default 20)
  • offset=60 - Skips the first 60 items (effectively "page 2")

In n8n, you implement these by appending query parameters to your API URL:

 https://pokeapi.co/api/v2/pokemon?limit=60&offset=60

This combination gives you precise control over batch sizes and starting positions. However, manually calculating offsets becomes impractical for large datasets. That's where n8n's automated pagination shines.

n8n Pagination Settings Demystified

The HTTP Request node's pagination options might appear intimidating, but they boil down to three critical configurations:

1. Pagination Mode

Set to "Response Contains Next URL" for APIs like Pokemon that provide direct links to subsequent pages.

2. Next Page URL

Configured as response.next to automatically follow the API's provided next page link.

3. Completion Condition

Set to "Other" with expression response.next - stops when next URL becomes null.

68 requests - That's how many automated calls n8n makes to collect all 1350 Pokemon records, handling each 20-item page seamlessly in the background.

Cleaning Up Paginated Data in n8n

Paginated API responses often arrive with metadata you don't need in your final dataset. The Pokemon API returns each page as:

  • count (total records)
  • next/previous URLs
  • results array (the actual Pokemon data)

To clean this in n8n:

  1. Use Edit Fields to extract just the results array
  2. Apply Split Out to transform arrays into table rows
  3. Optionally use another Edit Fields to select specific columns

This approach avoids messy JavaScript code while delivering clean, analysis-ready data.

Rick & Morty API: Different Pagination Approach

The Rick & Morty API demonstrates an alternative pagination structure. Instead of a direct next URL at the root level, it nests pagination controls inside an "info" object:

 {   "info": {     "count": 826,     "pages": 42,     "next": "https://rickandmortyapi.com/api/character?page=2",     "prev": null   },   "results": [...] }

In n8n, you configure this as:

  • Next Page URL: response.info.next
  • Completion: Still response.info.next becoming null

This example proves that while pagination implementations vary, the core concept remains consistent across APIs.

Handling Rate Limits with Pagination

Aggressive pagination can trigger API rate limits. n8n provides two safeguards:

1. Max Pages Fetched

Limit how many pages n8n processes (e.g., set to 10 for testing).

2. Request Interval

Add delays between requests (e.g., 1000ms) to stay under rate limits.

These controls help balance data collection with API provider requirements. Always check the API documentation for specific rate limiting guidelines.

Pro Tip: Start with conservative settings (e.g., 1 second between requests) and only increase speed after verifying the API can handle it.

Frequently Asked Questions

Common questions about API pagination in n8n

API pagination splits large datasets into smaller chunks returned across multiple API requests. This improves response times, reduces server load, and prevents timeouts when working with extensive datasets.

Just like websites show limited items per page (like Amazon showing 50 products per page), APIs use pagination to manage data delivery efficiently without overwhelming systems or users.

  • Performance: Faster responses with smaller payloads
  • Reliability: Prevents server overload and timeouts
  • Usability: Allows progressive loading of large datasets

Most APIs that return collections of data use pagination. The first indicators are receiving fewer items than expected or documentation mentioning limits.

Look for metadata in the response like total count, next page URL, previous page URL, or tokens to navigate through the complete dataset. The documentation will typically mention pagination parameters like limit and offset if supported.

  • Response headers: Check for pagination-related headers
  • Documentation: Search for "pagination" or "limit"
  • Trial: Request a large dataset and check if truncated

Limit controls how many items are returned per request (e.g., limit=60 returns 60 items). Offset determines the starting point in the dataset (e.g., offset=60 skips the first 60 items).

Combining them lets you page through results systematically. For example, limit=60&offset=60 gets items 61-120. This approach works well for static datasets but may have issues with real-time changing data.

  • Limit: Controls batch size (items per page)
  • Offset: Sets starting position (page number)
  • Together: Enable precise pagination control

In n8n, use the HTTP Request node's pagination options. Configure: 1) How to get the next page (usually via response.next or similar field), 2) When to stop paginating (when next URL is null), and 3) Optional limits or delays between requests.

The key is properly setting the "Response Contains Next URL" mode and correctly pointing to where the next page information lives in the API response (like response.next or response.info.next).

  • Mode: Response Contains Next URL
  • Next URL: response.next or similar path
  • Completion: When next URL becomes null

The two most common patterns are offset-based and cursor-based pagination. Offset-based uses limit and offset parameters (like the Pokemon API), while cursor-based uses tokens or pointers to the next set of results.

Some APIs use page numbers instead of offsets, and others implement more complex token-based systems. The Rick & Morty API example uses a cursor-like approach with next URLs contained in an info object.

  • Offset-based: Uses limit and offset parameters
  • Cursor-based: Uses opaque tokens for next page
  • Page-based: Uses simple page numbers

Set an interval between requests in n8n's pagination settings (e.g., 1000ms delay). You can also limit the total pages fetched if you only need a sample. Always check the API documentation for rate limit guidelines.

Some APIs allow more aggressive polling than others. When in doubt, start with conservative settings (like 1 request per second) and monitor for rate limit errors before increasing speed.

  • Interval: Add delay between requests
  • Limit: Restrict total pages fetched
  • Monitor: Watch for rate limit errors

After collecting paginated data, use n8n's Edit Fields node to extract just the results array, then use Split Out to transform arrays into usable table formats. This avoids messy code nodes and keeps your workflow clean.

For example, with the Pokemon API: 1) Edit Fields to keep only the results array, 2) Split Out to expand the array items into rows, and 3) Optional Edit Fields to select specific columns like name and URL.

  • Edit Fields: Extract relevant arrays
  • Split Out: Expand arrays to table rows
  • Edit Fields: Select final columns
