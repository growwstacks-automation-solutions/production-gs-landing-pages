How Vector Sync Works Behind the Scenes

When a document enters your Google Drive folder (like an updated employee handbook), our n8n workflow triggers immediately. It downloads the file, splits it into 1000-character chunks, and converts each chunk into a numerical vector using OpenAI's embeddings.

These vectors get stored in Supabase alongside metadata about their source document. Crucially, the workflow first identifies and removes any existing vectors from older versions of the same document - preventing duplicate or conflicting information.

At query time, when your AI agent receives a question, it converts that question into a vector and searches Supabase for the most relevant document chunks. Because we've maintained perfect synchronization, it always retrieves current information rather than outdated versions.

Step-by-Step n8n Setup

Step 1: Configure the Google Drive Trigger

In n8n, set up a Google Drive node that watches your designated folder for new or modified files. At 2:15 in the video, we show how to authenticate and select the exact folder to monitor.

Step 2: Download and Process Documents

Add a "Download File" node that grabs the document content whenever the trigger fires. We include automatic PDF conversion to handle various file types consistently.

Step 3: Connect to Supabase

Configure your Supabase credentials in n8n using the API keys from your project settings. The workflow needs permissions to query, insert, and delete records in your vectors table.

Step 4: Set Up Document Chunking

The "Text Splitter" node divides documents into 1000-character segments with some overlap to maintain context. This matches most embedding models' optimal input sizes.

Step 5: Generate and Store Embeddings

Each text chunk passes through OpenAI's embeddings model before being stored in Supabase with metadata linking it to the source document.

Pro Tip: Add error handling nodes after each step to catch and log issues without breaking the entire workflow. This is especially important for production systems.

Handling Document Changes and Deletions

While Google Drive triggers work perfectly for new and modified files, detecting deletions requires a secondary system. We implement a scheduled workflow that runs every 30 seconds to:

  1. List all files currently in the monitored folder
  2. Compare against documents stored in Supabase
  3. Remove vectors for any documents no longer present

This ensures your RAG agent won't reference documents that team members have removed from the shared drive. The cleanup process uses document IDs rather than names for perfect accuracy.

Performance Optimizations for Large Document Sets

For businesses with thousands of documents, we recommend these optimizations:

  • Batch processing: Group documents into batches of 200 for more efficient embedding generation
  • Selective syncing: Use multiple Google Drive folders with separate workflows for different document types
  • Metadata filtering: Add custom metadata tags to documents for more targeted vector searches
  • Parallel processing: Split the workflow to handle multiple documents simultaneously

These techniques can reduce processing time by 60-80% for large document collections while maintaining perfect synchronization.
