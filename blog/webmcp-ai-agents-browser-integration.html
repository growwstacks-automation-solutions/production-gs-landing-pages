The Screenshot Problem

Today's AI agents interact with websites like tourists in a foreign country without a map or language skills. When you ask an AI to "find a blue jacket under $50 and check out," it must take screenshots (costing ~2000 tokens each), send them to vision models, and guess where interface elements are located.

This approach creates three major problems. First, it's expensive - a simple product search can consume 10,000 tokens. Second, it's slow - each action requires multiple vision model calls taking seconds to complete. Third, it's fragile - any website redesign breaks all existing AI integrations overnight.

Cost comparison: Searching for running shoes with screenshots costs ~10,000 tokens versus just 40 tokens with WebMCP - an 89% reduction in processing costs.

How WebMCP Works

WebMCP (Web Model Context Protocol) provides a standardized way for websites to declare their capabilities to AI agents. Instead of guessing through screenshots, AI receives a structured menu of available actions with clear input requirements.

The protocol works through the browser's navigator.mmodelcontext API. Websites can expose tools either declaratively (adding attributes to existing HTML forms) or imperatively (registering JavaScript functions). Each tool includes a name, description, input schema, and execute function.

Key benefit: WebMCP tools survive website redesigns because they're based on functional contracts rather than visual layouts. Changing CSS or button positions doesn't break the AI integration.

Declarative vs Imperative Implementation

WebMCP offers two implementation paths suited for different complexity levels. The declarative approach works with existing HTML forms by simply adding tool name and tool description attributes. The browser automatically creates structured tool definitions from the form fields.

For more complex interactions, the imperative JavaScript approach provides finer control. Developers call navigator.mmodelcontext.registerTool() with an object defining the tool's name, description, input schema, and execute function. This function can call existing APIs, modify the DOM, or request user confirmation for sensitive actions.

Implementation tip: Start with declarative HTML for simple forms, then use the JavaScript API for complex workflows like checkout processes that require multiple steps and user confirmation.

WebMCP works through two primary methods that websites can use to expose functionality to AI agents. The declarative approach adds simple HTML attributes to existing forms, while the imperative approach uses JavaScript for more complex interactions.

The declarative method requires just two attributes added to HTML forms: tool-name and tool-description. The browser automatically creates structured tool definitions from the form fields, their types, and validation rules. This approach requires no JavaScript and works with existing form submissions.

  • Declarative: Add attributes to HTML forms
  • Imperative: Register tools via JavaScript API
  • Execute functions can call existing backend APIs

WebMCP tools remain functional through website redesigns because they're based on functional contracts rather than visual elements. When a website changes its CSS, layout, or button positions, the underlying tool definitions (what actions are available and what inputs they require) stay exactly the same.

This contract-based approach means AI integrations don't break when visual designs change. As long as the website continues to provide the same functionality (search products, add to cart, etc.), even complete visual overhauls won't affect the AI agent's ability to interact with the site. This reliability is impossible with screenshot-based approaches.

  • Tools defined by function, not appearance
  • Input contracts remain stable
  • No need to retrain vision models after redesigns

WebMCP and Anthropic's Model Context Protocol (MCP) serve complementary but distinct purposes in AI integration. WebMCP focuses on front-end web interactions in the browser, while MCP connects to back-end systems and databases.

The key difference is user visibility. WebMCP operates in the browser where users can see and confirm AI actions, while MCP works behind the scenes. WebMCP uses the browser's native API, while MCP uses JSON-RPC. Together they provide complete coverage - MCP for server-side operations and WebMCP for client-side interactions with user visibility.

  • WebMCP: Browser-based with user visibility
  • MCP: Backend-only operations
  • Together they enable complete AI integration
