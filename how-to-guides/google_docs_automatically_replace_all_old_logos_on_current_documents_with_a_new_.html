Google Docs - Automatically Replace All Old Logos on Current Documents with a New Logo

Introduction to Automated Logo Replacement

In today's fast-paced business environment, rebranding often involves updating company assets across numerous platforms and documents. One common, yet time-consuming, task is replacing old logos with new ones in existing Google Docs. Manually sifting through countless documents to update each instance of a logo can be a tedious and error-prone process. This guide introduces an efficient automation solution using Make.com (formerly Integromat) to streamline this task.

This powerful automation scenario is designed to retrieve Google Docs from a specified folder and automatically update their embedded logos. Rated at an intermediate difficulty level, the setup typically takes around 20 minutes. To get started, you'll need a Google account with an active Make.com connection, a Google Drive folder containing the documents you wish to modify, and a basic understanding of Make.com's Iterator and Array Aggregator functions. By following these steps, you can transform a laborious manual process into a quick and seamless automated workflow.

Configuring the Scenario

The core of this automation lies in a carefully structured Make.com scenario, which orchestrates the entire logo replacement process. This scenario leverages several key modules to interact with Google Docs and manipulate document content. Understanding how each module functions and how they connect is crucial for successful implementation.

The scenario is composed of the following modules:

  • Google Docs > List documents
  • Google Docs > Get content of a document
  • Iterator
  • Array Aggregator
  • Google Docs > Replace an image with a new image

This sequence ensures that documents are identified, their content is analyzed, the specific logo is isolated, and finally, the replacement is executed. Let’s delve into the configuration of each step.

Overview of the Make.com scenario for automating Google Docs logo replacement.

Google Docs > List documents

The first step in our automation journey begins with the "Google Docs > List documents" module. This module acts as the trigger, initiating the workflow by identifying all relevant Google Docs that require logo updates. Its primary function is to retrieve a list of documents from a designated folder within your Google Drive.

To configure this module, you simply need to select the specific folder ID that contains the Google Docs you intend to modify. By default, the module is set to retrieve a maximum of 10 results, but this can be adjusted to accommodate a larger number of documents if needed. This ensures that all target documents are included in the automation process.

Configuring the 'List documents' module to specify the target folder in Google Drive.

Google Docs > Get content of a document

Following the listing of documents, the "Google Docs > Get content of a document" module comes into play. This crucial module is responsible for fetching the complete content of each document identified in the previous step. By accessing the full content, the automation can then locate and interact with specific elements, such as embedded images.

To set this up, you'll either select a specific file (if you're testing with a single document) or, more commonly, toggle the map switch. This allows you to dynamically map the "Document ID" element outputted by the preceding "List documents" module to the "Document ID" field of this module. This dynamic mapping ensures that each document retrieved is processed sequentially.

Mapping the Document ID to retrieve the content of each Google Doc.

Iterator

The "Iterator" module is a fundamental flow control tool in Make.com, designed to break down a collection of items into individual bundles for separate processing. In this scenario, it plays a vital role after the "Get content of a document" module, as it converts the array of "Inline Objects" (which includes images) found within a document into distinct bundles of data.

To configure the Iterator, you can either click "Add item" or toggle the map switch. Then, map the "Inline Objects Array[]" element, which is outputted by the "Google Docs > Get content of a document" module, to the "Array" field of the Iterator. This action ensures that each image or inline object in your document is treated as a separate item, allowing for targeted manipulation in subsequent steps.

Setting up the Iterator to process each inline object within the document.

Array Aggregator

The "Array Aggregator" module serves as the counterpart to the Iterator, bringing together multiple bundles of data back into a single array. This is essential for consolidating the information needed to perform the image replacement operation in a structured manner. It effectively reverses the action of the Iterator, but with modified data.

For this module, select the "Iterator" as the "Source module." Under "Target structure type," choose "Images Replacement." Then, map the "Object ID" element, which is outputted by the Iterator, to the "Image Object ID" field. Crucially, you will also need to enter the public URL of your new logo in the "Image URL" field. This tells the aggregator which image to use for the replacement.

Configuring the Array Aggregator to prepare the new logo for insertion.

Google Docs > Replace an image with a new image

The final and most critical module in this automation is "Google Docs > Replace an image with a new image." This module executes the actual logo swap within your documents, leveraging the aggregated data from the previous steps. It's where the old logo is identified and seamlessly replaced with the new one you've provided.

Within this module, under "Replace an Image," select the "By Mapping" option. You'll then map the "Document ID" element (from the "Google Docs > Get content of a document" module) to the "Document ID" field. Next, toggle the map switch for the "Images Replacement" field and map the "Array[]" element outputted by the "Array Aggregator." This comprehensive mapping ensures that the correct document is targeted and the new image is inserted in place of the old one, completing the automated rebranding task.

Final module setup to replace the old image with the new one in Google Docs.

Scenario Execution

Once all modules are configured and connected, the scenario is ready for execution. Running the scenario will initiate the automated process, allowing you to observe how each step contributes to the final outcome. The efficiency of this workflow becomes evident as the system handles the document processing and image replacement seamlessly.

Upon execution, you will see the data bundles flow through each module, from the initial listing of documents to the final image replacement. This visual representation in Make.com confirms that the automation is working as intended, updating the logos in your Google Docs without any manual intervention required.
