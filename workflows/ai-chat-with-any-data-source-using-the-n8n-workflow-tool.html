What This Workflow Does

This n8n workflow template creates an AI-powered chat interface that can query and analyze data from any connected source. Instead of building separate chatbots for each database or application, you create one intelligent agent that can pull information from multiple workflows. The system interprets natural language questions, executes the appropriate data-fetching workflow, and returns formatted answers.

Businesses often struggle with data accessibility—critical information exists in siloed systems that require technical skills to query. This template solves that by letting anyone in your organization ask questions like "What were our top products last quarter?" or "Show me unresolved customer tickets" and get instant answers drawn directly from your live data.

The workflow acts as a bridge between conversational AI (like OpenAI's models) and your existing data infrastructure. It's particularly valuable for customer support teams needing quick access to account information, sales teams analyzing pipeline data, or executives seeking real-time business metrics without waiting for reports.

How It Works

Step 1: User asks a question

The workflow starts with a manual chat trigger or webhook that captures the user's natural language question. This could come from a Slack message, website chat widget, or internal dashboard interface.

Step 2: AI interprets the request

The question passes to an AI model (OpenAI Chat Model node) that analyzes the intent and determines what data is needed. The AI identifies key entities, timeframes, and metrics mentioned in the query.

Step 3: Execute data-fetching workflow

Using the "Execute Workflow" node, the system calls another n8n workflow specifically designed to fetch the required data. This secondary workflow could query databases, call APIs, or pull from spreadsheets.

Step 4: Process and format results

The raw data returns to the main workflow where it's cleaned, filtered, and formatted using n8n's data transformation nodes. The Code node can perform calculations, while the Set node structures the output.

Step 5: Generate conversational response

The formatted data goes back to the AI model, which creates a natural language response answering the original question. The response includes relevant data points in a conversational format.

Step 6: Deliver answer to user

The final answer returns through the original channel—Slack message, chat response, or dashboard update—completing the conversation loop.

Pro tip: Create separate data-fetching workflows for each data source (Salesforce, PostgreSQL, Google Sheets). Your main AI chat workflow can then call the appropriate one based on the question context, keeping your system modular and maintainable.

What You'll Need

  1. n8n instance (version 1.19.4 or later) – either self-hosted or n8n.cloud
  2. OpenAI API key or compatible LLM access for natural language processing
  3. Data source workflows already built in n8n that can fetch specific datasets
  4. Access credentials for your target data systems (database connections, API keys)
  5. Input channel such as Slack, webhook endpoint, or manual trigger interface

Quick Setup Guide

Follow these steps to implement this AI data chat system:

  1. Download and import the template JSON file into your n8n instance.
  2. Configure the OpenAI node with your API key and preferred model settings.
  3. Build or identify your data-fetching workflows that this main workflow will call.
  4. Update the "Execute Workflow" nodes to point to your specific data workflows.
  5. Test with sample questions to ensure the AI correctly routes to appropriate data sources.
  6. Connect your preferred interface (Slack, webhook, or manual trigger) to start the conversations.
  7. Train your team on how to ask effective questions and interpret the AI's responses.

Pro tip: Start with one well-defined data source (like "customer support tickets") before expanding to multiple systems. This lets you refine the prompt engineering and data formatting for that use case, then replicate the pattern for additional data types.
