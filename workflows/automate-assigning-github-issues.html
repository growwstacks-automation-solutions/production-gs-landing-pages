Automate GitHub Issue Assignment

Automatically assign GitHub issues to users who comment "assign me". Streamline project management and reduce manual triage with this free n8n automation template.

What This Workflow Does

Managing a busy GitHub repository often means manually triaging issues and assigning them to team members or contributors. This process is time-consuming, prone to delays, and can lead to confusion about ownership. This automation template solves that by listening for a specific trigger phrase—like "assign me"—in new issue titles or comments, and automatically assigning that user to the issue.

The workflow eliminates the back-and-forth communication typically required for task assignment. When a developer expresses interest in working on an issue, the system instantly grants them ownership, streamlining the contribution process. This is particularly valuable for open-source projects or large engineering teams where maintainers are overwhelmed with administrative overhead.

By automating this routine task, you ensure faster response times, clearer accountability, and a more engaged contributor community. It turns a manual, reactive process into a proactive, self-service system that scales with your project's growth.

How It Works

The workflow uses a simple but powerful logic chain to listen, evaluate, and act within your GitHub repository.

Step 1: Trigger on GitHub Activity

The automation is initiated by a webhook from GitHub. It listens for two key events: when a new issue is opened, or when a new comment is posted on an existing issue. This ensures the automation catches both new contributions and ongoing discussions.

Step 2: Parse the Content for the Trigger Phrase

Once triggered, the workflow extracts the text from the issue title or comment body. It then checks if the text contains the predefined trigger phrase, such as "assign me". This is done using a conditional logic node (like an "IF" node) to filter only the relevant activity.

Step 3: Identify the Requesting User

The workflow identifies the GitHub user who created the issue or posted the comment. It captures their username from the webhook payload. This user is the one who will be assigned to the task.

Step 4: Execute the Assignment

Finally, the workflow uses the GitHub API to update the issue, assigning the identified user to it. The assignment happens instantly, and the issue's status is updated in real-time for everyone on the team to see.

Pro tip: You can customize the trigger phrase to match your team's jargon, like "I'll take this" or "volunteer". You can also add additional checks, such as verifying the user is a repository collaborator before assigning.

What You'll Need

  1. A GitHub account with admin or write permissions to the repository you want to automate.
  2. Access to an n8n instance (either self-hosted or n8n.cloud).
  3. The GitHub node credentials configured in n8n, requiring a Personal Access Token (PAT) with repo scope.
  4. A basic understanding of webhooks (n8n will provide a URL for GitHub to send events to).
  5. (Optional) A test repository to validate the workflow before deploying it to production.

Quick Setup Guide

Follow these steps to get this automation running in your environment in under 10 minutes.

  1. Download the template: Click the "Download Template" button above to get the JSON file.
  2. Import into n8n: In your n8n dashboard, go to Workflows > Import from File and select the downloaded JSON.
  3. Configure GitHub credentials: In the workflow, click on the GitHub trigger and action nodes to set up your connection using a GitHub Personal Access Token.
  4. Set your webhook URL: Copy the webhook URL provided by n8n's GitHub trigger node and add it to your GitHub repository settings under Settings > Webhooks.
  5. Activate the workflow: Toggle the workflow to "Active" in n8n.
  6. Test it: Create a new issue or comment with "assign me" in your connected repo and watch the automation assign it instantly.

Pro tip: Start by testing in a private repository or a dedicated test repo. Use n8n's built-in execution history to monitor the workflow's actions and debug any permission issues before going live on your main project.
