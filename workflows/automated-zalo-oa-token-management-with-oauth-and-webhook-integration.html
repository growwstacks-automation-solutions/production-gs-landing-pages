What This Workflow Does

If you use Zalo Official Account for business messaging, marketing, or customer support, you know the headache: access tokens expire every few hours. Manually refreshing them is tedious, and if you forget, your entire Zalo automation grinds to a halt. This workflow solves that problem permanently.

It automatically refreshes your Zalo OA access token before it expires, stores it securely in a central location, and makes it available to any other automation you run. It also includes operational webhooks to peek at the current token or reset the cache if needed—all without requiring an external server or manual intervention.

Think of it as a dedicated, reliable assistant that ensures your connection to Zalo's API is always alive and ready. This foundational automation unlocks the true potential of Zalo for scalable business communication.

How It Works

The workflow runs on a scheduled trigger and uses OAuth 2.0 to maintain a valid token. Here's the step-by-step process:

1. Scheduled Trigger & Token Check

A schedule node runs at a configured interval (e.g., every 50 minutes). It first checks the globally stored token's expiry timestamp. If the token is still valid for a safe buffer period, the workflow can skip refresh to avoid unnecessary API calls.

2. OAuth Token Refresh

If the token is nearing expiry, the workflow calls Zalo's OAuth v4 endpoint with your stored refresh token, app ID, and secret key. It requests a fresh access token and receives the new token along with its expiry time.

3. Secure Token Storage

The new access token, refresh token (if a new one is provided), and expiry timestamp are saved into a global data store (like n8n's Static Data or a database). This central cache is encrypted and accessible only to your workflows.

4. Output & Readiness

The workflow outputs the current valid access token, making it immediately available for the next node in the chain. Downstream workflows can read the token from the global store at any time.

5. Operational Webhooks

Two webhook endpoints are provided: a token peek endpoint to retrieve the current token (for external systems) and a reset endpoint to clear the cache (useful during credential rotation or testing). These are secured with a secret parameter.

Pro tip: Set your schedule interval to be shorter than your token's lifetime minus a safety margin (e.g., if tokens last 1 hour, run every 50 minutes). This ensures you always refresh before expiry, even if a single run is delayed.

Quick Setup Guide

Get your automated token management running in under 15 minutes:

  1. Download & Import: Click the download button above to get the JSON file. In your n8n instance, go to Workflows → Import from File and select the downloaded JSON.
  2. Configure Credentials: In the "Set Refresh Token and App ID" node, replace the placeholder values with your actual Zalo app_id, secret_key, and refresh_token. Use environment variables for security.
  3. Adjust Schedule: Open the Schedule Trigger node. Set the interval based on your token's expiry (e.g., "Every 50 minutes").
  4. Secure Webhooks (Optional): In the Webhook nodes, add a query parameter or header check (like a secret key) to prevent unauthorized access to the peek and reset endpoints.
  5. Test & Activate: Click "Execute Workflow" once manually. Check the output to confirm a new token was fetched and stored. Then activate the workflow to let it run on schedule.

Pro tip: After activation, use the "token peek" webhook (with your secret) from another tool like Postman or a browser to verify the token is stored and accessible. This confirms the system is working end-to-end.

Zalo uses OAuth 2.0 for authentication. Your app gets an initial access token and a refresh token. The access token is short-lived (often 1-2 hours). Before it expires, your automation uses the refresh token to request a new access token from Zalo's API without requiring user re-authentication, keeping your integration alive indefinitely.

The refresh token itself has a longer lifespan (days, weeks, or indefinite). Your automation stores this securely. The workflow periodically exchanges it for a new access token, and if Zalo issues a new refresh token, it updates the stored one. This cycle maintains a perpetual, valid session.

Yes, that's a key advantage of centralized token management. Once you have a secure, automated system refreshing and storing the token, any other workflow (like sending broadcasts, replying to messages, or syncing contacts) can safely read and use that single token. This creates a unified, reliable foundation for all your Zalo automations.

This approach is far better than each workflow managing its own token, which leads to conflicts, redundant API calls, and inconsistent expiry states. Centralized management ensures consistency, reduces API load, and simplifies credential rotation.

If a refresh token expires or is revoked, the automated system will fail to obtain a new access token. Your workflow should include alerting (e.g., email or Slack notification) to inform you immediately. You'll then need to re-authenticate with Zalo to get a new refresh token and update your automation credentials.

To minimize disruption, design your alerting to trigger with ample warning. Monitor the success/failure of the refresh operation, not just the token expiry. This way, you're notified of problems before the current access token expires, giving you time to intervene.

When implemented correctly, it's secure. Use environment variables for your app secret and initial tokens—never hardcode them. Additional security measures include: using HTTPS for all webhooks, implementing IP allowlisting for administrative endpoints, regularly rotating credentials, and auditing access logs. Treat your automation tokens with the same care as database passwords or API keys.
