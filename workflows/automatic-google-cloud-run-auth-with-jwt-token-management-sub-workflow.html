Automatic Google Cloud Run Auth with JWT Token Management

A reusable authentication layer that manages JWT tokens for Google Cloud Run services, eliminating manual token wiring in every workflow.

What This Workflow Does

If you're calling Google Cloud Run services from n8n, you've likely faced the tedious task of managing authentication tokens manually. This sub-workflow solves that problem by creating a reusable authentication layer that handles JWT token lifecycle automatically.

The workflow checks if an existing ID token is still valid (with a 5-minute buffer for safety). If the token is good, it reuses it. If not, it signs a new short-lived JWT using your service account credentials, exchanges it at Google's token endpoint, and returns a fresh authentication token. This eliminates the need to wire authentication into every single workflow that calls Google Cloud Run services.

Beyond just token management, the workflow passes through your service URL and optional service path, so the calling workflow can immediately use the authenticated endpoint. It's designed to be called via n8n's "Execute Workflow" node from your main business logic workflows.

How It Works

1. Token Validation Check

The workflow first checks if an incoming ID token exists and is still valid. It uses a 5-minute buffer to ensure tokens don't expire during execution. This intelligent caching prevents unnecessary token generation calls.

2. JWT Generation & Exchange

If no valid token exists, the workflow creates a signed JWT using your Google service account private key. This JWT includes the necessary claims and is then exchanged at Google's OAuth2 token endpoint for a proper ID token that can authenticate to Cloud Run services.

3. Service Context Preservation

The workflow maintains your service URL and optional path parameters, ensuring the calling workflow receives both the fresh authentication token and the complete endpoint context needed for immediate API calls.

4. Error Handling & Retry Logic

Built-in error handling manages token exchange failures, network issues, and invalid credentials, with appropriate logging and fallback behavior to prevent workflow crashes.

What You'll Need

  1. n8n instance (cloud or self-hosted) with access to the HTTP Request and JWT nodes
  2. Google Cloud service account with appropriate permissions for your Cloud Run services
  3. Service account private key in PEM format for JWT signing
  4. Basic understanding of Google Cloud IAM and service account authentication
  5. Target Cloud Run service URL that you need to authenticate against

Pro tip: Store your service account credentials in n8n's built-in credential manager rather than hardcoding them in the workflow. This improves security and makes credential rotation much easier.

Quick Setup Guide

Follow these steps to implement this authentication layer in your n8n environment:

  1. Import the template into your n8n instance using the downloaded JSON file
  2. Add JWT (PEM) credential in n8n using your service account's private key
  3. Set workflow variables including your client_email (from the service account key) and confirm token_uri is https://oauth2.googleapis.com/token
  4. Test the sub-workflow by calling it from a simple test workflow with your service URL
  5. Integrate into main workflows using the Execute Workflow node, passing service_url and optional service_path parameters

The workflow is designed for loops—pair it with Merge/Split strategies to attach fresh authentication tokens to each item in batch processing scenarios.
