Automatically Assign Jira Service Management Reporter from Forwarded Emails

Fix broken customer attribution in Jira tickets created from forwarded emails. Automatically parse and assign the original sender as the reporter.

Download Template JSON · n8n compatible · Free

What This Workflow Does

This workflow solves a critical problem in IT support and customer service teams using Jira Service Management. When customers email a support address that gets forwarded to Jira, the system incorrectly attributes the ticket to the forwarding system instead of the actual customer. This breaks customer tracking, makes reporting inaccurate, and complicates follow-up communications.

The automation intelligently parses forwarded email content to extract the original sender's name and email address, then updates the Jira ticket to assign them as the proper reporter. If the customer doesn't exist in your Jira Service Management system, it automatically creates their profile before making the assignment.

By eliminating manual ticket updates, this workflow saves support teams 5-10 minutes per ticket, ensures accurate customer attribution, and provides cleaner data for service level agreement tracking and customer satisfaction analysis.

How It Works

1. Trigger on New Jira Issue

The workflow activates whenever a new issue is created in your specified Jira project. It captures all ticket details including description, reporter email, and project information.

2. Filter Forwarding Emails

An "If" node checks whether the ticket was created by one of your internal forwarding addresses (like support@company.com or helpdesk@company.com). The workflow only processes tickets from these specific sources.

3. Parse Customer Details from Email Body

A Code node uses regular expressions to scan the ticket description (which contains the forwarded email) and extract the original sender's name and email address from standard email headers.

4. Search for Existing Customer

The workflow queries your Jira Service Management instance to check if a customer with the extracted email already exists in the system.

5. Create New Customer Profile (If Needed)

If no matching customer is found, the automation creates a new customer profile in Jira Service Management using the parsed name and email information.

6. Assign Correct Reporter

Finally, the Jira node updates the ticket's "Reporter" field to the existing or newly created customer, ensuring proper attribution and tracking.

  2. Jira Software Cloud API credentials with Service Management permissions
  3. Jira project configured with email integration enabled
  4. List of internal forwarding email addresses used in your organization
  5. Basic understanding of your email forwarding format for potential regex adjustments

Pro tip: Test the workflow with a few sample forwarded emails first to ensure the regex pattern correctly extracts customer information from your specific email client format.

Quick Setup Guide

  1. Import the template into your n8n instance using the downloaded JSON file.
  2. Configure Jira credentials in the trigger, Jira, and HTTP Request nodes with your API access.
  3. Set your project in the Jira Trigger node to monitor the correct Jira project.
  4. Update forwarding addresses in the "Filter Forwarding Emails" node with your actual internal email addresses.
  5. Replace domain placeholder in both HTTP Request nodes with your Atlassian domain (e.g., company.atlassian.net).
  6. Activate the workflow and test with a forwarded email to verify correct reporter assignment.
