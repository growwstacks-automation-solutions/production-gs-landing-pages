CI Artifact Completeness Gate

Automatically validate Sentry release artifacts (dSYM or mapping files) before merging GitHub commits—ensuring every mobile release is debuggable.

What This Workflow Does

This workflow solves a critical problem in mobile app development: un-debuggable releases. When crashes occur, developers need symbolication files (dSYM for iOS, mapping.txt for Android) to translate cryptic stack traces into readable code locations. If these files are missing, debugging becomes impossible, leading to prolonged outages and frustrated users.

The automation acts as a quality gate in your CI/CD pipeline. Whenever a new commit is pushed to GitHub, it automatically checks the corresponding Sentry release for required artifacts. If dSYM files or both mapping.txt and proguard.txt are present, it updates the GitHub commit status to success, allowing the PR to merge. If artifacts are incomplete, the workflow fails silently—blocking the merge until the issue is fixed.

This ensures every release shipped to users is fully debuggable, reducing crash resolution time by up to 80% and eliminating human oversight in artifact validation.

How It Works

Step 1: GitHub Push Trigger

The workflow starts when a push event occurs in your GitHub repository. It captures the commit SHA, repository name, branch, and other metadata. This trigger ensures the automation runs for every new commit, providing real-time validation.

Step 2: Fetch Sentry Releases

Using Sentry's API, the workflow retrieves all releases for your project. It matches the release version with your build system's tagging convention to locate the correct release.

Step 3: Retrieve Release Artifacts

Once the release is identified, the workflow fetches the list of uploaded files. It scans for dSYM files (iOS) or both mapping.txt and proguard.txt (Android).

Step 4: Validation Logic

A custom code node runs validation rules: success if either dSYM exists or both Android mapping files are present. If neither condition is met, the workflow stops and prevents status update.

Step 5: Update GitHub Commit Status

Upon successful validation, the workflow posts a success status to the GitHub commit, displaying a green checkmark. This signals to your team that the release is ready and debuggable.

  2. GitHub repository with API credentials (repo status scope)
  3. Sentry project with org_slug, project_slug, and auth token
  4. Build process that uploads artifacts to Sentry releases
  5. Consistent version tagging between your build system and Sentry

Quick Setup Guide

  1. Download the template and import it into your n8n workspace.
  2. Configure the GitHub trigger node with your repository credentials.
  3. Update the Sentry HTTP request nodes with your org_slug and project_slug.
  4. Add your Sentry API credentials to authenticate requests.
  5. Test the workflow with a recent commit to verify artifact detection.
  6. Activate the workflow to start enforcing validation on all future pushes.
