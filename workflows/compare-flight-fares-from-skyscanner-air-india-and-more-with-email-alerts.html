What This Workflow Does

This automation solves a common but frustrating problem: manually checking multiple airline websites and booking platforms to find the best flight deals. The workflow automatically queries live fare data from sources like Skyscanner, Air India, IndiGo, and Akasa Air, compares prices in real-time, sorts the results, and sends you an email alert with the best options.

Instead of spending hours refreshing browser tabs, you set your search parameters once (routes, dates, budget) and let the system monitor prices 24/7. When a fare meets your criteria—whether it's the absolute cheapest or a significant drop from previous prices—you receive a formatted email summary immediately. This is particularly valuable for business travel managers, frequent flyers, and travel agencies who need to optimize travel budgets without constant manual oversight.

The business value is clear: reduced travel costs through better deal discovery, significant time savings for travel planners, and elimination of human error in fare comparison. For companies with regular employee travel, even a 10-15% reduction in average ticket price translates to substantial annual savings.

How It Works

The workflow follows a logical sequence to gather, process, and deliver flight information efficiently.

1. Scheduled Trigger Initiation

A Schedule Trigger node starts the workflow at predefined intervals—daily, twice daily, or even hourly during peak booking periods. This ensures you never miss a price drop because the system checks consistently without requiring your attention.

2. Search Parameter Configuration

A Set node defines your search criteria: departure and arrival airports/cities, travel dates, passenger count, cabin class, and any airline preferences. These parameters are structured into the proper format for each API endpoint.

3. Multi-Source Data Collection

HTTP Request nodes connect simultaneously to various flight data APIs. Skyscanner provides aggregated results from multiple booking platforms, while direct airline APIs (Air India, IndiGo, Akasa Air) offer official fares and availability. This multi-source approach ensures comprehensive coverage.

4. Data Normalization & Merging

Since each API returns data in different formats, Function nodes transform the responses into a consistent structure: airline, flight number, departure/arrival times, duration, stops, and price. Merge nodes then combine all results into a single dataset for comparison.

5. Intelligent Sorting & Filtering

The workflow sorts flights by total price (including taxes and fees) and applies any business rules you've configured—like excluding certain airlines, setting maximum layover times, or prioritizing direct flights. It can also compare current prices against historical data to identify genuine deals.

6. Alert Generation & Delivery

An Email node formats the top 3-5 best options into a clean, readable summary and sends it to your specified address. The alert includes key details: airline, times, total price, booking links, and how much you're saving compared to average prices.

Quick Setup Guide

Follow these steps to implement this fare comparison system in your n8n environment:

  1. Import the template using the Download button above. In your n8n editor, click the menu (three dots) → "Import from File" and select the downloaded JSON.
  2. Configure your Schedule Trigger to run at optimal times—consider time zones of your target airports and when airlines typically update fares.
  3. Set up API credentials in each HTTP Request node. For Skyscanner, sign up at RapidAPI; for airline APIs, check their developer portals. Store credentials securely using n8n's credential system.
  4. Customize search parameters in the Set node. Define your origin/destination airport codes (IATA format), date ranges, passenger count, and any airline filters.
  5. Configure your email alert with sender details, recipient addresses, and message formatting. Test with a single search first to verify the output looks correct.
  6. Activate the workflow and monitor initial runs. Check that all API calls succeed and data merges correctly before relying on it for important travel decisions.
