What This Workflow Does

This automation solves a critical business problem: data protection without manual effort. Database backups are essential for disaster recovery, compliance, and development workflows, but manual exports are time-consuming and easily forgotten.

The workflow automatically creates daily snapshots of all your Postgres tables, converting them to CSV format and storing them in a GitHub repository. It intelligently updates existing files when data changes and creates new backups for newly added tables, maintaining a complete version history of your database state.

Beyond basic backup, this system provides audit trails, enables easy data sharing across teams, and serves as a reliable source for analytics and reporting datasets. It transforms a reactive, manual task into a proactive, automated business process.

How It Works

1. Scheduled Trigger Initiation

Every 24 hours, the workflow automatically starts without any human intervention. The schedule trigger ensures consistent daily execution, eliminating the risk of missed backups during weekends, holidays, or team absences.

2. GitHub Repository Check

The system first connects to your specified GitHub repository and lists all existing CSV files. This inventory check prevents duplicate uploads and identifies which tables already have backups that might need updating.

3. Postgres Table Discovery

It queries your Postgres database to fetch all table names from the public schema. The workflow dynamically adapts to your database structure, automatically including new tables as they're added to your system.

4. Data Extraction & CSV Conversion

For each table, the workflow extracts all rows and converts them to clean CSV format. This standardized format ensures compatibility with analytics tools, spreadsheets, and other data processing systems.

5. Conditional Upload Logic

Intelligent decision-making: if a table already exists in GitHub, the file is updated; if it's new, a fresh file is created. This maintains version control while minimizing storage usage.

What You'll Need

  1. Postgres Database Access: Read credentials for the database you want to back up. The connection requires host, port, database name, username, and password.
  2. GitHub Repository: A repository where CSV files will be stored. You'll need repository write permissions and OAuth2 authentication configured in n8n.
  3. n8n Instance: Either n8n.cloud account or self-hosted n8n installation where the workflow will run.
  4. Basic Understanding: Familiarity with database concepts and GitHub operations helps with troubleshooting, though the workflow is designed to run autonomously.

Quick Setup Guide

  1. Download the Template: Click the download button above to get the JSON workflow file.
  2. Import to n8n: In your n8n instance, go to Workflows → Import from File and select the downloaded JSON.
  3. Configure Postgres Connection: Update the Postgres node with your database credentials (host, database, user, password).
  4. Set Up GitHub OAuth: Connect the GitHub node using OAuth2 authentication and specify your target repository.
  5. Adjust Schedule (Optional): Modify the Schedule Trigger node if you prefer different backup frequency (hourly, weekly, etc.).
  6. Test & Activate: Run the workflow once manually to verify connections, then activate it for automated daily execution.

Pro tip: Create a dedicated GitHub repository for database backups with appropriate branch protection rules. Consider implementing a cleanup workflow that archives old backups after a certain period to manage storage costs.
