Automate Microsoft Entra ID Secret Expiry Reports

Prevent service outages by automatically detecting and reporting expiring application secrets and certificates across your Microsoft Entra ID tenant.

Download Template JSON · n8n compatible · Free

What This Workflow Does

Expired client secrets and certificates in Microsoft Entra ID (formerly Azure AD) are a leading cause of unexpected application outages and failed integrations. When these credentials expire, any service or application relying on them for authentication immediately stops working—often during critical business hours.

This automation solves this problem by systematically scanning your entire Microsoft Entra ID tenant for application credentials approaching their expiration date. It identifies both client secrets (passwordCredentials) and certificates (keyCredentials), filters them based on your configured alert window, and generates a clear, actionable email report—only when action is needed.

Instead of relying on manual checks that are easily forgotten or missed, this workflow provides consistent, scheduled monitoring that scales with your organization. It transforms a reactive, panic-driven process into a proactive, controlled security practice.

How It Works

1. Fetch Application Credentials

The workflow uses Microsoft Graph API with appropriate application permissions to retrieve all registered applications in your Entra ID tenant. For each application, it extracts both client secrets and certificate metadata, including their display names, IDs, and—most importantly—their expiration dates.

2. Process and Normalize Data

Credentials are split into individual entries for uniform processing. Each entry is enriched with application context (name, App ID) and credential-specific details. The system calculates days remaining until expiration based on current date.

3. Apply Expiry Filter

Using your configured alert window (e.g., "warn me 14 days before expiry"), the workflow filters credentials that require attention. Credentials already expired or with sufficient remaining time are excluded from the report.

4. Generate HTML Report

A clean, sortable HTML table is constructed, organizing results by application name. Each entry shows credential type, name, ID, and days remaining, with visual highlighting for urgent items (e.g., <7 days).

5. Send Conditional Email

If expiring credentials are found, an email with the HTML report is sent to designated administrators. If no credentials require attention, the workflow completes silently—no unnecessary emails.

Pro tip: Replace the manual trigger with a Schedule Trigger to run this workflow daily or weekly automatically. This creates a "set and forget" monitoring system that consistently protects your environment.

Who This Is For

This automation is essential for IT administrators and DevOps teams managing Microsoft Entra ID tenants with multiple registered applications. Security teams benefit from the audit trail and compliance reporting. Managed Service Providers (MSPs) can use it to monitor multiple client tenants simultaneously. Application owners and development teams responsible for maintaining integrations that rely on Entra ID authentication will prevent unexpected downtime.

Any organization using Microsoft 365, Azure services, or custom applications integrated with Entra ID should implement this monitoring—especially those with compliance requirements (SOC 2, ISO 27001) that mandate regular credential rotation and monitoring.

What You'll Need

  1. Microsoft Entra ID Tenant with applications registered that use client secrets or certificates.
  2. App Registration in Entra ID with Microsoft Graph Application.Read.All application permission.
  3. n8n instance (cloud or self-hosted) with Microsoft Graph OAuth2 credentials configured for Client Credentials flow.
  4. Email sending capability configured in n8n (SMTP or integrated email service).
  5. Administrator email address where reports should be delivered.

Quick Setup Guide

  1. Download and import the template JSON file into your n8n instance.
  2. Configure Microsoft Graph credentials in n8n using your app registration's Client ID, Tenant ID, and Client Secret.
  3. Update the Set Variables node with your notification email and preferred alert window (e.g., 14 days).
  4. Test the workflow manually to verify permissions and email delivery.
  5. Replace the manual trigger with a Schedule Trigger for automated, regular execution.

Security note: This workflow reads credential metadata only—never the actual secret values. It requires appropriate permissions but follows the principle of least privilege by only accessing what's necessary for monitoring.
