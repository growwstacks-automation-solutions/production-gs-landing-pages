What This Workflow Does

Businesses working with location data often struggle with fragmented information. You might have GPS coordinates from mobile apps, IoT devices, or user check-ins, but raw coordinates lack business context. This creates manual work to look up addresses, check local times, or assess weather conditions—processes that slow down operations and create inconsistent customer experiences.

This workflow solves that by creating a single API endpoint that transforms basic latitude and longitude into 28 enriched data fields. It aggregates information from four free services in parallel: OpenStreetMap for addresses, TimezoneDB for local time, Sunrise-Sunset for daylight data, and OpenWeatherMap for current conditions. The result is a comprehensive location profile delivered in 1-3 seconds, ready for immediate use in your applications.

Instead of building and maintaining multiple API integrations, you get a production-ready solution that handles error cases, respects rate limits, and structures data consistently. Whether you're building a travel platform that needs local weather, a fleet management system tracking vehicle conditions, or a marketing tool personalizing messages by timezone, this workflow provides the foundation.

How It Works

The workflow follows a streamlined parallel processing approach to minimize response time while maximizing data completeness.

1. Webhook Trigger

The workflow activates via a webhook that accepts GET requests with latitude and longitude parameters. This creates a public API endpoint you can call from any application, device, or service. The webhook validates input coordinates and prepares them for parallel processing.

2. Parallel API Calls

Four API requests fire simultaneously to different free services: OpenStreetMap for reverse geocoding (address details), TimezoneDB for timezone information, Sunrise-Sunset for solar times, and OpenWeatherMap for current weather. This parallel approach reduces total response time from what would be 10+ seconds sequentially to just 1-3 seconds.

3. Data Processing & Merging

Each API response gets parsed and transformed into a consistent format. The Merge node combines all data streams into a single object, handling cases where some services might return errors or time out. Data normalization ensures consistent field names and units across all sources.

4. Response Formatting

The final step structures the enriched data into 28 clearly labeled fields including complete address components, timezone details with local datetime, comprehensive weather metrics, sunrise/sunset times with day length, and URLs for weather icons and country flags. The JSON response is optimized for easy parsing by downstream applications.

What You'll Need

  1. n8n instance (cloud or self-hosted) to run the workflow
  2. OpenWeatherMap API key (free tier: 60 calls/minute, 1M calls/month)
  3. TimezoneDB API key (free tier available with registration)
  4. Basic understanding of webhooks and API concepts
  5. Applications or devices that generate or consume location data

Pro tip: Implement caching in your application layer to stay well within free API rate limits. Most location data doesn't need real-time updates every second—caching for 5-30 minutes dramatically reduces API calls while maintaining acceptable freshness.

Quick Setup Guide

  1. Download and import the JSON template into your n8n instance
  2. Register for free API keys at OpenWeatherMap and TimezoneDB (takes 2 minutes)
  3. Configure credentials in the OpenWeatherMap node and HTTP Request node for TimezoneDB
  4. Activate the workflow to generate your webhook URL
  5. Test immediately with sample coordinates: ?lat=27.1751495&lon=78.0395673
  6. Integrate the endpoint into your applications via simple HTTP GET requests
