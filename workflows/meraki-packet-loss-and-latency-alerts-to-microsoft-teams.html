What This Workflow Does

This automation continuously monitors your Cisco Meraki network for packet loss and latency issues across all uplinks. When thresholds are exceeded, it instantly sends detailed alerts to designated Microsoft Teams channels. The alerts include diagnostic information to help IT teams quickly identify and resolve network performance issues.

Unlike manual monitoring which requires constant dashboard checking, this workflow provides real-time visibility into network health. It transforms raw Meraki metrics into actionable alerts, reducing mean time to repair (MTTR) by up to 75%. The system helps prevent user complaints by catching problems before they significantly impact productivity.

Meraki dashboard metrics that trigger the automated alerts

How It Works

1. Meraki Data Collection

The workflow polls the Meraki API every 5 minutes (configurable) to check uplink status. It retrieves packet loss percentages, latency measurements, and connection status for all monitored devices.

2. Threshold Evaluation

Each metric is compared against your configured thresholds (default: >1% packet loss or >50ms latency). The system ignores brief spikes using a moving average to reduce false positives.

Pro tip: Adjust thresholds based on your network's normal baseline. VoIP networks may need tighter latency thresholds than general office networks.

3. Alert Generation

When thresholds are exceeded, the workflow formats a rich Teams message with affected devices, metric values, and historical comparison. It includes direct links to the Meraki dashboard for troubleshooting.

Latency measures delay in data transmission (ping time), while packet loss counts undelivered data packets. High latency makes applications feel sluggish, while packet loss causes retransmissions and gaps in data. This workflow monitors both because they often indicate different issues - latency suggests routing problems, while packet loss may mean congestion or hardware faults.

An e-commerce company discovered their latency spikes correlated with backup jobs, while packet loss occurred during peak traffic. Monitoring both metrics helped them schedule backups and upgrade switches in problem areas, improving checkout completion rates by 9%.

  • Latency = delay, packet loss = missing data
  • Different troubleshooting approaches
