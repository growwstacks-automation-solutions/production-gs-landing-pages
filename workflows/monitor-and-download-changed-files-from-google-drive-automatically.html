What This Workflow Does

Manually checking a Google Drive folder for new or updated files is a repetitive, time-consuming task that leads to missed updates and version confusion. This automation solves that by acting as your intelligent file assistant.

The workflow monitors a specified Google Drive folder on a schedule you set (e.g., every hour, daily). It intelligently tracks when it last ran using a timestamp file, and on each execution, it searches for and downloads only files that have been added or modified since that last check. This ensures you have an up-to-date local copy without duplicates or manual effort.

It's perfect for teams that receive regular reports, invoices, submissions, or any documents via a shared Drive folder and need them automatically synced to a local server, cloud storage, or to trigger further processing.

How It Works

The workflow uses a smart, idempotent design to process files efficiently and reliably.

Step 1: Schedule Trigger & Timestamp Check

The workflow is initiated by a schedule trigger (like a cron job). Its first action is to look for a small control file (e.g., n8n_last_run.txt) within the same Google Drive folder. This file contains the ISO timestamp of the last successful run.

Step 2: Determine the "Changed Since" Time

If the timestamp file is found, the workflow reads it. If it's missing (first run or file deleted), it gracefully defaults to looking for files changed in the last 24 hours. This makes the workflow robust and beginner-friendly.

Step 3: Query Google Drive for Changes

Using the Google Drive API, the workflow queries the specified folder for all files where the modifiedTime is after the determined "changed since" timestamp. It efficiently retrieves only the metadata of files that need processing.

Step 4: Download Each New/Updated File

For each file identified, the workflow downloads the binary content to the local machine or server where n8n is running. You can configure the download path, file naming conventions, and error handling at this stage.

Step 5: Update the Timestamp for Next Run

After all files are processed successfully, the workflow creates or updates the n8n_last_run.txt file with the current time. This ensures the next run will only pick up files changed after this point, preventing any file from being processed twice.

Pro tip: You can easily extend this workflow to send a Slack notification with a list of downloaded files, move the files to another cloud service like Dropbox or S3, or extract text/data from them for further analysis.

What You'll Need

  1. A running n8n instance: This could be n8n.cloud, a self-hosted n8n on your server, or the desktop app.
  2. Google Drive API credentials: You'll need to set up OAuth 2.0 credentials in the Google Cloud Console and connect them to n8n. The workflow needs read access to the target folder.
  3. The Folder ID of your target Google Drive folder: This is the unique identifier found in the folder's URL. You'll paste this into the workflow nodes.
  4. Storage space: Ensure the server running n8n has sufficient disk space to store the downloaded files.

Quick Setup Guide

Follow these steps to import and configure this template in under 10 minutes.

  1. Download the template using the button above. You'll get a .json file.
  2. In your n8n instance, go to Workflows and click Import from file. Select the downloaded JSON file.
  3. Set up your Google Drive credentials in n8n if you haven't already. Go to Credentials > Add Credential and select Google Drive. Follow the OAuth flow to grant access.
  4. In the imported workflow, edit each Google Drive node (there are several). For each one, select your newly created credentials and paste your target Google Drive Folder ID into the appropriate field.
  5. Configure the Schedule Trigger node to your preferred frequency (e.g., every 30 minutes, daily at 9 AM).
  6. Test the workflow by clicking "Execute Workflow". Check the execution logs to see it find and process files.
  7. Once verified, activate the workflow to let it run automatically on your schedule.

Pro tip: Before going live, test with a folder containing a few dummy files. Verify the timestamp file is created and that subsequent runs only download files changed after the first run.
