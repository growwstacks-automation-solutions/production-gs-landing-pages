Monitor Dropbox Folders for New Files with Database Comparison

Automatically detect and process new files uploaded to Dropbox folders using a database to track processed files—eliminating manual checks and duplicate processing.

What This Workflow Does

Dropbox is a central hub for file sharing in many businesses, but manually checking folders for new uploads wastes time and leads to missed files. This workflow solves that by automating the detection of new files in monitored Dropbox folders using a database comparison approach.

Instead of relying on simple triggers that can miss renamed files or reprocess the same content, this system maintains a database record of every file processed. When a new file appears, the workflow compares it against the database, identifies truly new entries, and then triggers specialized sub‑workflows for processing—such as converting audio notes, generating social posts, or adding quotes to a CRM.

It acts as a central monitoring hub that can scale to watch multiple folders simultaneously, each triggering different actions based on the content type. This eliminates the daily manual folder checks that drain productivity and ensures files are processed immediately upon upload.

How It Works

Step 1: Dropbox Webhook Trigger

The workflow starts with a Dropbox webhook that listens for file changes. To prevent webhook disabling, it responds immediately to Dropbox, then branches out based on which folder received the update.

Step 2: Folder Path Configuration

A variable folder_to_watch is set for each branch, defining the specific Dropbox folder path to monitor. This keeps configuration clean and allows easy replication for additional folders.

Step 3: List Files & Filter

The Dropbox node lists all files in the target folder. The workflow filters out subfolders, focusing only on files that need processing.

Step 4: Database Query & Comparison

Simultaneously, the workflow queries a database (like NocoDB) for previously processed files from that folder. It compares the Dropbox file IDs against the database records.

Step 5: Identify New Files

Using a merge operation, the workflow isolates files whose IDs aren’t in the database—these are the genuinely new uploads. This method catches renamed files that simple name‑based checks would miss.

Step 6: Record & Trigger Processing

Each new file is logged in the database with its full Dropbox metadata (ID, name, timestamps, hash, etc.). Then, a specialized sub‑workflow is called to handle the file based on its folder type—audio processing, transcription, quote ingestion, etc.

What You'll Need

  1. A Dropbox account with API access.
  2. A database (NocoDB, PostgreSQL, MySQL, etc.) to store processed file records.
  3. n8n instance (cloud or self‑hosted) with Dropbox and database nodes configured.
  4. Sub‑workflows defined for each folder’s specific processing tasks.
  5. Webhook setup in Dropbox to trigger the workflow.

Pro tip: Use a separate database table with columns for folder_to_watch, file_id, data (JSON), and timestamp. This structure supports scaling to multiple folders and provides auditability.

Quick Setup Guide

  1. Import the template JSON into your n8n workspace.
  2. Configure your Dropbox connection in the “Dropbox” node.
  3. Set up your database connection (NocoDB node in this template).
  4. Define your folder_to_watch variable for each monitored folder.
  5. Create sub‑workflows for each folder’s processing logic and connect them.
  6. Set up the Dropbox webhook and activate the workflow.
  7. Test by uploading a file to your monitored folder and verifying the database entry and sub‑workflow execution.
