What This Workflow Does

Public webhooks are vulnerable to abuse. Without protection, malicious actors or malfunctioning systems can flood your endpoint with thousands of requests, overwhelming your servers, triggering expensive API calls, and potentially causing service outages. This n8n workflow adds production-grade rate limiting to any webhook in minutes, rejecting abusive traffic before your expensive business logic executes.

The workflow integrates with Ainoflow Guard to provide edge-style decisions at the application level. It checks incoming requests against configurable rate limits (like 30 requests per minute) and immediately returns 429 Too Many Requests responses to clients who exceed their quota. This protects your infrastructure, controls costs, and maintains service availability for legitimate users.

Unlike network-level protection, this solution operates within your automation logic, allowing for sophisticated decisions based on API keys, user identity, or specific business rules. It's stateless, proxy-aware, and designed to fail-open so Guard outages don't block your production traffic.

How It Works

Step 1: Webhook Receives Request

The workflow starts with a standard n8n webhook node that listens for POST requests. This could be any public endpoint you expose for third-party services, internal systems, or customer integrations.

Step 2: Identity Extraction

The system extracts the client's identity from request headers. In IP mode, it reads the X-Forwarded-For or x-real-ip header to get the real client IP (even behind proxies). In API key mode, it extracts the x-api-key header for per-client limiting. This identity becomes the key for rate limiting decisions.

Step 3: Guard Decision

The workflow calls Ainoflow Guard's API with the extracted identity and your configured rate limits. Guard checks if this client has exceeded their quota for the current time window. The decision happens in milliseconds before any business logic runs.

Step 4: Allowed Path

If the request is within limits, it proceeds to your business logic. You can process the webhook data, make API calls, update databases, or trigger other automations. The workflow then returns a 200 OK response to the client.

Step 5: Denied Path

If the client exceeds their rate limit, the workflow immediately returns a 429 Too Many Requests response with a Retry-After header. The denied request doesn't count toward their limit, and no expensive processing occurs on your end.

Quick Setup Guide

Follow these steps to implement webhook rate limiting in your n8n environment:

  1. Download and Import: Download the template file and import it into your n8n instance via the workflow import feature.
  2. Configure Ainoflow Credentials: Add your Ainoflow Bearer token to the GuardCheck HTTP Request node credentials.
  3. Set Your Limits: Edit the Config node with your desired rate_limit (e.g., 30), window_sec (e.g., 60), identity_mode (ip or apiKey), and route_name.
  4. Add Your Business Logic: Replace the placeholder BusinessLogic node with your actual webhook processing workflow.
  5. Activate and Test: Activate the workflow and test with rapid requests to verify rate limiting works correctly.

IP limiting works well for contact forms, public APIs, or any endpoint without authentication. API key limiting is better for B2B integrations, partner APIs, or customer-facing webhooks where each client has unique credentials. This workflow supports both modes with automatic fallback.
