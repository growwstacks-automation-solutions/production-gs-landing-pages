What This Workflow Does

When your website, API, or critical service goes down, every minute of delay costs money and damages customer trust. Manually triaging alerts, creating Jira tickets, and figuring out who's on call wastes precious time. This automation solves that by creating a seamless bridge between your monitoring tools, your ticketing system, and your team's communication platform.

The workflow listens for webhook alerts from any uptime monitoring service (like UptimeRobot, Pingdom, or custom checks). When a critical "down" status is detected, it instantly creates a detailed Jira incident ticket with all relevant context—service name, timestamp, error details, and customer impact. Then, it performs the real magic: it checks your designated Slack on-call channel, evaluates each member's current presence status, and intelligently selects the best available person to notify via a direct message containing all incident details and a direct link to the Jira ticket.

This end-to-end automation ensures incidents are logged, assigned, and communicated within seconds—not minutes or hours—dramatically reducing your mean time to resolution (MTTR) and eliminating alert fatigue.

How It Works

The workflow is built with a logical, fault-tolerant sequence designed for production reliability.

Step 1: Webhook Trigger & Validation

The workflow is initiated by an incoming HTTP webhook from your monitoring tool. The payload is parsed, and a conditional check (IF node) filters for alerts where the status equals "down" or "critical." Only genuine incidents proceed, preventing false positives from creating noise.

Step 2: Jira Incident Creation

Using the Jira node, the workflow creates a new Task or Incident issue in your specified project. The ticket summary, description, priority, and labels are dynamically populated from the alert data, creating a complete audit trail from the moment the outage was detected.

Step 3: Fetch & Evaluate On-Call Team

The workflow calls the Slack API to retrieve all members of your designated on-call channel. It then loops through each member, using the Slack node to fetch their real-time presence status (active, away, offline).

Step 4: Smart Member Selection

A custom Code node applies the selection logic: it first looks for all "active" users and picks one at random. If only one member is active, they are selected. If no one is active, it defaults to the first member in the list as a fallback, ensuring someone is always notified.

Step 5: Targeted Slack Notification

A formatted direct message is sent to the selected on-call engineer via Slack. The message includes the service name, downtime timestamp, Jira ticket link, error code, and a clear call to action, providing everything needed for a swift response.

What You'll Need

  1. An active n8n instance (cloud or self-hosted).
  2. Slack API credentials (a bot token with the users:read and chat:write scopes).
  3. Jira Cloud credentials with permissions to create issues in your desired project.
  4. A monitoring service capable of sending webhooks (e.g., UptimeRobot, Uptime Kuma, StatusCake, Datadog, or a custom script).
  5. A dedicated Slack channel containing your on-call team members.

Pro tip: Test the webhook format from your monitoring tool before connecting the workflow. Use a tool like Postman or webhook.site to capture a sample payload and ensure the field names (like status, serviceName) match what the workflow expects.

Quick Setup Guide

Follow these steps to deploy this automation in under 15 minutes.

  1. Import the Template: Click the "Download Template" button above and import the JSON file into your n8n instance.
  2. Configure Credentials: Set up the Slack and Jira nodes with your API keys. For Slack, create a new app in your workspace and install it with the required scopes.
  3. Set Your Slack Channel: In the "Slack: Get Channel Members" node, replace the placeholder Channel ID with your actual on-call channel ID.
  4. Configure Jira Project: In the "Jira: Create Issue" node, select your project and set the issue type to "Task" or "Incident."
  5. Test the Webhook: Activate the workflow and copy its unique webhook URL. Configure your monitoring tool to send POST requests to this URL when an alert triggers.
  6. Dry Run: Send a test alert from your monitoring tool or using a curl command to verify the entire flow—from Jira ticket creation to Slack DM.
