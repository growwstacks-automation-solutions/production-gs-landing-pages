What This Workflow Does

This workflow solves the problem of manually monitoring MQTT message queues from IoT devices, sensors, or industrial equipment. Instead of having staff constantly watch dashboards or log files, it automatically listens to specified MQTT topics, captures incoming messages, and processes them based on content.

The template acts as a bridge between your physical devices and business systems. When a device publishes data (like temperature readings, location updates, or status alerts), the workflow receives it, can filter for specific conditions, transform the data, and route it to databases, notification channels, or other applications. This turns raw sensor data into actionable business intelligence without human intervention.

Common use cases include smart building management (monitoring HVAC systems), fleet tracking (vehicle location updates), industrial equipment monitoring (predictive maintenance alerts), energy consumption analysis, and agricultural sensor networks. By automating this pipeline, you ensure real-time responsiveness while freeing technical teams from repetitive monitoring tasks.

How It Works

Step 1: MQTT Trigger Connection

The workflow starts with an MQTT Trigger node configured to connect to your MQTT broker (like Mosquitto, AWS IoT Core, or Azure IoT Hub). It subscribes to specific topics you define—such as devices/temperature or vehicles/status—and waits for incoming messages.

Step 2: Message Reception & Parsing

When a device publishes a message to the subscribed topic, the trigger node captures it. The workflow then extracts the message payload, which typically contains JSON, XML, or plain text data from the sensor or device.

Step 3: Data Validation & Filtering

Optional filter nodes can check message content against thresholds or patterns. For example, only process messages where temperature exceeds 30°C, or ignore status messages that are "normal". This prevents unnecessary processing of irrelevant data.

Step 4: Data Transformation & Routing

The workflow transforms the raw message into structured data suitable for downstream systems. It might convert units, add timestamps, or enrich with location metadata. Then it routes the processed data to destinations like databases (MySQL, PostgreSQL), analytics platforms, Slack alerts, email notifications, or custom APIs.

Step 5: Action Triggering

Based on message content, the workflow can trigger additional actions—like sending an alert to maintenance staff if equipment vibration exceeds limits, updating a dashboard visualization, or logging an incident in a CRM system.

What You'll Need

  1. An MQTT broker (self-hosted like Mosquitto or cloud-based like AWS IoT Core, Azure IoT Hub, Google Cloud IoT Core).
  2. MQTT-enabled devices or sensors publishing data to topics.
  3. Access credentials (username/password or certificates) for the MQTT broker.
  4. n8n instance (cloud or self-hosted) to run the workflow.
  5. Destination systems ready to receive processed data (database, notification tool, API).

Quick Setup Guide

  1. Download the template using the button above and import it into your n8n instance.
  2. Configure the MQTT Trigger node with your broker URL, port, topics to subscribe to, and authentication details.
  3. Adjust filter conditions if you only want to process specific message patterns (e.g., values above thresholds).
  4. Set up destination nodes like database insert, Slack message, or email notification with your specific connection details.
  5. Test with sample messages by publishing to your MQTT topics and verifying the workflow processes them correctly.
  6. Activate the workflow and let it run continuously, monitoring your MQTT queues automatically.

Pro tip: Use topic hierarchies (like building/floor1/roomA/temperature) to organize messages and make filtering easier. Also consider adding a deduplication step if devices occasionally send duplicate messages.

MQTT (Message Queuing Telemetry Transport) is a lightweight messaging protocol designed for IoT devices and low-bandwidth networks. It enables efficient communication between sensors, devices, and servers, making it essential for real-time monitoring, smart home systems, industrial automation, and energy management.

Its publish-subscribe model allows devices to send data to topics without knowing the recipients, simplifying architecture. For businesses, automating MQTT message processing means immediate reaction to device events without manual oversight.

MQTT is publish-subscribe based with minimal overhead, ideal for constrained networks and real-time updates. HTTP is request-response based with higher overhead, suited for web interactions.
