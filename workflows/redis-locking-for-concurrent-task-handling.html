What This Workflow Does

This n8n workflow implements Redis-based distributed locking to prevent concurrent execution of critical tasks. When multiple workflow instances could run simultaneously (through scheduling, webhooks, or manual triggers), Redis locking ensures only one instance proceeds while others wait or exit gracefully.

The solution solves common automation problems like duplicate API calls, race conditions in data processing, and overlapping scheduled jobs. By using Redis as a coordination point, it works reliably even across distributed n8n instances in high-availability setups.

How It Works

1. Lock Acquisition

The workflow attempts to acquire a Redis lock with a unique key at startup. The lock includes an expiration time (TTL) to prevent deadlocks if the workflow fails.

2. Execution Control

If the lock is acquired successfully, the workflow proceeds with normal execution. If the lock already exists (indicating another instance is running), the workflow either waits with retries or exits based on configuration.

3. Lock Release

Upon successful completion, the workflow explicitly releases the lock. If the workflow fails mid-execution, the lock automatically expires after the TTL period.
