Reduce LLM Costs with Semantic Caching

Stop paying for duplicate AI responses. Cache similar questions using Redis vector search and HuggingFace embeddings to cut API costs by up to 80%.

What This Workflow Does

Every time your AI chatbot answers "What's your refund policy?" and then "How do I get my money back?" as separate API calls, you're paying twice for essentially the same answer. This semantic caching workflow solves that problem intelligently.

Instead of treating questions as exact string matches, it understands meaning. Using HuggingFace embeddings, it converts questions into mathematical vectors that capture semantic similarity. Redis Vector Store then performs lightning-fast similarity searches to find previously answered questions that mean the same thing. When a match is found above your configured threshold, the cached answer returns instantly—no LLM call, no cost, no latency.

This isn't just keyword matching. It understands that "What are your business hours?" and "When are you open?" should return the same answer. For customer support, FAQ systems, and internal knowledge bases, this can reduce LLM API costs by 50-80% while improving response times from seconds to milliseconds.

How It Works

Step 1: Question Processing & Embedding

When a user submits a question, the workflow first normalizes the text and sends it to HuggingFace's embedding model. This converts the natural language question into a high-dimensional vector—a numerical representation of its meaning that machines can compare mathematically.

Step 2: Vector Similarity Search

The generated embedding is queried against Redis Vector Store, which contains embeddings of all previously asked questions along with their answers. Redis performs a cosine similarity search to find the most semantically similar questions in its database based on your configured threshold.

Step 3: Smart Routing Decision

If Redis returns a match with sufficient similarity (typically 0.3-0.4 cosine similarity), the workflow immediately returns the cached answer. If no good match is found, the question proceeds to the LLM (like OpenAI GPT) for a fresh response.

Step 4: Response Generation & Caching

For cache misses, the LLM generates a new answer. This fresh response, along with the original question embedding, is then stored in Redis for future use. The system also maintains conversation context, ensuring cached answers remain relevant within multi-turn dialogues.

What You'll Need

  1. Redis with vector search capabilities (Redis Stack 7.2+ recommended)
  2. HuggingFace API key (free tier available for the sentence-transformers model)
  3. OpenAI API key or access to another LLM provider (Anthropic, Google, etc.)
  4. n8n instance (self-hosted or n8n.cloud)
  5. Basic understanding of API keys and environment variables

Quick Setup Guide

  1. Download and import the JSON template into your n8n instance.
  2. Configure credentials for HuggingFace Inference API and your chosen LLM provider in n8n.
  3. Set up Redis connection with vector search index configured for your embedding dimensions.
  4. Adjust similarity threshold in the "Analyze results from store" node (start with 0.35).
  5. Test with sample questions to verify caching behavior before deploying to production.
  6. Monitor cache hit rates and adjust thresholds based on your actual query patterns.

Pro tip: Start with a higher similarity threshold (0.4-0.5) and gradually lower it as you build confidence in the system. This prevents false matches while the cache is still small.
