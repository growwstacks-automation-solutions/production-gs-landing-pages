Scan Confluence Pages for Inactive Owners

Automatically identify orphaned documentation and maintain clean, up-to-date knowledge bases with this free n8n workflow template.

What This Workflow Does

As companies grow, their Confluence knowledge bases often become cluttered with pages owned by employees who have left the company or changed roles. These "orphaned" pages create significant problems: outdated information persists, security permissions remain unchanged, and teams waste time searching for current documentation owners.

This automated workflow solves this problem by systematically scanning selected Confluence spaces, identifying page owners, checking their account status, and filtering out pages where the owner is inactive. It provides a clear, actionable report that helps documentation managers maintain governance, reduce security risks, and ensure knowledge base integrity.

How It Works

Step 1: Configure Space Selection

The workflow begins by defining which Confluence spaces to scan. You specify space keys (like "ENG", "HR", "MARKETING") and your Atlassian domain. This targeted approach lets you focus on critical documentation areas without scanning irrelevant spaces.

Step 2: Retrieve Spaces and Pages

Using Confluence's REST API v2, the workflow fetches all spaces matching your criteria, then retrieves every page within those spaces. It handles pagination automatically, ensuring no pages are missed even in large documentation sets.

Step 3: Resolve Page Ownership

The workflow extracts unique owner IDs from all pages and uses Atlassian's bulk user lookup API to efficiently resolve account status. This approach is far more efficient than checking each user individually, especially when dealing with hundreds or thousands of pages.

Step 4: Filter and Report

Pages are filtered to include only those where the owner's account status is not "active." The workflow generates a comprehensive report with page titles, owner emails, account status, last updated dates, and direct URLs for easy follow-up.

What You'll Need

  1. Confluence Cloud or Server instance with administrative or appropriate read permissions
  2. Atlassian API credentials (email + API token) with access to spaces, pages, and user data
  3. n8n instance (self-hosted or cloud) to run the workflow
  4. List of space keys you want to scan (comma-separated)
  5. Basic understanding of your Confluence space structure and documentation hierarchy

Pro tip: Start with a single, non-critical space to test the workflow before scanning your entire Confluence instance. This helps you understand the output format and adjust filters as needed.

Quick Setup Guide

  1. Download and import the template JSON file into your n8n instance
  2. Create HTTP Basic Auth credentials using your Atlassian email and API token
  3. Configure the "Set Variables" node with your Atlassian domain and space keys
  4. Assign credentials to all HTTP Request nodes in the workflow
  5. Test with a single space to verify the workflow connects and returns expected data
  6. Schedule execution (monthly or quarterly) or trigger manually as needed
