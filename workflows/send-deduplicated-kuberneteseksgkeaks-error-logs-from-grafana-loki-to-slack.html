What This Workflow Does

Kubernetes environments generate massive volumes of logs. When something goes wrong—a pod crashes, a service times out, or memory limits are exceeded—the same error can repeat hundreds of times, flooding your monitoring channels with duplicate alerts. This creates alert fatigue: teams start ignoring notifications, and critical issues get missed.

This n8n workflow solves that problem by acting as an intelligent filter between Grafana Loki (your centralized logging system) and Slack (your team's communication hub). It queries Loki every 5 minutes for error logs, extracts the meaningful metadata (pod, namespace, container, timestamp), removes duplicate entries within that time window, and sends only unique, actionable alerts to a designated Slack channel.

The result: your DevOps or SRE team receives clean, context-rich notifications about genuine problems, without the noise of repetitive log entries. You maintain visibility into cluster health while dramatically reducing notification overload.

How It Works

1. Scheduled Trigger

The workflow activates on a configurable schedule (default: every 5 minutes). This polling interval balances responsiveness with system load, ensuring you catch issues quickly without overwhelming your Loki instance.

2. Loki Query Execution

An HTTP request node queries your Grafana Loki instance for logs from the last 10 minutes, filtering for keywords like "error", "failed", "timeout", "exception", and "OOM". The query uses Loki's LogQL syntax to target only critical-level entries across all namespaces.

3. Log Parsing & Field Extraction

Each log result is parsed to extract structured fields: pod name, namespace, container, node, exact error message, and timestamp. Empty or malformed results are automatically skipped to ensure data quality.

4. Intelligent Deduplication

A custom code node compares incoming logs against previously seen entries within the time window. If the same error (matching pod, message, and container) appears multiple times, only the first instance is passed forward. This eliminates repetitive alerts while preserving new error patterns.

5. Formatted Slack Notification

Each unique alert is formatted into a clear Slack message with emoji indicators, colored borders for severity, and all metadata presented in an easy-to-scan layout. The message includes direct links to your Grafana dashboard for deeper investigation.

What You'll Need

  1. Grafana Loki Instance – Either self-hosted or cloud-managed, with API access from your n8n environment.
  2. Slack Workspace – With permissions to create a dedicated channel for alerts (e.g., #k8s-alerts).
  3. Slack App & Token – A Slack app with the chat:write OAuth scope installed to your workspace.
  4. n8n Instance – Cloud or self-hosted, with network access to both Loki and Slack APIs.
  5. Basic Kubernetes Knowledge – Understanding of pods, namespaces, and common error patterns.

Pro tip: Start with a dedicated Slack channel like #k8s-alerts-test to verify the workflow works correctly before routing to production channels. This lets you tune keyword filters and deduplication logic without disturbing your team.

Quick Setup Guide

1. Download the template using the button above and import it into your n8n instance.

2. Configure the Loki HTTP Request node with your Loki API URL (e.g., http://loki-gateway.monitoring.svc.cluster.local).

3. Set up Slack credentials in n8n using OAuth2, ensuring the token has chat:write permissions.

4. Update the Slack channel ID in the "Send Message to Slack" node to your desired destination.

5. Adjust the schedule trigger if needed—5 minutes is recommended, but you can increase or decrease based on your alert sensitivity requirements.

6. Activate the workflow and monitor the #k8s-alerts channel for test notifications.

7. Verify network access – ensure your n8n instance can reach both Loki and Slack APIs.
