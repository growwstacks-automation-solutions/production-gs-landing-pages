What This Workflow Does

When an API call fails in your Azure environment, finding the root cause often means manually searching through Application Insights, API Management logs, and Service Bus messages—a process that can take hours during critical incidents. This workflow automates that entire investigation process.

It queries Application Insights for failed requests, correlates them with Service Bus message traces and exceptions using operation IDs, and generates a comprehensive failure report. You get a complete picture of what broke, where, and why—delivered automatically as a structured report, Excel export, or API response.

This turns reactive troubleshooting into proactive monitoring, giving your DevOps team the insights they need to fix issues faster and prevent recurrences.

How It Works

Step 1: Configuration & Authentication

The workflow begins by securely loading your Azure credentials and query parameters. It authenticates with Azure AD using a service principal with Monitoring Reader role, ensuring secure access to your Application Insights data without storing sensitive credentials in plain text.

Step 2: Query Application Insights

A single optimized query retrieves three key data types from Application Insights: failed APIM requests, Service Bus operation traces, and related exceptions. The workflow uses Kusto Query Language (KQL) with time-range filters (24h, 7d, or 30d) to focus on relevant failures.

Step 3: Correlate Data Across Services

Using operation IDs that flow through Azure's distributed tracing system, the workflow links API requests to specific Service Bus messages and exceptions. This correlation reveals the complete failure chain—from the initial API call through message processing to the actual error.

Step 4: Generate Analysis & Reports

The workflow analyzes the correlated data to identify patterns, calculate failure rates, and determine root causes. It then generates detailed reports in multiple formats: Markdown for team communication, Excel for audit trails, or JSON for integration with other systems.

Step 5: Output & Integration

Results can be sent to various destinations: saved to cloud storage, posted to Slack/Teams channels, emailed to on-call engineers, or returned via webhook for integration with incident management platforms like PagerDuty or ServiceNow.

Pro tip: Schedule this workflow to run hourly and alert only when failure thresholds are exceeded. This gives you continuous monitoring without notification fatigue, focusing attention only when attention is needed.

What You'll Need

  1. Azure Application Insights resource with diagnostic data flowing from your services
  2. Azure API Management (APIM) instance configured to send logs to Application Insights
  3. Azure Service Bus namespace with diagnostic settings enabled for Application Insights logging
  4. Service Principal with "Monitoring Reader" role assigned to your Application Insights resource
  5. n8n instance (cloud or self-hosted version 1.0+) with internet access to Azure APIs
  6. Basic understanding of Azure CLI for initial service principal setup (scripts provided)

Quick Setup Guide

  1. Download the template using the button above and import it into your n8n instance.
  2. Create Azure Service Principal using the provided Azure CLI command in the "Set Configuration" node notes.
  3. Update credentials in the "Set Configuration" node with your Application Insights App ID, client ID, client secret, and tenant ID.
  4. Verify diagnostic logging is enabled on your APIM and Service Bus resources, sending data to Application Insights.
  5. Test the workflow by clicking "Execute Workflow" and reviewing the generated report in the "Generate Report" node.
  6. Configure outputs by enabling the Excel export or webhook response nodes based on your needs.
  7. Schedule execution (optional) using n8n's schedule trigger to run hourly or daily for continuous monitoring.
