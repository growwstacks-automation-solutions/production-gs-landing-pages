What This Workflow Does

This workflow solves a critical problem for any business exposing APIs: preventing abuse and ensuring fair resource usage. Without rate limiting, a single user or malicious actor can overwhelm your API, causing downtime, increased costs, and poor experience for legitimate users.

The template implements a production-ready rate limiting system using Redis as a high-speed counter store. It intercepts incoming API requests, checks the user's API key against their usage quota, and either processes the request or returns a "Too Many Requests" error. This protects your backend systems while providing clear feedback to developers using your API.

Unlike simple fixed limits, this solution allows for per-user quotas, time-based resets (hourly, daily), and different tiers of service—all configurable without code changes. It's the foundation for monetizing your API or offering freemium access models.

How It Works

The workflow acts as an API gateway that sits between your users and your backend services. Here's the step-by-step process:

1. Request Authentication & Identification

When an API request arrives (via Webhook node), the workflow extracts the API key from headers or parameters. This key uniquely identifies the user and determines their rate limit tier.

2. Redis Counter Check

The workflow uses the Redis node to check the current count for that API key. If the key doesn't exist (first request), it's created with an initial value of 1 and an expiration time based on your limit window (e.g., 1 hour).

3. Limit Evaluation

An IF node compares the current count against the user's allowed limit. Different users can have different limits stored in a database or configuration.

4. Request Processing or Rejection

If under the limit, the counter is incremented and the request proceeds to your backend. If over the limit, a 429 HTTP response is returned with details about when the limit resets.

5. Response Enrichment

Successful responses include headers showing remaining requests and reset time, following API best practices like GitHub's rate limit headers.

Pro tip: Store user tier information (free/paid/enterprise) in Redis alongside counters to avoid database lookups on every request, dramatically improving performance.

Quick Setup Guide

Get this rate limiting system running in under 30 minutes:

  1. Download and import the JSON template into your n8n instance
  2. Configure Redis connection in the Redis node with your host, port, and password
  3. Set up the Webhook to receive API requests (note the public URL n8n provides)
  4. Adjust limit thresholds in the IF node based on your user tiers (start with 100/hour for testing)
  5. Connect to your backend by replacing the HTTP Request node with your API endpoint
  6. Test thoroughly by sending requests with different API keys and watching Redis counters
  7. Deploy and update DNS to point your API domain to n8n instead of directly to your backend

Pro tip: Use n8n's workflow versioning and testing features to safely iterate on your rate limiting rules without affecting production traffic.

API rate limiting controls how many requests a user or client can make to your API within a specific time window. It's crucial for preventing abuse, protecting server resources, ensuring fair usage among customers, and maintaining service stability.

Without rate limiting, a single user could overload your system with thousands of requests, causing downtime for everyone else and potentially incurring massive cloud costs. Rate limiting is standard practice for any public API and is increasingly important for internal APIs as well.

  • Prevents DDoS and brute force attacks
  • Enables usage-based billing models
  • Improves overall system reliability

Redis is an in-memory data store that offers extremely fast read/write operations, making it ideal for rate limiting where you need to check and increment counters for every API request. Traditional databases are too slow for this use case at scale.

Redis also supports automatic key expiration with millisecond precision, so you can easily implement time-based limits like "100 requests per hour" without manual cleanup. Its atomic operations ensure accurate counting even with concurrent requests.

  • Sub-millisecond response times
  • Built-in expiration (TTL) features
  • Atomic increment operations

When a user exceeds their rate limit, the workflow returns a 429 Too Many Requests HTTP response with a clear error message. Following API best practices, the response includes headers indicating when the limit resets and how many requests are allowed.

You can customize this response to guide users toward upgrading their plan or contacting support. Some implementations also include a "Retry-After" header suggesting when to try again, improving the developer experience.
