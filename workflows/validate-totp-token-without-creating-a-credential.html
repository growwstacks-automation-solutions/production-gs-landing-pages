What This Workflow Does

This n8n automation template solves a critical security challenge: verifying Time-based One-Time Password (TOTP) codes without storing sensitive credentials in your automation platform. TOTP is the standard behind authenticator apps like Google Authenticator and Authy, providing secure two-factor authentication for millions of users worldwide.

The workflow takes a base32-encoded TOTP secret (typically stored securely in your user database) and a 6-digit code provided by a user, then validates whether the code is correct for the current 30-second time window. It returns a simple binary result: valid (1) or invalid (0), enabling you to build secure authentication flows, self-service portals, or API access controls without compromising security standards.

Businesses use this automation to reduce authentication-related support tickets, enable secure customer self-service, and maintain compliance with security standards that require proper 2FA implementation. By automating what was previously a manual or custom-coded process, you save development time while implementing enterprise-grade security.

How It Works

The workflow uses n8n's Function Node to perform cryptographic validation without external API calls or credential storage, keeping your security implementation self-contained and reliable.

Step 1: Input Processing

The workflow receives two string inputs: a base32-encoded TOTP secret and a 6-digit code. The system ensures the code is treated as text (not a number) to preserve leading zeros, which is critical for accurate validation.

Step 2: Code Generation

Using the same HMAC-SHA1 algorithm that authenticator apps use, the Function Node generates the expected 6-digit code for the current 30-second time window based on the provided secret.

Step 3: Validation Logic

The generated code is compared against the user-provided code. If they match exactly, the workflow returns a validation success signal; otherwise, it returns a failure signal that you can route to appropriate error handling or user notification processes.

Step 4: Result Routing

Based on the validation result, the workflow branches to different paths—typically allowing access to protected resources for valid codes or presenting re-authentication options for invalid attempts.

Pro tip: Always store TOTP secrets encrypted in your database and never log them. This workflow retrieves the secret securely but doesn't store it in n8n credentials, maintaining your security architecture.
